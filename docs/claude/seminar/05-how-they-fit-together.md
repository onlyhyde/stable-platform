# 05. 4337 + 7702 + 7579 조합 구조

## 배경
실서비스는 한 표준만으로 완결되지 않는다.
실행 파이프라인(운영), 주소 전환(온보딩), 기능 확장(계정 구조) 문제가 동시에 존재한다.

문제-해결 관점에서 자연스러운 읽기 순서는 다음이다.
1. 4337: 실행/검증/정산 파이프라인 확보
2. 7702: 기존 EOA 주소 유지 전환 문제 보완
3. 7579: 계정 내부 기능을 모듈화해 확장성 확보

## 문제
- 4337/7702/7579의 책임 경계를 혼동하면 설계가 꼬인다.
- 예: 7702로 가스 대납까지 해결하려 하거나, 7579로 번들링/정산 문제를 해결하려는 오해.

## 해결: 역할 분리
- `ERC-4337`: UserOperation 기반 실행/검증/정산 파이프라인
- `EIP-7702`: 기존 EOA 주소를 유지한 스마트 동작 전환 메커니즘
- `ERC-7579`: 계정 내부 기능(Validator/Executor/Hook/Fallback) 모듈 구조화

핵심:
- 4337은 "어떻게 실행할지"
- 7702는 "어떤 주소를 유지할지"
- 7579는 "계정 내부를 어떻게 확장할지"를 해결한다.

## 표준별 질문 프레임
- 4337 질문: "가스 대납/배치/자동화를 운영 인프라로 처리해야 하는가?"
- 7702 질문: "기존 주소를 유지한 채 전환해야 하는가?"
- 7579 질문: "기능을 모듈로 독립 배포/교체해야 하는가?"

## 추천 조합 패턴 (문제 기준)
1. 기존 EOA 자산/이력 연속성이 핵심
- `4337 + 7702`를 기본으로 시작
- 확장 필요 시 `7579`를 추가

2. 운영 UX(가스 대납/자동화)가 핵심
- `4337` 필수
- 주소 유지 요구가 있으면 `7702` 결합

3. 계정 기능 확장/교체가 핵심
- `7579` 채택
- 실제 운영 파이프라인은 `4337` 결합
- 기존 주소 유지 필요 시 `7702` 추가

## 실무 결론
- 세 표준은 대체관계가 아니라 보완관계다.
- 실무 기본형은 `4337 + 7702 + 7579` 조합이지만,
  도입 순서는 "현재 해결해야 할 문제"(운영/주소/확장) 우선순위에 맞춰 단계적으로 선택하는 것이 합리적이다.
