## 2. CRITICAL — 즉시 수정 필요

### 2.1 Permit2Paymaster: validation 단계에서 외부 상태변경 호출

- **분류**: 신규 발견
- **위치**: `Permit2Paymaster.sol:223`
- **코드**:
  ```solidity
  try PERMIT2.permit(userOp.sender, permitSingle, payload.permitSig) {
  ```
- **현상**: `_validatePaymasterUserOp` 내에서 Permit2 컨트랙트의 `permit()`을 호출한다. 이는 외부 컨트랙트의 상태(allowance nonce)를 변경하는 호출이다.
- **영향**:
  - ERC-4337 simulation rules는 validation 단계에서 외부 스토리지 변경을 금지한다.
  - 표준 번들러(Stackup, Pimlico, Alchemy 등)에서 거의 확실히 거부된다.
  - 감사표에서 지적한 `senderNonce++`(내부 상태변경)보다 훨씬 심각한 위반이다.
- **권장**:
  - (A) Permit2 permit 실행을 execution 단계(`postOp`)로 이동
  - (B) 사용자가 사전에 Permit2 approve를 완료하는 방식으로 전환
  - (C) 신뢰 번들러 전제 운영이라면 문서에 명시

### 2.2 ~~Permit2Paymaster: `getTokenAmount()`에서 division by zero~~ **[RESOLVED]**

- **분류**: 신규 발견
- **위치**: `Permit2Paymaster.sol:163`
- **상태**: **수정 완료** — `if (tokenPrice == 0) revert InvalidPrice();` 추가됨 (`Permit2Paymaster.sol:163`)
- ~~**코드**~~: (이전 코드에서 tokenPrice == 0 체크가 누락되어 있었음)
- **해결**: ERC20Paymaster와 동일한 `InvalidPrice()` 체크가 추가되었다.
- **확인일**: 2026-02-25

## 3. HIGH — 번들러 호환성 리스크

### 3.1 VerifyingPaymaster / SponsorPaymaster: validation 단계 상태변경 **[재분류: BUNDLER COMPAT]**

- **분류**: 감사표 기존 (4.3.1)
- **위치**: `VerifyingPaymaster.sol:147`, `SponsorPaymaster.sol:139`
- **코드**:
  ```solidity
  senderNonce[userOp.sender]++;
  ```
- **현상**: `validatePaymasterUserOp` 실행 경로 내에서 스토리지를 변경한다.
- **영향**: IPaymaster 주석에 따르면, 번들러는 상태변경 paymaster를 신뢰 목록으로 제한할 수 있다. 스펙 MUST 위반은 아니지만 **번들러 호환성 리스크**이다.
- **2026-02-25 재검토**: EIP-4337 `IPaymaster.validatePaymasterUserOp`은 non-view 함수로 정의되어 컨트랙트 인터페이스상 상태 변경이 허용됨. `senderNonce[userOp.sender]`는 sender-associated storage로 associated storage 규칙을 위반하지 않음. 코드 내 `[EIP-4337 Bundler Compatibility Warning]` 인라인 주석으로 문서화 완료. **스펙 MUST 위반이 아닌 BUNDLER COMPAT으로 재분류**.
- **권장**:
  - (A) nonce 증가를 `_postOp`으로 이동 (무상태 validation 달성)
  - (B) 신뢰 paymaster 전제 운영으로 명문화 **(현재 코드에 이미 반영)**

### 3.2 ERC20Paymaster: validation 단계에서 외부 view 호출 **[재분류: BUNDLER COMPAT]**

- **분류**: 신규 발견
- **위치**: `ERC20Paymaster.sol:226-233`
- **코드**:
  ```solidity
  uint256 balance = IERC20(payload.token).balanceOf(userOp.sender);
  uint256 allowance = IERC20(payload.token).allowance(userOp.sender, address(this));
  ```
- **현상**: validation 중 외부 ERC-20 컨트랙트의 스토리지를 읽는다.
- **영향**: ERC-4337 simulation rules는 associated storage rule을 적용하여, validation 중 접근 가능한 스토리지를 제한한다. account와 paymaster 자체 스토리지 및 "associated storage"만 허용되며, 임의 ERC-20 컨트랙트 접근은 위반될 수 있다. 엄격한 번들러에서 거부 가능.
- **2026-02-25 재검토**: `balanceOf(sender)`는 `mapping(address=>uint256)`에서 sender를 키로 읽는 것으로 sender-associated storage에 해당. `allowance(sender, this)`는 중첩 매핑이나 첫 키가 sender이므로 대부분의 번들러에서 허용됨. Pimlico 등 프로덕션 ERC-20 Paymaster에서 동일 패턴 사용. 코드 내 `[EIP-4337 Bundler Compatibility Warning]` 인라인 주석으로 문서화 완료. **스펙 MUST 위반이 아닌 BUNDLER COMPAT으로 재분류**.
- **권장**:
  - (A) balance/allowance 체크를 제거하고 postOp의 transferFrom 실패로 대체
  - (B) staked paymaster로 운영하여 스토리지 접근 제한 완화 **(권장)**

## 4. MEDIUM — 설계/운영 리스크

### 4.1 ~~Envelope nonce와 senderNonce 간 정합성 부재~~ **[재분류: NOT AN ISSUE]**

- **분류**: 감사표 기존 (4.3.2)
- **위치**: `VerifyingPaymaster.sol:117-147`, `SponsorPaymaster.sol:106-139`
- **현상**: Envelope 구조체에서 `env.nonce`(uint64)를 디코딩하지만, 실제 `senderNonce` 매핑과 비교/검증하지 않는다.
- **2026-02-25 재검토**: 두 nonce는 **상호 보완적** 메커니즘으로 각각 올바르게 동작한다:
  - `envelope.nonce`는 `hashForSignature()` 함수에서 서명 해시에 포함됨 (`PaymasterDataLib.sol:206`). 전체 envelope 바이트가 해시에 들어가므로, nonce가 다르면 서명도 달라진다. 이는 **오프체인 서명 고유성**을 보장한다.
  - `senderNonce[userOp.sender]++`는 **온체인 리플레이 방지**를 제공한다. 한 번 실행된 UserOp는 senderNonce가 증가하므로 재실행될 수 없다.
  - `env.nonce == senderNonce` 검증을 강제하면, SDK가 매 요청마다 온체인 senderNonce를 조회해야 하는 UX 부담이 발생하며, signer의 nonce 체계 자유도가 제한된다.
  - **리플레이 방지는 senderNonce만으로 완전하며, envelope nonce는 오프체인 구분 목적으로 설계됨.**
- **판정**: 이슈 아님. 의도된 설계.
- ~~**권장**~~: 조치 불필요

### 4.2 `MIN_DEPOSIT` 선언 후 미사용

- **분류**: 감사표 기존 (4.3.3)
- **위치**: `BasePaymaster.sol:23`, `BasePaymaster.sol:26`
- **현상**: 상수와 에러가 선언되어 있으나 어디서도 사용되지 않는다.
  ```solidity
  uint256 public constant MIN_DEPOSIT = 0.01 ether;
  error InsufficientDeposit(uint256 required, uint256 available);
  ```
- **영향**: 스펙 위반은 아니다. 그러나 deposit 시 최소 금액 검증이 없어 운영 안전성이 약하다.
- **권장**: `deposit()` 또는 별도 함수에서 `MIN_DEPOSIT` 검증을 추가하거나, 불필요하면 상수와 에러를 제거.

## 5. LOW — 코드 품질/정리

### 5.1 ERC20Paymaster/Permit2Paymaster: postOp revert 시 paymaster 가스 손실

- **위치**: `ERC20Paymaster.sol:266-269`, `Permit2Paymaster.sol:271-278`
- **현상**: validation과 execution 사이에 사용자의 token balance/allowance가 변할 수 있다. `postOp`에서 `safeTransferFrom` 또는 Permit2 `transferFrom`이 실패하면, EntryPoint가 `PostOpReverted` 모드로 재호출하며 paymaster가 가스 비용을 전액 부담한다.
- **참고**: `ERC20Paymaster.sol:266` 주석에서 이 리스크를 인지하고 있다.
- **권장**: validation 단계에서 사전 토큰 잠금(pre-charge)을 수행하거나, postOp에서 실패 시 별도 채권 기록.

### 5.2 ~~VerifyingPaymaster `_postOp`가 `pure`로 선언~~ **[재분류: NOT AN ISSUE]**

- **위치**: `VerifyingPaymaster.sol:162`
- **현상**: `function _postOp(...) internal pure override`로 선언.
- **2026-02-25 재검토**: Solidity에서 pure 함수는 non-pure 인터페이스의 유효한 구현이다. VerifyingPaymaster는 post-op에서 할 작업이 없으므로 pure 선언이 의도적이고 올바르다. 가스 효율도 약간 향상된다. 향후 post-op 로직이 필요하면 pure 제거만 하면 된다.
- **판정**: 이슈 아님.

### 5.3 EntryPoint `nonReentrant` 구현 방식

- **분류**: 감사표 기존 (3. 주의/확장)
- **위치**: `EntryPoint.sol:80-85`
- **코드**:
  ```solidity
  require(tx.origin == msg.sender && msg.sender.code.length == 0, Reentrancy());
  ```
- **현상**: EOA 직접 호출만 허용하는 방식의 reentrancy guard이다.
- **영향**: 스펙 위반은 아니나, 다른 컨트랙트에서 `handleOps`를 호출하는 구조를 원천 차단한다.

## 8. 추가 권장 테스트 시나리오

감사표 7장의 권장 테스트에 더하여:

1. **Permit2Paymaster 번들러 시뮬레이션**: `eth_estimateUserOperationGas`에서 `PERMIT2.permit()` 호출이 차단되는지 확인
2. **Permit2Paymaster 오라클 0가격 시나리오**: `tokenPrice=0` 반환 시 적절한 에러 revert 확인
3. **ERC20Paymaster 번들러 storage access**: `balanceOf`/`allowance` 호출이 번들러 시뮬레이션에서 허용되는지 확인
4. **postOp 실패 경로**: 사용자가 validation 후 토큰을 전송하여 balance 부족 상태에서 postOp 동작 확인
5. **Envelope nonce 조작**: envelope nonce를 임의 값으로 설정한 서명이 정상 처리되는지 확인 (현재 검증 없음)
