## 2. BUNDLER COMPAT — 번들러 호환성 리스크

코드 내 `[EIP-4337 Bundler Compatibility Warning]` 인라인 주석으로 문서화 완료. 스펙 MUST 위반 아님. trusted bundler 전제 운영.

### 2.1 Permit2Paymaster: validation 단계에서 외부 상태변경 호출

- **위치**: `Permit2Paymaster.sol:239`
- **현상**: `_validatePaymasterUserOp` 내에서 `PERMIT2.permit()` 호출 → 외부 상태(allowance nonce) 변경
- **영향**: 표준 번들러에서 거부 가능
- **권장**: (A) permit을 `postOp`으로 이동, (B) 사전 Permit2 approve 방식, (C) trusted bundler 명문화

### 2.2 VerifyingPaymaster / SponsorPaymaster: validation 단계 상태변경

- **위치**: `VerifyingPaymaster.sol:147`, `SponsorPaymaster.sol:139`
- **현상**: `senderNonce[userOp.sender]++` — validation 중 스토리지 변경
- **참고**: sender-associated storage로 associated storage 규칙 위반 아님
- **권장**: (A) nonce 증가를 `_postOp`으로 이동, (B) trusted paymaster 명문화 **(현재 반영됨)**

### 2.3 ERC20Paymaster: validation 단계에서 외부 view 호출

- **위치**: `ERC20Paymaster.sol:226-233`
- **현상**: `balanceOf(sender)`, `allowance(sender, this)` — validation 중 외부 ERC-20 스토리지 읽기
- **참고**: sender-associated storage에 해당, Pimlico 등 프로덕션에서 동일 패턴 사용
- **권장**: (A) balance/allowance 체크 제거 후 postOp 실패로 대체, (B) staked paymaster 운영 **(권장)**

## 3. MEDIUM — 검증 필요

### 3.1 v0.9 postOp 정산 메커니즘

- **영향 범위**: `ERC20Paymaster.sol`, `Permit2Paymaster.sol`
- **내용**: v0.9에서 `postOp` revert 시 `PostOpReverted` 모드 재호출 없이 prefund에서 직접 정산
- **검증 필요**: 현재 `postOp` 구현이 이중 호출을 전제하지 않는지 확인
- 참조: 스펙표준 line 342-348

### 3.2 Paymaster 스테이킹 요건

- **영향 범위**: 전체 Paymaster
- **내용**: `postOp` 사용 Paymaster(context 비어있지 않음)와 자체 스토리지 접근 Paymaster는 스테이킹 필요
- **검증 필요**: ERC20/Permit2 Paymaster 스테이킹/미스테이킹 시 번들러 시뮬레이션 결과 확인
- 참조: 스펙표준 §7.3 (line 497-502)

## 4. LOW — 운영 리스크 / 설계 선택

### 4.1 postOp revert 시 paymaster 가스 손실

- **위치**: `ERC20Paymaster.sol:275-276`, `Permit2Paymaster.sol:287-293`
- **현상**: validation 후 사용자 token balance/allowance 변경 시 `postOp`에서 transferFrom 실패 → paymaster 가스 전액 부담
- **참고**: `ERC20Paymaster.sol:273-274` 주석에서 리스크 인지
- **권장**: validation 단계 사전 토큰 잠금(pre-charge) 또는 postOp 실패 시 채권 기록

### 4.2 EntryPoint nonReentrant EOA 제한

- **위치**: `EntryPoint.sol:80-85`
- **현상**: `tx.origin == msg.sender && msg.sender.code.length == 0` — EOA 직접 호출만 허용
- **영향**: 다른 컨트랙트에서 `handleOps` 호출 원천 차단. 설계 선택.

### 4.3 nonReentrant EIP-7702 EOA 차단

- **위치**: `EntryPoint.sol:80-85`
- **현상**: `code.length == 0` 조건이 EIP-7702 delegation EOA도 차단 (delegation designation → code.length=23)
- **검증 필요**: 번들러가 EIP-7702 delegation EOA에서 `handleOps` 호출하는 시나리오 확인
- 참조: 스펙표준 line 114-117

## 5. 권장 테스트 시나리오

1. **Permit2Paymaster 번들러 시뮬레이션**: `eth_estimateUserOperationGas`에서 `PERMIT2.permit()` 호출 차단 여부
2. **ERC20Paymaster 번들러 storage access**: `balanceOf`/`allowance` 호출이 번들러 시뮬레이션에서 허용되는지
3. **postOp 실패 경로**: validation 후 토큰 전송하여 balance 부족 상태에서 postOp 동작
4. **v0.9 postOp 실패 정산 경로**: postOp revert 시 EntryPoint가 prefund에서 직접 정산 (이중 호출 미발생 검증)
5. **Paymaster 스테이킹 상태 확인**: ERC20/Permit2 Paymaster 스테이킹/미스테이킹 시 번들러 시뮬레이션
