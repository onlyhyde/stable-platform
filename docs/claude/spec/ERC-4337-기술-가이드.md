# ERC-4337 계정 추상화 기술 가이드

> 개발자를 위한 ERC-4337 표준 구현 가이드. 이 문서는 ERC-4337의 아키텍처, 핵심 컴포넌트, 인터페이스 명세, 그리고 실제 구현에 필요한 기술적 세부사항을 다룹니다.

---

## 1. 개요

### 1.1 계정 추상화란?

계정 추상화(Account Abstraction)는 이더리움의 두 가지 계정 유형(EOA, 스마트 컨트랙트)의 한계를 극복하기 위한 제안입니다. 현재 이더리움에서 트랜잭션을 발생시킬 수 있는 것은 오직 EOA(Externally Owned Account)뿐이며, EOA는 단일 개인키에 의존하는 고정된 서명 방식만 사용할 수 있습니다.

ERC-4337은 **합의 계층(consensus layer)의 변경 없이**, 상위 인프라 계층에서 스마트 컨트랙트 지갑을 통해 다음을 가능하게 합니다:

- 커스텀 서명 검증 로직 (멀티시그, 소셜 리커버리 등)
- 가스비 대납 (Paymaster를 통한 스폰서 트랜잭션)
- ERC-20 토큰으로 가스비 납부
- 배치 트랜잭션 (한 번의 요청으로 다수 호출 실행)
- 서명 압축을 통한 가스비 최적화

### 1.2 핵심 설계 원칙

| 원칙 | 설명 |
|------|------|
| **합의 계층 무변경** | 하드포크 없이 상위 계층에서 구현 |
| **탈중앙화** | 누구나 Bundler가 될 수 있음 |
| **EOA 불필요** | 스마트 컨트랙트 지갑만으로 모든 기능 수행 |
| **DoS 방어** | 시뮬레이션, 스테이킹, 평판 시스템을 통한 보호 |

### 1.3 아키텍처 전체 흐름

```
사용자                  Bundler              EntryPoint              Account
  │                      │                     │                      │
  │  UserOperation 생성   │                     │                      │

  │                      │  시뮬레이션           │                      │
  │                      │  (simulateValidation)│                      │

  │                      │                     │  validateUserOp()    │
  │                      │                     │─────────────────────>│
  │                      │                     │<─────────────────────│
  │                      │<────────────────────│                      │
  │                      │                     │                      │
  │                      │  번들링 후 handleOps()│                      │
  │                      │────────────────────>│                      │
  │                      │                     │  [검증 루프]           │
```

## 2. 핵심 컴포넌트

### 2.1 참여자 정의

| 참여자 | 역할 | 특징 |
|--------|------|------|
| **Sender (Smart Account)** | UserOperation을 전송하는 스마트 컨트랙트 지갑 | `IAccount` 인터페이스 구현 필수 |
| **EntryPoint** | 번들링된 UserOperation을 처리하는 싱글톤 컨트랙트 | 체인당 하나의 인스턴스, 모든 지갑이 공유 |
| **Bundler** | UserOperation을 수집하여 번들 트랜잭션을 생성하는 노드 | 블록 생성자가 겸할 수 있음 |
| **Paymaster** | 사용자 대신 가스비를 납부하는 헬퍼 컨트랙트 | 스폰서 트랜잭션, ERC-20 납부 등 |
| **Factory** | 새로운 Smart Account를 배포하는 컨트랙트 | CREATE2 사용으로 결정론적 주소 생성 |
| **Aggregator** | 다수 UserOperation의 서명을 압축 검증하는 컨트랙트 | BLS 등 압축 서명 방식 지원 |

### 2.2 컴포넌트 관계도

```
┌─────────────────────────────────────────────────────────────┐
│                       EntryPoint (싱글톤)                     │
│                                                              │
│  handleOps()  ←──── Bundler가 호출                           │
│      │                                                       │
│      ├── [검증 루프] ──> Smart Account.validateUserOp()       │
│      │                ──> Paymaster.validatePaymasterUserOp() │
│      │                ──> Factory (initCode가 있는 경우)       │
│      │                                                       │
│      ├── [실행 루프] ──> Smart Account (callData 실행)         │
│      │                                                       │
│      └── [후처리]   ──> Paymaster.postOp()                    │
│                                                              │
│  deposit / withdraw / stake 관리                              │
└─────────────────────────────────────────────────────────────┘
```

## 3. UserOperation 구조

### 3.1 필드 명세 (v0.6 기준)

```solidity
struct UserOperation {
    address sender;               // 지갑 컨트랙트 주소
    uint256 nonce;                // 리플레이 방지 값
    bytes   initCode;             // 지갑 생성 코드 (factory주소 20bytes + calldata)
    bytes   callData;             // 지갑에서 실행할 함수 호출 데이터
    uint256 callGasLimit;         // 메인 실행에 할당할 가스
    uint256 verificationGasLimit; // 검증 단계에 할당할 가스
    uint256 preVerificationGas;   // 콜데이터 및 오버헤드 가스 보정값
    uint256 maxFeePerGas;         // EIP-1559 최대 가스 가격
    uint256 maxPriorityFeePerGas; // EIP-1559 우선순위 수수료
    bytes   paymasterAndData;     // paymaster주소 20bytes + 데이터 (없으면 빈값)
    bytes   signature;            // 서명 데이터 (지갑 구현에 따라 다름)
}
```

### 3.2 PackedUserOperation (v0.7, 최신 표준)

v0.7에서는 가스비 효율을 위해 packed 형식을 사용합니다:

```solidity
struct PackedUserOperation {
    address sender;
    uint256 nonce;
    bytes   initCode;             // factory(20bytes) + factoryData, 또는 빈값
    bytes   callData;
    bytes32 accountGasLimits;     // verificationGasLimit(16bytes) + callGasLimit(16bytes)
    uint256 preVerificationGas;
    bytes32 gasFees;              // maxPriorityFeePerGas(16bytes) + maxFeePerGas(16bytes)
    bytes   paymasterAndData;     // paymaster(20bytes) + verificationGasLimit(16bytes)
                                  //   + postOpGasLimit(16bytes) + data
    bytes   signature;
}
```

### 3.3 필드 상세 설명

**`sender`**: UserOperation을 보내는 스마트 컨트랙트 지갑의 주소. EOA 주소가 아님에 주의.

**`nonce`**: 192비트 "key" + 64비트 "sequence"로 구성.
- 같은 key 내에서 sequence는 순차적으로 증가
- 다른 key를 사용하면 병렬적인 nonce 채널 운용 가능
- 관리 작업과 일반 작업의 nonce를 분리하는 등의 활용

```solidity
// EntryPoint에서 nonce 조회
function getNonce(address sender, uint192 key) external view returns (uint256 nonce);
```

**`initCode`**: 지갑이 아직 배포되지 않은 경우, 첫 20바이트는 Factory 주소, 나머지는 Factory에 전달할 calldata. 이미 배포된 지갑이면 빈 값.

**`paymasterAndData`**: paymaster가 있는 경우 첫 20바이트는 paymaster 주소, 이후 16바이트는 paymasterVerificationGasLimit, 이후 16바이트는 postOpGasLimit, 나머지는 paymaster에 전달할 데이터.

**`signature`**: 프로토콜에서 형식을 강제하지 않음. 지갑의 `validateUserOp` 구현에 따라 ECDSA, 멀티시그, BLS 등 자유롭게 사용 가능. 단, 리플레이 방지를 위해 `chainId`와 `EntryPoint` 주소에 의존해야 함.

### 3.4 userOpHash 계산

```solidity
// EIP-712 typed message hash
bytes32 userOpHash = keccak256(abi.encode(
    keccak256(pack(userOp)),  // signature를 제외한 UserOp의 해시
    address(entryPoint),
    block.chainid
));
```

v0.7에서는 EIP-712 도메인 분리자를 사용합니다:

```solidity
bytes32 constant TYPE_HASH = keccak256(
    "EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)"
);

bytes32 constant PACKED_USEROP_TYPEHASH = keccak256(
    "PackedUserOperation(address sender,uint256 nonce,bytes initCode,"
    "bytes callData,bytes32 accountGasLimits,uint256 preVerificationGas,"
    "bytes32 gasFees,bytes paymasterAndData)"
);
```

## 4. EntryPoint 컨트랙트

### 4.1 인터페이스

EntryPoint는 체인당 하나만 존재하는 싱글톤 컨트랙트입니다. 모든 Smart Account와 Bundler가 동일한 EntryPoint를 사용합니다.

```solidity
interface IEntryPoint is IStakeManager {

    // 번들 실행 (aggregator 미사용)
    function handleOps(
        PackedUserOperation[] calldata ops,
        address payable beneficiary
    ) external;

    // 번들 실행 (aggregator 사용)
    function handleAggregatedOps(
        UserOpsPerAggregator[] calldata opsPerAggregator,
        address payable beneficiary
    ) external;

    // UserOp 해시 계산
    function getUserOpHash(PackedUserOperation calldata userOp)
        external view returns (bytes32);

    // 유효성 검사 시뮬레이션 (항상 revert로 결과 반환)
    function simulateValidation(PackedUserOperation calldata userOp) external;

    // counterfactual 주소 조회
    function getSenderAddress(bytes memory initCode) external;

    // nonce 조회
    function getNonce(address sender, uint192 key)
        external view returns (uint256 nonce);
}
```

### 4.2 handleOps 실행 흐름

`handleOps`는 **검증 루프**와 **실행 루프**를 순차적으로 수행합니다. 이 분리가 매우 중요합니다.

#### 검증 루프 (Verification Loop)

각 UserOperation에 대해:

1. **계정 생성**: `initCode`가 있으면 Factory를 호출하여 지갑 생성
2. **최대 가스비 계산**: 검증 + 실행 가스 한도 기반으로 최대 비용 산정
3. **지갑 검증**: `account.validateUserOp(userOp, userOpHash, missingAccountFunds)` 호출
4. **예치금 확인**: 최대 가스비를 커버할 수 있는 예치금이 있는지 확인
5. **Paymaster 검증** (있는 경우): `paymaster.validatePaymasterUserOp(userOp, userOpHash, maxCost)` 호출

> **핵심**: 모든 UserOperation의 검증이 완료된 후에야 실행 루프로 진입합니다. 이는 한 UserOperation의 실행이 다른 UserOperation의 검증 결과에 영향을 주는 것을 방지합니다.

#### 실행 루프 (Execution Loop)

각 UserOperation에 대해:

1. **callData 실행**: 지갑에 callData를 전달하여 실행
   - callData가 `IAccountExecute.executeUserOp`의 함수 시그니처로 시작하면, `executeUserOp(userOp, userOpHash)`로 호출
2. **postOp 호출** (Paymaster가 있고 context를 반환한 경우): `paymaster.postOp()` 호출
3. **가스비 정산**: 미사용 가스비를 환불하고, 사용된 가스비를 `beneficiary`에게 지급

#### 미사용 가스 패널티

```
callGasLimit 또는 paymasterPostOpGasLimit의 미사용 가스가
40,000 (PENALTY_GAS_THRESHOLD) 이상이면
미사용 가스의 10% (UNUSED_GAS_PENALTY_PERCENT)를 패널티로 부과
```

이 패널티는 UserOperation이 과도한 가스를 예약하여 번들의 다른 UserOperation을 차단하는 것을 방지합니다.

### 4.3 예치금(Deposit)과 스테이크(Stake) 관리

```solidity
interface IStakeManager {
    struct DepositInfo {
        uint112 deposit;          // 가스비 지불용 예치금
        bool    staked;           // 스테이킹 여부
        uint112 stake;            // 스테이킹 금액
        uint32  unstakeDelaySec;  // 언스테이크 대기 시간
        uint48  withdrawTime;     // 인출 가능 시점
    }

    // 예치금 관리
    function balanceOf(address account) external view returns (uint256);
    function depositTo(address account) external payable;
    function withdrawTo(address payable withdrawAddress, uint256 withdrawAmount) external;

    // 스테이크 관리
    function addStake(uint32 _unstakeDelaySec) external payable;
    function unlockStake() external;
    function withdrawStake(address payable withdrawAddress) external;
}
```

**예치금(Deposit)**: 가스비 지불에 사용되는 잔액. 즉시 인출 가능.

**스테이크(Stake)**: Paymaster, Factory, Aggregator가 평판 시스템에 참여하기 위해 잠그는 금액. `unlockStake()` 호출 후 `unstakeDelaySec` 만큼 대기해야 인출 가능. 슬래싱(몰수)은 없음.

## 5. Smart Account (지갑) 구현

### 5.1 IAccount 인터페이스

모든 ERC-4337 호환 지갑은 반드시 이 인터페이스를 구현해야 합니다:

```solidity
interface IAccount {
    function validateUserOp(
        PackedUserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 missingAccountFunds
    ) external returns (uint256 validationData);
}
```

### 5.2 validateUserOp 구현 요구사항

**필수(MUST)**:
1. 호출자가 신뢰할 수 있는 EntryPoint인지 확인
2. 서명이 유효한 `userOpHash`의 서명인지 확인
3. 서명 불일치 시 `SIG_VALIDATION_FAILED(1)`을 반환 (revert 하지 않음)
4. `missingAccountFunds`만큼 EntryPoint에 이더 전송

**권장(SHOULD)**:
- 서명 실패 시에도 조기 반환하지 않고 정상 흐름을 완료 (가스 추정 가능하도록)
- 미래 트랜잭션을 위해 초과 금액 예치 가능

### 5.3 validationData 반환값 구조

```
┌────────────────────┬──────────────┬──────────────┐
│   authorizer       │  validUntil  │  validAfter  │
│   (20 bytes)       │  (6 bytes)   │  (6 bytes)   │
└────────────────────┴──────────────┴──────────────┘
```

- `authorizer`: 0 = 유효한 서명, 1 = 서명 실패, 그 외 = aggregator 주소
- `validUntil`: 이 UserOp이 유효한 마지막 시간 (0 = 무제한)
- `validAfter`: 이 UserOp이 유효해지는 시작 시간

```solidity
// 패킹/언패킹 헬퍼
function _packValidationData(
    bool sigFailed,
    uint48 validUntil,
    uint48 validAfter
) pure returns (uint256) {
    return (sigFailed ? 1 : 0)
        | (uint256(validUntil) << 160)
        | (uint256(validAfter) << (160 + 48));
}
```

### 5.4 SimpleAccount 구현 예제

```solidity
import "@openzeppelin/contracts/proxy/utils/Initializable.sol";
import "@openzeppelin/contracts/proxy/utils/UUPSUpgradeable.sol";

contract SimpleAccount is BaseAccount, UUPSUpgradeable, Initializable {

    IEntryPoint private immutable _entryPoint;
    address public owner;

    // === 생성 및 초기화 ===

    constructor(IEntryPoint anEntryPoint) {
        _entryPoint = anEntryPoint;
        _disableInitializers(); // 로직 컨트랙트의 직접 초기화 방지
    }

    function initialize(address anOwner) public virtual initializer {
        owner = anOwner;
    }

    // === IAccount 구현 ===

    function entryPoint() public view override returns (IEntryPoint) {
        return _entryPoint;
    }

    function _validateSignature(
        UserOperation calldata userOp,
        bytes32 userOpHash
    ) internal override virtual returns (uint256 validationData) {
        bytes32 hash = userOpHash.toEthSignedMessageHash();
        if (owner != hash.recover(userOp.signature))
            return SIG_VALIDATION_FAILED;  // 1을 반환, revert 하지 않음
        return 0;
    }

    // === 실행 ===

    function execute(
        address dest,
        uint256 value,
        bytes calldata func
    ) external {
        _requireFromEntryPointOrOwner();
        _call(dest, value, func);
    }

    function executeBatch(
        address[] calldata dest,
        uint256[] calldata value,
        bytes[] calldata func
    ) external {
        _requireFromEntryPointOrOwner();
        for (uint256 i = 0; i < dest.length; i++) {
            _call(dest[i], value[i], func[i]);
        }
    }

    // === 예치금 관리 ===

    function addDeposit() public payable {
        entryPoint().depositTo{value: msg.value}(address(this));
    }

    function withdrawDepositTo(
        address payable withdrawAddress,
        uint256 amount
    ) public onlyOwner {
        entryPoint().withdrawTo(withdrawAddress, amount);
    }

    // === 업그레이드 ===

    function _authorizeUpgrade(address) internal view override {
        require(msg.sender == owner, "only owner");
    }

    receive() external payable {}
}
```

### 5.5 업그레이드 패턴 (UUPS)

ERC-4337에서는 업그레이드 가능한 지갑을 권장합니다. EntryPoint 주소가 변경되면 새로운 로직 컨트랙트로 업그레이드해야 하기 때문입니다.

**UUPS 패턴의 핵심**:
- 프록시 컨트랙트: 변하지 않는 주소, 상태 변수 저장
- 로직 컨트랙트: 업그레이드 가능한 코드, `delegatecall`로 실행
- 업그레이드 함수(`upgradeTo`)가 로직 컨트랙트에 위치
- ERC-1967 표준 슬롯에 로직 컨트랙트 주소 저장

```
Storage slot: 0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc
= bytes32(uint256(keccak256('eip1967.proxy.implementation')) - 1)
```

**배포 시 주의사항**:
- 생성자(`constructor`)는 프록시의 상태를 변경하지 못함 → `initialize` 함수 사용
- `initializer` modifier로 한 번만 호출되도록 제한
- `immutable` 변수(예: EntryPoint 주소)는 생성자에서 설정 가능 (바이트코드에 저장되므로)

```javascript
// Hardhat 배포 스크립트
const proxy = await upgrades.deployProxy(
    SimpleAccount,
    [ownerAddress],              // initialize 파라미터
    {
        kind: "uups",
        unsafeAllow: ["constructor", "state-variable-immutable"],
        constructorArgs: [ENTRY_POINT_ADDRESS]
    }
);
```

### 5.6 IAccountExecute (선택적)

```solidity
interface IAccountExecute {

        PackedUserOperation calldata userOp,
```

이 인터페이스를 구현하면, EntryPoint가 callData를 직접 실행하는 대신 `executeUserOp`를 통해 전체 UserOperation 정보와 함께 실행합니다.

## 6. Paymaster 구현

### 6.1 IPaymaster 인터페이스

```solidity
interface IPaymaster {
    enum PostOpMode {
        opSucceeded,      // UserOp 성공
        opReverted,       // UserOp 실패 (가스비는 여전히 부담)
        postOpReverted    // UserOp 성공 → 첫 번째 postOp 실패 → 두 번째 호출
    }

    function validatePaymasterUserOp(
        PackedUserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 maxCost
    ) external returns (bytes memory context, uint256 validationData);

    function postOp(
        PostOpMode mode,
        bytes calldata context,
        uint256 actualGasCost,
        uint256 actualUserOpFeePerGas
    ) external;
}
```

### 6.2 Paymaster 동작 흐름

```
1. EntryPoint → paymaster.validatePaymasterUserOp()
   - 이 요청의 가스비를 대납할지 결정
   - context 데이터 반환 (postOp에서 사용)
   - validationData 반환 (시간 범위, 서명 검증 결과)

2. EntryPoint → account (callData 실행)

3. EntryPoint → paymaster.postOp()
   - 실행 후 처리 (ERC-20 토큰 인출 등)
   - mode로 실행 결과 확인
```

### 6.3 postOp의 이중 호출 메커니즘

Paymaster의 `postOp`은 최대 두 번 호출될 수 있습니다:

```
시나리오 1: 정상 실행
  validatePaymasterUserOp → 실행(성공) → postOp(opSucceeded)

시나리오 2: 실행 실패
  validatePaymasterUserOp → 실행(실패) → postOp(opReverted)

시나리오 3: 실행 성공, 첫 번째 postOp 실패
  validatePaymasterUserOp → 실행(성공) → postOp(opSucceeded) [실패!]
                                       → 실행 revert
                                       → postOp(postOpReverted)
```

시나리오 3이 중요한 이유: 실행 중에 사용자가 ERC-20 잔액을 모두 인출하면 postOp에서 토큰을 청구할 수 없습니다. 이 경우 실행이 revert되고, `validatePaymasterUserOp` 직후 상태에서 다시 `postOp`이 호출되어 토큰을 청구할 수 있습니다.

### 6.4 Paymaster 유즈케이스별 구현 패턴

#### ERC-20 토큰 납부 Paymaster

```solidity
contract TokenPaymaster is IPaymaster {
    IERC20 public token;
    uint256 public pricePerGas; // 토큰/가스 환율

    function validatePaymasterUserOp(
        PackedUserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 maxCost
    ) external returns (bytes memory context, uint256 validationData) {
        // 사용자가 충분한 토큰 승인(approve)을 했는지 확인
        uint256 tokenCost = maxCost * pricePerGas;
        require(
            token.allowance(userOp.sender, address(this)) >= tokenCost,
            "insufficient token allowance"
        );
        // context에 sender와 최대 비용 저장
        return (abi.encode(userOp.sender, tokenCost), 0);
    }

    function postOp(
        PostOpMode mode,
        bytes calldata context,
        uint256 actualGasCost,
        uint256 actualUserOpFeePerGas
    ) external {
        (address sender, ) = abi.decode(context, (address, uint256));
        uint256 actualTokenCost = actualGasCost * pricePerGas;
        // 실제 사용된 가스비에 해당하는 토큰만 인출
        token.transferFrom(sender, address(this), actualTokenCost);
    }
}
```

#### 스폰서 Paymaster (특정 조건에서 가스비 대납)

```solidity
contract SponsorPaymaster is IPaymaster {
    address public sponsor;
    mapping(address => bool) public allowedContracts;

    function validatePaymasterUserOp(
        PackedUserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 maxCost
    ) external returns (bytes memory context, uint256 validationData) {
        // paymasterData에서 스폰서 서명 추출 및 검증
        bytes memory paymasterData = _extractPaymasterData(userOp);
        require(_verifySponsorship(userOp, paymasterData), "invalid sponsorship");

        return ("", 0); // context 없음 → postOp 호출하지 않음
    }

    function postOp(PostOpMode, bytes calldata, uint256, uint256) external {
        // context가 빈 값이므로 호출되지 않음
    }
}
```

### 6.5 Paymaster 보안 요구사항

- `validatePaymasterUserOp`과 `postOp` 호출자가 EntryPoint인지 **반드시** 확인
- EntryPoint에 충분한 이더 예치 필요
- 글로벌 스토리지 접근 시 스테이킹 필요
- `postOp`이 있는 Paymaster는 반드시 스테이킹

### 6.6 paymasterAndData 인코딩

```
┌──────────────┬─────────────────────────────┬──────────────────┬──────────────┐
│ paymaster    │ paymasterVerificationGasLimit│ postOpGasLimit   │ paymasterData│
│ (20 bytes)   │ (16 bytes)                  │ (16 bytes)       │ (variable)   │
└──────────────┴─────────────────────────────┴──────────────────┴──────────────┘
```

선택적으로 `paymasterSignature`를 추가할 수 있으며, 이 경우 UserOp 해시에 영향을 주지 않으므로 Sender와 Paymaster의 서명을 병렬로 수행할 수 있습니다.

## 7. Factory를 통한 지갑 생성

### 7.1 설계 목표

EOA처럼 로컬에서 주소를 생성하고, 실제 배포 전에도 자산을 수신할 수 있어야 합니다.

### 7.2 CREATE2를 이용한 결정론적 주소

`CREATE2` opcode는 다음 입력값으로 미리 계산 가능한 주소에 컨트랙트를 배포합니다:

```
address = keccak256(0xff ++ deployerAddress ++ salt ++ keccak256(initCode))[12:]
```

- `deployerAddress`: Factory 컨트랙트 주소
- `salt`: 32바이트 난수값
- `initCode`: 배포될 컨트랙트의 초기화 코드

### 7.3 Factory 구현 패턴

```solidity
contract SimpleAccountFactory {
    SimpleAccount public immutable accountImplementation;
    IEntryPoint public immutable entryPoint;

    constructor(IEntryPoint _entryPoint) {
        entryPoint = _entryPoint;
        accountImplementation = new SimpleAccount(_entryPoint);
    }

    function createAccount(
        address owner,
        uint256 salt
    ) public returns (SimpleAccount) {
        // counterfactual 주소 계산
        address addr = getAddress(owner, salt);

        // 이미 배포된 경우 기존 주소 반환
        if (addr.code.length > 0) {
            return SimpleAccount(payable(addr));
        }

        // CREATE2로 프록시 배포 + 초기화
        return SimpleAccount(payable(
            new ERC1967Proxy{salt: bytes32(salt)}(
                address(accountImplementation),
                abi.encodeCall(SimpleAccount.initialize, (owner))
            )
        ));
    }

    function getAddress(
        address owner,
        uint256 salt
    ) public view returns (address) {
        return Create2.computeAddress(
            bytes32(salt),
            keccak256(abi.encodePacked(
                type(ERC1967Proxy).creationCode,
                abi.encode(
                    address(accountImplementation),
                    abi.encodeCall(SimpleAccount.initialize, (owner))
                )
            ))
        );
    }
}
```

### 7.4 initCode 인코딩

```solidity
// UserOperation.initCode 구성
bytes memory initCode = abi.encodePacked(
    address(factory),           // Factory 주소 (20 bytes)
    abi.encodeCall(             // Factory에 전달할 calldata
        factory.createAccount,
        (ownerAddress, salt)
    )
);
```

### 7.5 보안 요구사항

- Factory의 `createAccount`은 반드시 `entryPoint.senderCreator()` 주소에서만 호출 가능해야 함
- 생성되는 컨트랙트 주소가 서명에 의존하도록 설계 (누군가가 같은 주소로 다른 지갑을 배포하는 것 방지)
- Factory가 글로벌 스토리지에 접근하면 스테이킹 필요

## 8. Bundler 구현

### 8.1 Bundler의 역할

1. 사용자로부터 UserOperation을 RPC로 수신
2. 기본 검사(sanity check) 수행
3. 시뮬레이션을 통한 유효성 검증
4. 멤풀에 저장 및 다른 Bundler에게 전파
5. UserOperation을 번들링하여 `handleOps` 트랜잭션 전송

### 8.2 기본 검사 (Sanity Check)

UserOperation 수신 시 수행하는 사전 검증:

```
1. sender가 이미 존재하는 컨트랙트이거나, initCode가 비어있지 않아야 함 (동시에 둘 다는 불가)

2. initCode가 있으면 첫 20바이트를 factory 주소로 파싱
   - factory가 스테이킹되어 있는지 기록 (나중에 시뮬레이션에서 필요할 수 있음)

3. verificationGasLimit < MAX_VERIFICATION_GAS (500,000)
   preVerificationGas >= 콜데이터 가스비 + PRE_VERIFICATION_OVERHEAD_GAS (50,000)

4. paymasterAndData가 비어있지 않으면:
   - paymaster가 온체인에 코드가 있는 컨트랙트인지
   - 충분한 예치금이 있는지
   - 차단되지 않았는지

5. callGasLimit >= CALL opcode 비용

6. maxFeePerGas, maxPriorityFeePerGas >= 현재 block.basefee

7. 동일 sender의 UserOp이 멤풀에 이미 없어야 함
   (더 높은 maxPriorityFeePerGas로 대체 가능)
   한 번들에 sender당 하나만 허용 (스테이킹된 sender는 예외)
```

### 8.3 시뮬레이션

번들러는 `EntryPoint.simulateValidation(userOp)`을 로컬에서 RPC 호출하여 유효성을 검증합니다. 이 메소드는 **항상 revert**하며, 결과를 커스텀 에러로 반환합니다:

```solidity
// 성공 시 반환되는 에러
error ValidationResult(
    ReturnInfo returnInfo,
    StakeInfo senderInfo,
    StakeInfo factoryInfo,
    StakeInfo paymasterInfo
);

// aggregator 사용 시
error ValidationResultWithAggregation(
    ReturnInfo returnInfo,
    StakeInfo senderInfo,
    StakeInfo factoryInfo,
    StakeInfo paymasterInfo,
    AggregatorStakeInfo aggregatorInfo
);
```

```solidity
struct ReturnInfo {
    uint256 preOpGas;       // 검증에 사용된 가스
    uint256 prefund;        // 필요한 선납금
    bool    sigFailed;      // 서명 검증 실패 여부
    uint48  validAfter;     // 유효 시작 시간
    uint48  validUntil;     // 유효 종료 시간
    bytes   paymasterContext;// paymaster가 반환한 context
}
```

### 8.4 시뮬레이션 검증 규칙 (Validation Rules)

시뮬레이션 중 `validateUserOp`, `validatePaymasterUserOp`, Factory 호출에서 다음 규칙을 적용합니다:

#### 금지된 Opcode

```
GASPRICE, GASLIMIT, DIFFICULTY, TIMESTAMP, BASEFEE, BLOCKHASH,
NUMBER, SELFBALANCE, BALANCE, ORIGIN, GAS, CREATE, COINBASE, SELFDESTRUCT
```

> `GAS`는 `CALL`, `DELEGATECALL`, `CALLCODE`, `STATICCALL` 바로 앞에서는 허용

#### 스토리지 접근 제한

주소 A와 "연관된 스토리지(associated storage)"만 접근 가능:

1. **A 자신의 스토리지 슬롯**
2. **다른 컨트랙트에서 `mapping(address => value)` 타입의 키가 A인 슬롯**
   - 슬롯 번호: `keccak256(A || storageSlot) + offset`
3. **다른 컨트랙트에서 A와 동일한 번호의 슬롯** (비표준적, 드문 경우)

```
목적: 서로 다른 UserOperation의 검증이 서로 간섭하지 않도록 보장
→ 시뮬레이션에서 성공한 것이 실제 번들링 실행에서도 성공할 것이라는 확신
```

#### 호출 관련 제약

- `CALL`에서 value 전송 불가 (지갑 → EntryPoint 예외)
- out-of-gas로 revert 금지
- 호출 대상에 코드가 있어야 함 (`EXTCODESIZE > 0`)
- EntryPoint의 `depositFor` 외 메소드 호출 금지 (재귀 방지)
- `CREATE2`는 `initCode.length > 0`인 경우에만 허용

### 8.5 번들링 프로세스

```
1. 멤풀에서 UserOperation 선택

2. 충돌 검사:
   - 같은 번들 내 다른 sender의 스토리지에 접근하는 UserOp 제외
   - Factory에 의해 생성되는 다른 지갑 주소에 접근하는 UserOp 제외

3. Paymaster 잔액 추적:
   - 각 Paymaster별 예치금이 번들 내 모든 UserOp의 가스비를 커버하는지 확인

4. Aggregator별 그룹핑 (사용하는 경우):
   - 같은 Aggregator를 사용하는 UserOp을 그룹핑
   - aggregateSignatures()로 압축 서명 생성

5. debug_traceCall로 전체 번들 검증:
   - 최대 가능 가스로 실행
   - revert 시 원인 엔티티 추적 (EntryPoint가 마지막으로 CALL한 엔티티)
   - 문제 UserOp 제거 후 재시도

6. handleOps 트랜잭션 전송
```

### 8.6 Bundler 경제학

번들러의 수익원:
- **가스비 차액**: 사용자가 지정한 `maxPriorityFeePerGas`와 실제 지불한 수수료의 차이
- **MEV**: 번들 내 UserOperation의 순서를 조정하여 추출 가능한 가치
- **번들링 효율**: 여러 UserOp을 하나의 트랜잭션으로 처리하여 기본 가스(21,000)를 분담

### 8.7 Bundler JSON-RPC API

```
eth_sendUserOperation(userOp, entryPoint)
  → UserOperation을 번들러 멤풀에 제출

eth_estimateUserOperationGas(userOp, entryPoint)
  → 가스 추정 (callGasLimit, verificationGasLimit, preVerificationGas)

eth_getUserOperationByHash(hash)
  → UserOperation 조회

eth_getUserOperationReceipt(hash)
  → UserOperation 실행 결과 조회

eth_supportedEntryPoints()
  → 지원하는 EntryPoint 목록
```

## 9. Aggregator (서명 압축)

### 9.1 개요

서로 다른 키로 서명된 다수의 메시지를 하나의 서명으로 압축하여 검증하는 메커니즘입니다. BLS 서명 등이 대표적입니다.

### 9.2 IAggregator 인터페이스

```solidity
interface IAggregator {
    // 개별 UserOp 서명 검증 (번들러가 수신 시 호출)
    function validateUserOpSignature(
        PackedUserOperation calldata userOp
    ) external view returns (bytes memory sigForUserOp);

    // 다수 서명을 하나로 압축 (번들러가 오프체인에서 호출)
    function aggregateSignatures(
        PackedUserOperation[] calldata userOps
    ) external view returns (bytes memory aggregatedSignature);

    // 압축된 서명 검증 (EntryPoint.handleAggregatedOps에서 호출)
    function validateSignatures(
        PackedUserOperation[] calldata userOps,
        bytes calldata signature
    ) external view;
}
```

### 9.3 동작 흐름

```
1. 각 지갑이 getAggregator()로 사용할 Aggregator를 지정

2. Bundler가 같은 Aggregator를 사용하는 UserOp을 그룹핑

3. Bundler가 aggregateSignatures()로 서명 압축

4. handleAggregatedOps() 호출 시:
   - 각 그룹별로 aggregator.validateSignatures() 호출
   - 개별 validateUserOp 대신 그룹 단위 검증
```

### 9.4 UserOpsPerAggregator 구조

```solidity
struct UserOpsPerAggregator {
    PackedUserOperation[] userOps;
    IAggregator aggregator;    // aggregator 주소 (address(0)이면 미사용)
    bytes signature;            // 압축된 서명
}
```

### 9.5 보안 요구사항

- `validateSignatures`는 반드시 EntryPoint에서만 호출 가능
- Aggregator도 스토리지 접근 제한 및 스테이킹 규칙 적용
- 번들러는 Aggregator의 스테이킹 상태와 차단 여부를 확인

## 10. 평판 시스템과 스테이킹

### 10.1 목적

글로벌 엔티티(Paymaster, Factory, Aggregator)는 다수의 UserOperation에서 공유되므로, 하나의 엔티티가 유효성 검사에 실패를 유발하면 여러 UserOperation이 영향을 받습니다.

### 10.2 스테이킹이 필요한 경우

| 엔티티 | 스테이킹 필요 조건 |
|--------|-------------------|
| Factory | 글로벌 스토리지에 접근하는 경우 |
| Paymaster | 자신의 스토리지에 접근하거나 `postOp`이 있는 경우 |
| Aggregator | 항상 필요 |
| Account | sender 자신의 스토리지만 접근하면 불필요 |

### 10.3 스테이킹이 불필요한 경우

- 어떤 스토리지도 참조하지 않는 경우
- sender 계정의 스토리지만 참조하는 경우
- `mapping(address => ...)` 형태로 sender 주소에 연관된 스토리지만 접근하는 경우

### 10.4 평판 추적

번들러는 로컬에서 각 엔티티의 평판을 추적합니다:

```
- 최근 유효성 검사 실패 빈도 추적
- 자주 실패하는 엔티티를 사용하는 UserOp 제한(throttle) 또는 차단(ban)
- 각 번들러가 자체적인 평판 로직 구현 가능
```

### 10.5 Sybil 공격 방지

스테이킹 없이는 공격자가 다수의 엔티티를 저렴하게 생성하여 평판 시스템을 우회할 수 있습니다. 스테이킹을 요구함으로써 공격 비용을 높입니다.

> 슬래싱(몰수)은 없습니다. 스테이킹의 유일한 목적은 Sybil 공격 비용을 높이는 것입니다.

## 11. Nonce 관리

### 11.1 Semi-abstracted Nonce

ERC-4337은 단순한 순차 nonce 대신 **192비트 key + 64비트 sequence** 구조를 사용합니다.

```
uint256 nonce:
┌──────────────────────────────┬────────────────┐
│         key (192 bits)       │ sequence (64b) │
└──────────────────────────────┴────────────────┘
```

### 11.2 활용 예시

#### 순차적 nonce (기존 방식)

```solidity
// key = 0, sequence가 순차 증가
require(userOp.nonce < type(uint64).max);
```

#### 병렬 nonce 채널 (관리/일반 작업 분리)

```solidity
bytes4 sig = bytes4(userOp.callData[0:4]);
uint192 key = uint192(userOp.nonce >> 64);

if (sig == ADMIN_METHODSIG) {
    require(key == ADMIN_KEY, "wrong nonce-key for admin operation");
} else {
    require(key == 0, "wrong nonce-key for normal operation");
}
```

### 11.3 고유성 보장

- 같은 key 내에서 sequence는 EntryPoint가 순차적으로 검증
- 새로운 key는 언제든 임의의 값으로 시작 가능 (sequence = 0)
- 이를 통해 온체인에서 UserOperation 해시의 고유성을 프로토콜 수준에서 보장

## 12. EIP-7702 지원 (v0.7)

### 12.1 개요

EIP-7702가 활성화된 네트워크에서 EOA가 스마트 컨트랙트 코드를 위임(delegate)할 수 있습니다.

### 12.2 동작 방식

- `eth_sendUserOperation`에 `eip7702Auth` 파라미터 추가
- `factory` 필드에 `0x7702` 플래그 사용
- 번들러는 모든 UserOperation의 `eip7702Auth`를 `authorizationList`에 추가
- `SET_CODE_TX_TYPE` 트랜잭션으로 번들 실행

### 12.3 보안 고려사항

- EIP-7702 위임된 Smart Account의 초기화 함수는 `entryPoint.senderCreator()` 주소에서만 호출 가능해야 함
- EntryPoint는 계정이 초기화되었는지 알 수 없으므로, 계정 코드가 한 번만 초기화되도록 보장해야 함
- EIP-7702 authorization 가스 비용(25,000)은 `preVerificationGas`에 포함해야 함

## 13. 가스 추정

### 13.1 preVerificationGas 구성요소

```
preVerificationGas =
    (21,000 / 번들 내 UserOp 수)     // 기본 트랜잭션 가스 분담
  + 콜데이터 가스 (EIP-2028 기준)       // UserOp 직렬화 비용
  + EntryPoint 정적 실행 가스           // 고정 컨트랙트 코드 실행
  + 메모리 확장 비용                     // UserOp 필드를 EVM 메모리에 로드
  + paymaster context 메모리 비용       // (해당 시)
  + innerHandleOp 외부 호출 비용        // EntryPoint 내부 호출
  + EIP-7702 authorization 비용       // (해당 시, 25,000)
  + EIP-7623 calldata floor 증가분     // (해당 시)
```

번들러는 미래 번들의 메모리 확장과 UserOp의 번들 내 위치를 고려하여 여유분(slack)을 요구해야 합니다.

### 13.2 가스비 계산

```solidity
// 실제 가스 가격 결정 (UserOp이 서명한 최대 가격 이하)
function gasPrice(UserOperation calldata userOp) internal view returns (uint256) {
    uint256 maxFeePerGas = userOp.maxFeePerGas;
    uint256 maxPriorityFeePerGas = userOp.maxPriorityFeePerGas;
    if (maxFeePerGas == maxPriorityFeePerGas) {
        return maxFeePerGas; // 레거시 모드
    }
    return min(maxFeePerGas, maxPriorityFeePerGas + block.basefee);
}
```

## 14. 보안 고려사항 체크리스트

### 14.1 Smart Account

- [ ] `validateUserOp` 호출자가 신뢰할 수 있는 EntryPoint인지 확인

- [ ] 이미 배포된 경우 기존 주소 반환

### 15.1 주요 오픈소스 구현체

  - EntryPoint, SimpleAccount, SimpleAccountFactory
  - https://github.com/eth-infinitism/account-abstraction

### 15.2 EntryPoint 배포 주소 (v0.7)

배포 주소 목록은 공식 문서를 참조하세요. EntryPoint는 싱글톤이므로 체인당 하나의 주소를 공유합니다.

### 15.3 개발 순서 권장

```
1단계: Smart Account 구현
  - IAccount 인터페이스 구현
  - 서명 검증 로직 구현
  - 실행 함수 구현
  - UUPS 업그레이드 패턴 적용

2단계: Factory 구현
  - CREATE2 기반 결정론적 배포
  - counterfactual 주소 조회

3단계: Paymaster 구현 (필요한 경우)
  - validatePaymasterUserOp 로직
  - postOp 처리
  - 스테이킹

4단계: Bundler 연동
  - UserOperation 생성 및 서명
  - eth_sendUserOperation RPC 호출
  - 가스 추정 및 수수료 설정

5단계: 테스트
  - 로컬 EntryPoint 배포
  - 시뮬레이션 검증
  - 번들링 E2E 테스트
```

### 15.4 자주 발생하는 실수

| 실수 | 올바른 방법 |
|------|------------|
| `validateUserOp`에서 서명 실패 시 revert | `SIG_VALIDATION_FAILED(1)` 반환 |
| EntryPoint 호출자 검증 누락 | `require(msg.sender == address(entryPoint))` 필수 |
| Factory에서 CREATE 사용 | CREATE2 사용 필수 |
| Paymaster에서 postOp revert 미고려 | 이중 호출 시나리오(postOpReverted) 처리 |
| 업그레이드 시 스토리지 슬롯 충돌 | ERC-7201 또는 gap 패턴 사용 |
| initCode가 있는데 sender가 이미 존재 | initCode는 무시됨 (WARNING) |
| 검증 코드에서 TIMESTAMP 사용 | validAfter/validUntil 반환값 사용 |

## 용어 정리

| 용어 | 설명 |
|------|------|
| **Account Abstraction** | 스마트 컨트랙트를 일급 시민 계정으로 만드는 개념 |
| **UserOperation** | 사용자가 지갑을 통해 수행하려는 작업을 나타내는 데이터 구조 |
| **EntryPoint** | 번들링된 UserOperation을 처리하는 싱글톤 컨트랙트 |
| **Bundler** | UserOperation을 수집하여 번들 트랜잭션을 생성하는 노드 |
| **Paymaster** | 사용자 대신 가스비를 납부하는 스마트 컨트랙트 |
| **Factory** | 새로운 Smart Account를 CREATE2로 배포하는 컨트랙트 |
| **Aggregator** | 다수 서명을 압축하여 한 번에 검증하는 컨트랙트 |
| **Counterfactual Address** | 아직 배포하지 않았지만 배포 시 가지게 될 주소 |
| **Associated Storage** | 특정 주소와 연관된 스토리지 (자신 + mapping 키) |
| **Semi-abstracted Nonce** | 192비트 key + 64비트 sequence 구조의 nonce |
| **Validation Rules** | 시뮬레이션의 정확성을 보장하기 위한 opcode/스토리지 접근 제한 |
| **Reputation System** | 엔티티의 과거 행동을 추적하여 DoS를 방어하는 시스템 |
| **Stake** | 평판 시스템 참여를 위해 잠그는 이더 (슬래싱 없음) |
| **Deposit** | 가스비 지불을 위해 EntryPoint에 예치하는 이더 |
