기준 스펙: ERC-4337 / EntryPoint v0.9 계열 인터페이스

## 1. 결론
- EntryPoint/Account/Paymaster의 기본 인터페이스와 핵심 흐름은 전반적으로 4337 요구사항을 따름.
- 다만, Paymaster 구현에는 스펙 핵심 외의 서비스 확장이 크고, 일부는 번들러 호환성 리스크가 있음.

### 4.3 리스크/불일치 포인트

> **2026-02-25 재검토 갱신**: 항목 1, 2 재분류. 상세 근거는 `EIP-4337_EIP-7579_준수검토_보고서.md` 참조.

1. `validatePaymasterUserOp` 내 상태변경(Verifying/Sponsor) **[BUNDLER COMPAT]**
- 현상:
  - `senderNonce[userOp.sender]++`를 validation 단계에서 수행
  - 위치: `VerifyingPaymaster.sol:147`, `SponsorPaymaster.sol:139`
- 해석:
  - IPaymaster 주석상, 번들러는 상태변경 paymaster를 신뢰 목록으로 제한할 수 있음.
  - 즉, **스펙 위반이라기보다 bundler 호환성 리스크**.
  - **2026-02-25 재검토**: EIP-4337 컨트랙트 인터페이스상 상태 변경 허용. sender-associated storage 규칙 준수. 코드 내 인라인 문서화 완료.
- 권장:
  - 신뢰 paymaster 전제 운영이 아니라면 validation은 가급적 무상태로 유지.
  - 코드 내 `[EIP-4337 Bundler Compatibility Warning]` 주석으로 이미 문서화됨.

2. ~~Envelope nonce 필드 실사용 약함~~ **[NOT AN ISSUE]**
- 현상:
  - envelope에 nonce 필드가 있으나 on-chain에서 강제 비교/소모 정책이 약함(주로 senderNonce와 분리).
- ~~해석~~:
  - ~~4337 MUST 위반은 아니지만 재사용 방지 정책이 문서/SDK와 반드시 정합해야 함.~~
- **2026-02-25 재검토**: 두 nonce는 상호 보완적 메커니즘. envelope nonce는 서명 해시에 포함되어 오프체인 고유성 보장. senderNonce는 온체인 리플레이 방지. 양자 일치 강제 불필요. 의도된 설계.

3. `MIN_DEPOSIT` 상수 미사용 **[CODE QUALITY]**
- 현상: `BasePaymaster.sol:23` 선언 후 강제 체크 미사용
- 해석: 스펙 위반은 아님. 운영 정책 누락 가능성.

## 5. 스펙 준수 / 스펙 외 정리

### 스펙 준수
- Account `validateUserOp` / EntryPoint only-caller / prefund 처리
- EntryPoint `handleOps`, validation/postOp 파이프라인
- Paymaster `validatePaymasterUserOp`, `postOp`, onlyEntryPoint

### 스펙 외(서비스 구현)
- paymasterData envelope(v1), type별 payload ABI
- verifying/sponsor off-chain 서명 정책
- ERC20/Permit2 가격/마크업/토큰 결제
- campaign/analytics/nonce 운영 정책
- `handleAggregatedOps` — 참조 구현 확장(스펙 미정의). 스펙표준 §4.2 (line 144-148) 참조

## 5.5 v0.9 추가 감사 항목

> 아래 항목은 v0.9 스펙 변경 및 신규 보안 고려사항을 반영한 추가 감사 항목이다.

1. **[HIGH] PostOpMode v0.9 변경**
   - v0.9에서 `PostOpMode`는 `opSucceeded`(0)와 `opReverted`(1) 2개 값만 paymaster에 전달된다.
   - `postOpReverted`는 참조 구현 enum에 존재하지만, EntryPoint 내부 제어용이며 `paymaster.postOp()`에 전달되지 않는다.
   - 그러나 `IPaymaster.postOp()`은 public external이므로 EntryPoint 외부에서 직접 호출될 수 있다. 따라서 `postOpReverted` 분기를 포함하는 것은 **방어적 코딩(defensive coding)**으로 유효하며, dead code가 아니다.
   - 참조: 스펙표준 line 342-348

2. **[MEDIUM] Transient Storage (EIP-1153) 정리 의무**
   - 여러 sender의 UserOp이 같은 트랜잭션에서 실행되므로, 민감한 transient storage는 수동으로 정리 MUST.
   - 교차 오퍼레이션 누출 위험 존재.
   - 참조: 스펙표준 line 583-586

3. **[MEDIUM] Storage Layout — ERC-7201**
   - 업그레이드 가능한 계정은 storage layout 충돌 방지 MUST.
   - ERC-7201 diamond storage 패턴 권장.
   - 참조: 스펙표준 line 588-591

4. **[MEDIUM] EIP-7702 호환성**
   - `nonReentrant` 조건 `code.length == 0`이 EIP-7702 delegation이 설정된 EOA도 차단함 (delegation designation으로 code.length=23).
   - 번들러는 순수 EOA로 운용해야 하며, initCode `0x7702` 경로로 EIP-7702 계정 초기화 지원.
   - 참조: 스펙표준 line 114-117, 614-647

5. **[LOW] Block Number Mode (bit 47)**
   - `validUntil`과 `validAfter`의 최상위 비트(bit 47)를 1로 설정하면 timestamp 대신 block number 기준으로 동작.
   - 동일 UserOperation 내에서 timestamp과 block number를 혼용할 수 없다.
   - 참조: 스펙표준 line 219-228

## 6. 수정 우선순위 제안

> **2026-02-25 갱신**: 항목 2 이슈 아님으로 재분류. 항목 1은 BUNDLER COMPAT(코드 문서화 완료).

1. ~~Paymaster validation 단계 상태변경 최소화 여부 결정~~ → **BUNDLER COMPAT** (코드 내 인라인 문서화 완료. Public mempool 대응 시만 추가 작업 필요)
2. ~~envelope nonce 사용 정책을 컨트랙트/SDK/프록시 문서에 일치시켜 명문화~~ → **NOT AN ISSUE** (상호 보완적 설계, 조치 불필요)
3. 필요 시 `MIN_DEPOSIT` 강제 로직 추가 또는 문서에서 정책 제거 — CODE QUALITY, 선택적
4. `handleAggregatedOps` 재분류 — 참조 구현 확장(스펙 미정의)으로 변경 (§3, §5)
5. v0.9 추가 감사 항목 반영 — PostOpMode, Transient Storage, EIP-7702, Block Number Mode (§5.5)
