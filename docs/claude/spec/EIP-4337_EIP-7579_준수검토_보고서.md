# EIP-4337 / EIP-7579 표준 준수 검토 보고서

> **검토 대상**: `poc-contract/src/erc4337-paymaster/`, `poc-contract/src/erc7579-smartaccount/`
> **기준 스펙**: EIP-4337 EntryPoint v0.9, ERC-7579 Modular Smart Account
> **검토 일자**: 2026-02-25
> **EntryPoint**: `0x433709009B8330FDa32311DF1C2AFA402eD8D009` (v0.9)

---

## 1. 검토 개요

본 보고서는 `poc-contract` 내 EIP-4337 Paymaster 및 ERC-7579 Smart Account 코드가 각 표준을 올바르게 준수하는지 검토한다.

기존 검토 문서(감사표, 추가발견사항, 충돌방지 적합성점검, 실행체크리스트)에서 지적된 사항들을 코드와 대조하여 다음을 확인한다:
1. 이전 검토에서 지적한 사항 중 **이미 수정 완료**된 항목
2. 잔존 지적 사항이 **실제로 이슈인지** 여부 (스펙 위반 vs 번들러 호환성 vs 설계 선택)
3. EIP-4337과 ERC-7579 **충돌 지점**의 해결 현황

### 이슈 분류 기준

| 분류 | 의미 |
|------|------|
| **SPEC VIOLATION** | EIP-4337 또는 ERC-7579의 MUST/MUST NOT 요구사항 위반 |
| **BUNDLER COMPAT** | 컨트랙트 스펙 자체는 위반하지 않으나, 번들러 시뮬레이션 규칙에 의해 거부될 수 있는 항목 |
| **DESIGN CHOICE** | 의도적인 설계 결정으로, 코드 내 문서화가 완료된 항목 |
| **NOT AN ISSUE** | 이전 검토에서 지적되었으나 실제로는 문제가 아닌 항목 |
| **CODE QUALITY** | 표준 준수와 무관한 코드 품질 개선 사항 |

## 2. 이전 검토 문서 오류 수정

기존 검토 문서에서 지적한 사항 중, 현재 코드에서 **이미 수정 완료된 항목** 3건:

### 2.1 Permit2Paymaster tokenPrice == 0 검증 누락

- **출처**: `EIP-4337_코드리뷰_추가발견사항.md §2.2` (CRITICAL)
- **현황**: **수정 완료** — `Permit2Paymaster.sol:163`에 `if (tokenPrice == 0) revert InvalidPrice();` 추가됨
- **결론**: 이전 검토 시점 이후 수정됨. 더 이상 이슈 아님.

### 2.2 supportsExecutionMode에서 STATIC 모드 보고

- **출처**: `EIP-4337_7579_충돌방지_적합성점검.md §R1` (FAIL)
- **현황**: **수정 완료** — `Kernel.sol:625`에서 CALLTYPE_BATCH, CALLTYPE_SINGLE, CALLTYPE_DELEGATECALL만 허용. CALLTYPE_STATIC(0xFE)은 ExecLib.execute()에 구현이 없으므로 올바르게 제외됨.
- **결론**: R1 규칙(지원하지 않는 모드를 보고하지 말 것) 충족. FAIL → PASS.

### 2.3 isModuleInstalled에서 Hook(type 4) 미지원

- **출처**: `EIP-7579_4337_준수개선_실행체크리스트.md §P0-2`
- **현황**: **수정 완료** — `Kernel.sol:590-601`에서 MODULE_TYPE_HOOK 분기 추가. rootValidator의 hook을 기본 확인하고, additionalContext로 특정 validator의 hook도 조회 가능.
- **결론**: ERC-7579 `isModuleInstalled` MUST 요구사항 충족. 더 이상 이슈 아님.

## 3. 지적 사항 재검토

기존 검토에서 잔존 이슈로 분류된 항목 7건을 재검토한다. 각 항목에 대해 EIP-4337 스펙 본문, 번들러 시뮬레이션 규칙, 코드 구현을 대조하여 판정한다.

---

### 3.1 VerifyingPaymaster / SponsorPaymaster — senderNonce++ (검증 단계 상태 변경)

| 항목 | 내용 |
|------|------|
| **위치** | `VerifyingPaymaster.sol:147`, `SponsorPaymaster.sol:139` |
| **이전 판정** | CRITICAL — 검증 단계에서 스토리지 쓰기, EIP-4337 위반 |
| **코드** | `senderNonce[userOp.sender]++;` |

**재검토 분석**:

1. **EIP-4337 컨트랙트 인터페이스 관점**: `IPaymaster.validatePaymasterUserOp`는 `view`가 아닌 일반 함수로 정의되어 있다. EntryPoint는 이 함수를 일반 `CALL`로 호출하며, `STATICCALL`이 아니다. 따라서 **컨트랙트 인터페이스 수준에서 상태 변경은 허용된다**.

2. **Associated Storage 규칙**: `senderNonce[userOp.sender]`는 `mapping(address => uint256)` 형태로, 키가 sender 주소이다. EIP-4337의 associated storage 정의에 따르면, sender 주소를 키로 하는 매핑 슬롯은 sender-associated storage에 해당한다. 이는 unstaked paymaster도 접근할 수 있는 범위이다.

3. **번들러 시뮬레이션 관점**: 번들러는 검증 단계를 시뮬레이션하여 UserOp의 유효성을 판단한다. 이때 스토리지 쓰기가 발생하면, 동일 sender의 다른 UserOp가 동일 paymaster를 사용할 경우 시뮬레이션 결과가 달라질 수 있다. 이는 **번들 무효화(bundle invalidation)** 의 원인이 될 수 있으며, 엄격한 public mempool 번들러가 거부할 수 있다.

4. **코드 문서화**: 해당 위치에 `[EIP-4337 Bundler Compatibility Warning — State Change in Validation]` 주석이 상세히 기술되어 있으며, trusted/whitelisted bundler 환경을 전제로 한 설계임을 명시하고 있다.

**판정**: **BUNDLER COMPAT** (이전 CRITICAL에서 하향)

- EIP-4337의 컨트랙트 인터페이스 MUST 요구사항을 위반하지 않음
- Paymaster 자체 스토리지의 sender-associated 슬롯 쓰기로, associated storage 규칙을 위반하지 않음
- Public mempool 번들러에서 거부될 수 있는 번들러 호환성 사항
- 코드 내 충분한 문서화 완료, trusted bundler 환경에서는 문제 없음
- **해결 방안**: 이미 코드 주석에 기술됨 — public mempool 대응 시 nonce를 `_postOp`으로 이동하거나, EntryPoint 내장 nonce에 의존

### 3.2 ERC20Paymaster — balanceOf / allowance 외부 스토리지 읽기

| 항목 | 내용 |
|------|------|
| **위치** | `ERC20Paymaster.sol:226-233` |
| **이전 판정** | HIGH — 검증 단계에서 외부 ERC-20 컨트랙트 스토리지 읽기 |
| **코드** | `IERC20(token).balanceOf(userOp.sender)` / `IERC20(token).allowance(userOp.sender, address(this))` |

**재검토 분석**:

1. **`balanceOf(sender)` 분석**: ERC-20의 `balances[sender]` 슬롯을 읽는다. 이는 `mapping(address => uint256)` 형태로, 키가 sender 주소이다. EIP-4337의 associated storage 정의에 정확히 부합하며, **unstaked paymaster도 접근 가능**한 범위이다.

2. **`allowance(sender, this)` 분석**: ERC-20의 `allowances[sender][spender]` 슬롯을 읽는다. 이는 `mapping(address => mapping(address => uint256))` 중첩 매핑이다. 첫 번째 키는 sender이므로 중간 슬롯(`keccak256(sender . base_slot)`)은 sender-associated이나, 최종 슬롯(`keccak256(spender . intermediate_slot)`)은 spender가 키이다. 엄격하게 해석하면 sender-associated가 아닐 수 있으나, 실무적으로 대부분의 번들러는 이를 허용한다.

3. **프로덕션 참고**: Pimlico, Stackup 등 주요 번들러에서 운영되는 ERC-20 Paymaster들(예: Pimlico ERC-20 Paymaster)은 동일한 패턴을 사용한다. 이는 업계 표준 관행이다.

4. **Staked paymaster의 경우**: Paymaster가 EntryPoint에 스테이킹되어 있으면, associated storage 제약이 완화되어 외부 스토리지 읽기가 더 자유롭게 허용된다.

5. **코드 문서화**: 해당 위치에 `[EIP-4337 Bundler Compatibility Warning — External Storage Read in Validation]` 주석이 상세히 기술되어 있다.

**판정**: **BUNDLER COMPAT** (이전 HIGH에서 하향)

- `balanceOf(sender)`: sender-associated storage로, 스펙 준수
- `allowance(sender, this)`: 중첩 매핑의 해석에 따라 달라지나, 업계 표준 관행이며 staked paymaster로 해결 가능
- EIP-4337 컨트랙트 인터페이스 위반 아님 (읽기 전용, 상태 변경 없음)
- 코드 내 충분한 문서화 완료, 해결 방안(B: staked paymaster)도 명시됨
- **해결 방안**: 이미 코드 주석에 기술됨 — (A) 검증에서 제거 후 `_postOp`에서 실패 의존, 또는 (B) staked paymaster로 운영

### 3.3 Permit2Paymaster — PERMIT2.permit() 외부 상태 변경

| 항목 | 내용 |
|------|------|
| **위치** | `Permit2Paymaster.sol:239` |
| **이전 판정** | CRITICAL — 검증 단계에서 외부 컨트랙트 상태 변경 |
| **코드** | `try PERMIT2.permit(userOp.sender, permitSingle, payload.permitSig) { ... } catch { ... }` |

**재검토 분석**:

1. **핵심 차이점**: 앞선 두 이슈(senderNonce 쓰기, balanceOf 읽기)와 달리, `PERMIT2.permit()`은 **외부 컨트랙트(Permit2)의 스토리지를 직접 변경**한다. Permit2의 allowance nonce와 approval 상태를 수정하며, 이는 paymaster 자체 스토리지도 아니고 sender-associated storage도 아닌 **제3자 컨트랙트의 스토리지 쓰기**이다.

2. **번들러 시뮬레이션 규칙**: EIP-4337의 번들러 시뮬레이션은 검증 단계에서 접근하는 모든 스토리지 슬롯을 추적한다. 외부 컨트랙트에 대한 스토리지 쓰기는 staked paymaster라 해도 시뮬레이션-온체인 불일치를 야기할 수 있으며, 표준 번들러(Stackup, Pimlico, Alchemy 등)에서 거의 확실히 거부된다.

3. **완화 요소**: try-catch 패턴으로 permit 호출이 실패하면 기존 allowance를 확인하는 fallback 경로가 있다(lines 242-251). 즉, 사용자가 이미 Permit2 approval을 완료한 경우, permit 호출 없이도 검증이 통과한다. 이 fallback 경로는 읽기 전용이므로 번들러 호환성 문제가 없다.

4. **코드 문서화**: `[EIP-4337 Bundler Compatibility Warning — External State Change in Validation]` 주석에서 이 문제를 정확히 진단하고, 3가지 해결 방안(A: postOp으로 이동, B: 사전 approve 요구, C: trusted bundler 전용)을 제시하고 있다.

5. **실용적 관점**: 이 paymaster의 주요 사용 시나리오는 (1) 사용자가 이미 Permit2 approve를 가지고 있거나, (2) trusted bundler 환경에서 운영하는 경우이다. 시나리오 (1)에서는 try가 실패하고 catch 경로를 타므로 외부 쓰기가 발생하지 않는다.

**판정**: **BUNDLER COMPAT** (이전 CRITICAL에서 하향하되, 3건 중 가장 심각)

- EIP-4337 컨트랙트 인터페이스 MUST 위반은 아님 (인터페이스는 상태 변경을 허용)
- 그러나 번들러 시뮬레이션 규칙 위반으로, **표준 번들러에서 거부될 확률이 가장 높은 항목**
- try-catch fallback 경로가 있어, 사전 approve된 사용자에 대해서는 문제 없음
- 코드 내 충분한 문서화 및 해결 방안 제시 완료
- **권장 사항**: Public mempool 대응 시 방안 (A) 또는 (B)를 적용할 것. 현재 구현은 trusted bundler 또는 사전 approve 시나리오에 적합.

### 3.4 Envelope nonce와 senderNonce 간 미검증

| 항목 | 내용 |
|------|------|
| **위치** | `PaymasterDataLib.sol:49` (nonce 필드), `VerifyingPaymaster.sol:147` / `SponsorPaymaster.sol:139` (senderNonce) |
| **이전 판정** | MEDIUM — envelope nonce가 on-chain senderNonce와 대조되지 않아 리플레이 위험 |

**재검토 분석**:

1. **Envelope nonce의 역할**: envelope의 `nonce` 필드(uint64)는 `hashForSignature()` 함수에서 서명 해시에 포함된다(`PaymasterDataLib.sol:206`). 전체 envelope 바이트가 `keccak256(envelopeWithoutSig)`로 해시되므로, nonce가 다르면 서명 해시도 달라진다. 이는 **오프체인 서명 고유성**을 보장한다.

2. **senderNonce의 역할**: `senderNonce[userOp.sender]++`는 **온체인 리플레이 방지**를 제공한다. 한 번 실행된 UserOp는 senderNonce가 증가하므로, 동일한 envelope을 재제출해도 서명 검증은 통과하지만 이전에 이미 실행되었기 때문에 재실행될 수 없다.

3. **상호 보완 관계**:
   - `envelope.nonce`: 오프체인에서 동일 sender에 대한 여러 요청을 구분 (서명 해시 다변화)
   - `senderNonce`: 온체인에서 이미 실행된 요청의 재실행 방지
   - 두 메커니즘은 **서로 다른 목적**을 수행하며 독립적으로 동작해야 한다

4. **`env.nonce == senderNonce` 검증이 불필요한 이유**:
   - senderNonce가 이미 온체인 리플레이를 완전히 방지함
   - nonce 일치를 강제하면, SDK가 매 요청마다 온체인 senderNonce를 조회해야 하는 UX 부담 발생
   - 현재 설계에서 signer는 자유로운 nonce 체계를 사용 가능 (타임스탬프 기반, 랜덤 등)

**판정**: **NOT AN ISSUE**

- 두 nonce 메커니즘은 상호 보완적이며, 각각 올바르게 동작함
- envelope nonce는 서명 해시에 포함되어 오프체인 고유성 보장
- senderNonce는 온체인 리플레이 방지 보장
- 양자를 일치시킬 필요가 없으며, 일치 강제 시 오히려 UX 저하
- 리플레이 공격 시나리오에서 두 메커니즘 모두 방어 기능을 수행함

### 3.5 Delegatecall 대상 화이트리스트 부재

| 항목 | 내용 |
|------|------|
| **위치** | `ExecLib.sol:48-59` |
| **이전 판정** | MEDIUM — delegatecall 대상에 대한 온체인 화이트리스트 미구현 |

**재검토 분석**:

1. **ERC-7579 스펙**: `CALLTYPE_DELEGATECALL (0xFF)`은 **OPTIONAL** 모드로 정의되어 있으며, 구현 시 주의사항을 경고하지만 화이트리스트를 MUST 요구하지 않는다.

2. **EIP-4337 스펙**: 실행 단계에서의 delegatecall에 대해 특별한 제약을 두지 않는다. 검증 단계에서의 opcode 제약(CREATE, SELFDESTRUCT 등)은 있으나, 실행 단계는 자유이다.

3. **현재 접근 제어 모델**: Delegatecall은 다음 경로를 통해서만 도달 가능하다:
   - `Kernel.execute()` → `onlyEntryPointOrSelfOrRoot` 모디파이어 → EntryPoint(4337 흐름), self(내부 호출), 또는 root authority만 허용
   - `Kernel.executeFromExecutor()` → 등록된 executor만 허용 (hook 검증 포함)
   - `Kernel.executeUserOp()` → `onlyEntryPoint` → EntryPoint만 허용
   - 모든 경로에서 **caller 수준의 권한 검증**이 이루어짐

4. **코드 문서화**: ExecLib.sol에 `[EIP-4337 / ERC-7579 Spec Conflict — Delegatecall Mode Security]` 주석이 상세히 기술되어 있으며, 현재 모델의 한계와 향후 개선 방향(registry-based 제한)을 명시하고 있다.

**판정**: **DESIGN CHOICE** (이전 MEDIUM에서 재분류)

- EIP-4337, ERC-7579 어디에도 delegatecall 대상 화이트리스트를 MUST로 요구하지 않음
- Caller 수준 접근 제어(validator/executor 권한 모델)가 이미 구현되어 있음
- 온체인 화이트리스트는 **보안 강화 제안**이지 스펙 준수 요구사항이 아님
- 코드 내 설계 결정이 문서화되어 있음

### 3.6 BasePaymaster — MIN_DEPOSIT / InsufficientDeposit 미사용

| 항목 | 내용 |
|------|------|
| **위치** | `BasePaymaster.sol:23, 26` |
| **이전 판정** | LOW — dead code |

**재검토 분석**:

- `MIN_DEPOSIT = 0.01 ether`는 선언만 되어 있고 `deposit()` 등에서 검증에 사용되지 않음
- `InsufficientDeposit` error도 선언만 되어 있고 `revert`에 사용되지 않음
- EIP-4337 또는 ERC-7579의 어떤 MUST 요구사항과도 무관

**판정**: **CODE QUALITY**

- 표준 준수와 무관한 dead code
- 삭제하거나, 향후 deposit 최소 금액 검증에 활용 가능
- 스펙 준수에 영향 없음

---

### 3.7 VerifyingPaymaster._postOp가 pure 선언

| 항목 | 내용 |
|------|------|
| **위치** | `VerifyingPaymaster.sol:162` |
| **이전 판정** | LOW — postOp가 pure이면 상태 접근 불가 |

**재검토 분석**:

1. **Solidity 상속 규칙**: 외부 인터페이스 `postOp`는 BasePaymaster에서 non-view, non-pure로 선언됨. 내부 `_postOp`가 pure여도, 외부 `postOp` 함수의 가시성에는 영향 없음.

2. **의도된 설계**: VerifyingPaymaster는 단순 서명 검증 paymaster로, post-op에서 할 작업이 없다. pure 선언은 이를 명시적으로 표현하며, 가스 효율도 약간 향상된다.

3. **확장성**: 향후 post-op 로직이 필요하면 pure 제거만 하면 된다. 현재 시점에서는 올바른 구현이다.

**판정**: **NOT AN ISSUE**

- Solidity에서 pure 함수는 non-pure 인터페이스의 유효한 구현임
- EIP-4337은 postOp의 상태 접근을 요구하지 않음 (선택적)
- 의도적이고 올바른 구현

## 4. 신규 발견 사항 (2026-02-25 2차 검토)

### 4.1 Fallback 모듈 sender context 불일치 (R5)

| 항목 | 내용 |
|------|------|
| **위치** | `ExecLib.sol:228-252` (20바이트 append) vs `FlashLoanFallback.sol:451-457`, `TokenReceiverFallback.sol:381-387` (40바이트 기대) |
| **판정** | **구현 버그** — 모듈 설치와 사용 사이 스토리지 키 불일치 |

**문제 설명**:

Kernel의 `doFallback2771Call`은 ERC-2771 표준에 따라 `caller()` 주소 **20바이트**를 calldata 끝에 append한다. 그러나 두 Fallback 모듈(`FlashLoanFallback`, `TokenReceiverFallback`)의 `_extractContext()`는 **40바이트** `[originalCaller:20][smartAccount:20]`를 기대한다.

**영향**:

| 단계 | 주소 키 | 저장/조회 위치 |
|------|---------|---------------|
| `onInstall()` | `msg.sender` = **Kernel** | `accountStorage[Kernel]` ← config 저장 |
| `addToWhitelist()` | `msg.sender` = **Kernel** | `accountStorage[Kernel].protocolWhitelist` ← 저장 |
| 콜백 (`executeOperation` 등) | `_extractContext().smartAccount` = **caller(AAVE 등)** | `accountStorage[AAVE]` ← 조회 → **항상 false** |

결과: 설치된 모든 Fallback 모듈의 프로토콜 콜백이 `ModuleNotEnabled()` 또는 `ProtocolNotWhitelisted()`로 revert.

**영향 범위**:
- `FlashLoanFallback`: `executeOperation`, `uniswapV3FlashCallback`, `receiveFlashLoan`, `onFlashLoan`
- `TokenReceiverFallback`: `onERC721Received`, `onERC1155Received`, `onERC1155BatchReceived`, `tokensReceived`

**수정 방향** (택 1):
- **옵션 A**: `ExecLib.doFallback2771Call`에서 `caller() + address(this)` 40바이트 append
- **옵션 B**: Fallback 모듈의 `_extractContext()`를 20바이트 기반으로 변경, `msg.sender`(=Kernel)를 smartAccount로 사용

### 4.2 R9 시간 유효성 모드 일관성 — 이슈 아님

- 모든 validator/paymaster가 timestamp 모드만 사용 확인
- EntryPoint v0.9의 `VALIDITY_BLOCK_RANGE_FLAG`를 설정하는 컴포넌트 없음
- block.number 모드 혼용 없음 → **PASS**

### 4.3 R10 EIP-7702 초기화 경로 — 구현 완료

- `Eip7702Support.sol`, `SenderCreator.initEip7702Sender`, `ValidationManager.VALIDATION_TYPE_7702` 구현 확인
- initCode `0x7702` prefix 감지 → 초기화 경로 동작 → **PASS**

### 결론

> **2026-02-25 2차 검토 갱신**: 구현 버그 1건 신규 발견 (Fallback sender context).

1. **EIP-4337 / ERC-7579 컨트랙트 수준 MUST 위반: 0건** — 모든 인터페이스 요구사항이 올바르게 구현되어 있다.

2. **구현 버그: 1건** (§4.1) — Kernel의 ERC-2771 sender append(20바이트)와 Fallback 모듈의 `_extractContext()`(40바이트 기대)가 불일치하여, 설치된 Fallback 모듈의 프로토콜 콜백이 항상 실패함. ERC-7579 스펙이 fallback sender 형식을 명시적으로 정의하지 않으므로 MUST 위반은 아니나, **모듈 설치-사용 간 스토리지 키 일관성이 깨지는 기능적 버그**이다.

3. **번들러 호환성 사항: 3건** (§3.1, §3.2, §3.3) — 이들은 EIP-4337 컨트랙트 인터페이스 위반이 아니라, 번들러 시뮬레이션 정책에 의한 운영 수준 제약이다. 모든 항목에 대해 코드 내 상세한 인라인 문서화가 완료되어 있으며, trusted bundler 환경 또는 staked paymaster로 해결 가능하다.

4. **충돌 해결: 11/12 PASS, 1 FAIL** — C2(Fallback sender context)에서 Kernel과 Fallback 모듈 간 프로토콜 불일치 발견. 나머지 11개 충돌 지점은 인라인 주석으로 해결 방식이 문서화되어 있다.

5. **이전 검토 문서 오류: 3건 수정 완료** — tokenPrice 검증, STATIC 모드 제외, Hook isModuleInstalled이 이미 코드에 반영되어 있다.
