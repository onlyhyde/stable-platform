## 3. 주요 이슈 상세

### 3.1 [P0] ~~Bundler `userOpHash` 계산 방식이 v0.9 EIP-712 기준과 불일치~~ — **수정 완료**

#### 관찰

- 문서는 EntryPoint v0.9 + EIP-712 해시를 기준으로 명시
  - `EIP-4337_스펙표준_정리.md:5`
  - `EIP-4337_스펙표준_정리.md:697`
  - `EIP-4337_스펙표준_정리.md:724`
- Bundler RPC 유틸은 v0.7 계열 해시(`keccak(abi.encode(userOpHash, entryPoint, chainId))`) 사용
  - `services/bundler/src/rpc/utils.ts:158`
  - `services/bundler/src/rpc/utils.ts:192`
  - `services/bundler/src/rpc/server.ts:382`

#### 영향

- RPC 반환 `userOpHash`와 on-chain 이벤트 해시가 달라질 가능성.
- `eth_getUserOperationByHash`, `eth_getUserOperationReceipt`, 정산 추적, 운영 모니터링 정합성에 직접 영향.

#### 권고

1. `services/bundler/src/rpc/utils.ts#getUserOperationHash`를 제거/위임하고 `@stablenet/core`의 v0.9 EIP-712 구현을 단일 사용.
2. Bundler e2e 테스트에서 on-chain `EntryPoint.getUserOpHash()`와 RPC hash가 항상 일치하는 어설션 추가.

---

### 3.2 [P1] ~~`eth_getUserOperationReceipt`가 mempool 엔트리 의존(온체인 fallback 부재)~~ — **수정 완료**

#### 관찰

- `eth_getUserOperationReceipt`는 mempool 엔트리가 없거나 상태가 `included`가 아니면 `null` 반환
  - `services/bundler/src/rpc/server.ts:530`
  - `services/bundler/src/rpc/server.ts:531`
- 반면 `eth_getUserOperationByHash`는 on-chain 로그 fallback 탐색 구현이 있음
  - `services/bundler/src/rpc/server.ts:473`
- Paymaster settlement worker는 receipt API 결과에 의존
  - `services/paymaster-proxy/src/settlement/bundlerClient.ts:56`
  - `services/paymaster-proxy/src/settlement/settlementWorker.ts:121`

#### 영향

- Bundler 재시작/메모리 유실/해시 불일치 시 receipt를 장기간 회수하지 못할 수 있음.
- Reservation settle/cancel이 지연되어 회계 TTL 만료 의존도가 높아짐.

#### 권고

1. `eth_getUserOperationReceipt`에도 `eth_getUserOperationByHash`와 동일한 on-chain fallback 경로 추가.
2. fallback으로 찾은 이벤트 기준으로 receipt 구조를 재구성해 반환하도록 보강.

---

### 3.3 [P1] ~~EntryPoint 기본값 체계가 v0.7 중심으로 남아 문서와 불일치~~ — **수정 완료**

#### 관찰

- 문서에는 v0.9 핀닝 요구가 명시됨
  - `EIP-4337_스펙표준_정리.md:5`
  - `EIP-4337_7579_통합_스펙준수_보고서.md:626`
- 다수 패키지 기본 상수/주석/테스트가 v0.7 기준
  - `packages/contracts/src/addresses.ts:69`
  - `packages/config/src/entryPoints.ts:31`
  - `packages/wallet-sdk/src/entrypoint/index.ts:4`
  - `packages/sdk-go/core/constants.go:10`
  - `services/paymaster-proxy/src/config/constants.ts:145`
  - `apps/web/lib/config/env.ts:116`
  - `packages/types/tests/index.test.ts:17`
  - `packages/config/tests/index.test.ts:62`

#### 영향

- 환경 변수/체인별 override 누락 시 문서 의도와 다른 EntryPoint 주소/버전으로 실행될 수 있음.
- 세미나 자료와 실제 기본 동작이 어긋나 신뢰도 저하 가능.

#### 권고

1. 기본값 정책을 명시적으로 선택:
   - 옵션 A: v0.9 only
   - 옵션 B: v0.7/v0.9 듀얼 지원(체인/모드별)
2. 선택된 정책에 맞춰 `packages/*`, `services/*`, `apps/*`, 테스트/README를 동시 정리.

### 3.4 [P2] ~~가스/비용 추정이 문서의 v0.9 운영 주의사항을 충분히 반영하지 못함~~ — **수정 완료**

#### 관찰

- 문서는 다음 주의사항을 명확히 요구
  - 10% 미사용 가스 페널티 인지: `EIP-4337_Paymaster_개발자_구현가이드.md:244`
  - 운영 공통 규칙: `EIP-4337_7579_통합_스펙준수_보고서.md:631`
  - EIP-7702 preVerificationGas 가산(해당 시): `EIP-4337_스펙표준_정리.md:767`
- Paymaster 프록시 추정식은 `call + verification + preVerification`만 사용
  - `services/paymaster-proxy/src/utils/gasEstimator.ts:7`
- Bundler preVerification 계산 경로에서 7702 고정 가산(+25k) 분기 확인 불가
  - `services/bundler/src/gas/gasEstimator.ts:174`

#### 영향

- sponsorship/token payment 추정치 과소 산정 가능.
- 정책 reserve와 실제 settle 간 편차 증가 가능.

#### 권고

1. Paymaster 비용 추정 유틸에 `paymasterVerificationGasLimit`, `paymasterPostOpGasLimit` 포함.
2. 운영 모드별(7702 사용 여부) preVerification 가산 정책을 코드/문서에 동일 반영.
3. 문서에 있는 페널티 규칙을 실제 추정기 로직과 1:1 대응시키는 회귀 테스트 추가.

### 3.5 [P2] ~~통합 보고서 PASS 판정 일부가 현재 오프체인 구현 상태를 과대평가~~ — **수정 완료**

#### 관찰

- 보고서는 오프체인 정합성 및 Bundler 검증 파이프라인을 PASS 중심으로 기술
  - `EIP-4337_7579_통합_스펙준수_보고서.md:710`
  - `EIP-4337_7579_통합_스펙준수_보고서.md:741`
- 그러나 본 검토의 P0/P1 이슈(해시 계산, receipt fallback, 기본값 체계)가 반영되지 않음.

#### 영향

- "현재 구현 완성도"에 대한 독자 인식 오차 발생.
- 세미나 자료로 전환 시 질의응답에서 불필요한 혼선 가능.

#### 권고

1. 통합 보고서의 오프체인 섹션 PASS/주의 상태를 재평가해 즉시 반영.
2. "컨트랙트 레이어 PASS / 오프체인 레이어 보완 필요"를 분리 서술.

## 4. 정합성이 확인된 항목

다음 항목은 코드와 문서 방향이 대체로 일치함.

- EntryPoint 컨트랙트의 v0.9 핵심 기능 존재
  - `poc-contract/src/erc4337-entrypoint/EntryPoint.sol:166` (`getUserOpHash`)
  - `poc-contract/src/erc4337-entrypoint/EntryPoint.sol:487` (EIP-7702 init 경로)
  - `poc-contract/src/erc4337-entrypoint/EntryPoint.sol:503` (`IgnoredInitCode`)
- Kernel의 4337/7579 브릿지 핵심 경로 존재
  - `poc-contract/src/erc7579-smartaccount/Kernel.sol:328` (`validateUserOp`, onlyEntryPoint)
  - `poc-contract/src/erc7579-smartaccount/Kernel.sol:409` (`executeUserOp`, onlyEntryPoint)
  - `poc-contract/src/erc7579-smartaccount/Kernel.sol:879` (`supportsExecutionMode`)

> **참고**: 위 Kernel.sol 라인 번호는 현재 코드 기준이다. 통합 보고서(`EIP-4337_7579_통합_스펙준수_보고서.md`)의 Kernel.sol 라인 참조도 현재 코드 기준으로 **갱신 완료**(2026-03-02)되어 양 문서 간 라인 번호가 일치한다.

## 5. 우선순위 실행 계획 (제안)

1주차(긴급):

1. Bundler `userOpHash` 계산을 v0.9 EIP-712 단일 경로로 교체.
2. `eth_getUserOperationReceipt` 온체인 fallback 추가.
3. Paymaster settlement e2e 테스트로 해시/receipt 연계 검증.

2주차(정리):

1. EntryPoint 기본값 정책(v0.9 only vs dual) 결정 및 전 패키지 일괄 반영.
2. Paymaster/Bundler 가스 추정식에 문서 규칙(페널티/7702) 반영.
3. 통합 보고서 PASS/주의/리스크 섹션 재작성.
