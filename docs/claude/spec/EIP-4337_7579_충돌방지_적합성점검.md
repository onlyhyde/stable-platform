### ~~R5 PARTIAL~~ R5 FAIL: Fallback sender 문맥 **[구현 버그 발견]**

**2026-02-25 정밀 점검 결과**:

Kernel과 Fallback 모듈 간 ERC-2771 sender context 프로토콜이 불일치함.

**Kernel 측** (`ExecLib.doFallback2771Call`, lines 228-252):
- `caller()` 주소 **20바이트**만 calldata 끝에 append
- 이는 표준 ERC-2771 sender forwarding과 동일

**Fallback 모듈 측** (`FlashLoanFallback._extractContext()` line 451, `TokenReceiverFallback._extractContext()` line 381):
- calldata 끝에서 **40바이트** 읽기: `[originalCaller:20][smartAccount:20]`
- 모듈 NatSpec에 "The smart account appends 40 bytes"로 명시

**불일치 영향** (Flash Loan 콜백 예시):
1. 설치: `onInstall()` → `accountStorage[Kernel].config.isEnabled = true`
2. 화이트리스트: `addToWhitelist(AAVE)` → `accountStorage[Kernel].protocolWhitelist[AAVE] = true`
3. 콜백: AAVE → Kernel fallback → `doFallback2771Call` → 20바이트(AAVE 주소) append
4. `_extractContext()`: `smartAccount` = AAVE 주소(❌), `originalCaller` = ABI params 잔여 바이트(❌)
5. `_validateProtocol(AAVE, garbage)` → `accountStorage[AAVE].config.isEnabled` → **false** → `revert ModuleNotEnabled()`

**영향 범위**:
- `FlashLoanFallback`: `executeOperation`, `uniswapV3FlashCallback`, `receiveFlashLoan`, `onFlashLoan` — 4개 콜백 모두 영향
- `TokenReceiverFallback`: `onERC721Received`, `onERC1155Received`, `onERC1155BatchReceived`, `tokensReceived` — 4개 콜백 모두 영향

**수정 방향** (택 1):
- **A**: `ExecLib.doFallback2771Call`에서 `caller() + address(this)` 40바이트 append → 모듈 기대와 일치
- **B**: Fallback 모듈의 `_extractContext()`를 20바이트 기반으로 변경, `msg.sender`(=Kernel)를 smartAccount로 사용

**ERC-7579 스펙 관점**:
- ERC-7579는 fallback 모듈에 대한 sender propagation 형식을 명시적으로 MUST 정의하지 않음
- 그러나 모듈 설치(`installModule`/`onInstall`)와 실제 사용(콜백) 사이의 스토리지 키 일관성은 **모듈 시스템의 기본 계약**이며, 이 불일치로 인해 설치된 모듈이 사용 불가능한 상태

### ~~R8 NOT_CHECKED~~ R8 PASS (DESIGN CHOICE): delegatecall mode 신뢰 검증 **[점검 완료]**

**2026-02-25 점검 결과**:
- `ExecLib.sol:48-59`에 `[EIP-4337 / ERC-7579 Spec Conflict — Delegatecall Mode Security]` 인라인 주석으로 보안 경고 및 설계 결정이 상세히 문서화됨
- `supportsExecutionMode`에서 CALLTYPE_DELEGATECALL(0xFF) 지원 보고 (`Kernel.sol:625`) → ExecLib.execute()에 delegatecall 분기 구현됨 (`ExecLib.sol:48-70`) → **정합**
- Caller 수준 접근 제어 구현: `Kernel.execute()` → onlyEntryPointOrSelfOrRoot, `Kernel.executeFromExecutor()` → 등록된 executor + hook 검증, `Kernel.executeUserOp()` → onlyEntryPoint
- 온체인 target 화이트리스트는 EIP-4337/ERC-7579의 MUST 요구사항이 아님. ERC-7579는 delegatecall을 OPTIONAL로 정의하며 주의사항만 경고
- **판정**: 의도적 설계 선택. 코드 내 문서화 완료. 향후 보안 강화를 위한 registry-based 제한은 개선 제안으로 유효

### ~~R9 PARTIAL~~ R9 PASS: 시간 모드 일관성 **[점검 완료]**

**2026-02-25 점검 결과**:

모든 컴포넌트에서 timestamp 모드만 사용. block.number 모드 혼용 없음.

- **EntryPoint v0.9**: dual-mode 지원 (`VALIDITY_BLOCK_RANGE_FLAG = 0x800_000_000_000`). MSB가 설정되면 block.number, 아니면 block.timestamp로 해석
- **Validators**: 5개 모두 점검 완료
  - ECDSA/MultiChain/MultiSig/WebAuthn: time-range 미설정 (0/1만 반환)
  - WeightedECDSA: `block.timestamp + delay`로 `validAfter` 구성 → timestamp 모드 (MSB 미설정)
- **Paymasters**: 4개 모두 `_packValidationData(sigFailed, validUntil, validAfter)` 사용. `validUntil`/`validAfter`는 envelope에서 그대로 전달 (오프체인 signer가 timestamp로 설정해야 함)
- **ValidationManager**: `_checkPermissionSignature`에서 `block.timestamp` 비교 사용 (ERC-1271 경로)

**판정**: block.number 모드를 활성화하는 컴포넌트 없음. `VALIDITY_BLOCK_RANGE_FLAG`는 코드베이스 전체에서 설정되지 않음.

비고: 오프체인 SDK가 paymaster envelope의 `validUntil`/`validAfter`를 Unix timestamp로 설정해야 하는 규칙은 SDK 문서에 명시 필요.

### ~~R10 NOT_CHECKED~~ R10 PASS: EIP-7702 초기화 정합성 **[점검 완료]**

**2026-02-25 점검 결과**:

EIP-7702 경로가 EntryPoint v0.9 및 Kernel 양측에 완전히 구현되어 있음.

**EntryPoint 측**:
- `Eip7702Support.sol`: `_isEip7702InitCode()`, `_getEip7702Delegate()`, `_getEip7702InitCodeHashOverride()` 구현
- `SenderCreator.sol`: `initEip7702Sender(address sender, bytes memory initCallData)` — sender를 직접 call하여 초기화
- `EntryPoint.sol:481-518` (`_createSenderIfNeeded`): initCode가 `0x7702` prefix이면 EIP-7702 경로로 분기, `EIP7702AccountInitialized` 이벤트 발행

**Kernel 측**:
- `ValidationManager.sol`: `VALIDATION_TYPE_7702` 분기 구현. `ECDSA.recover(hash, sig) == address(this)` 검증 (EOA 서명)
- `Constants.sol`: `VALIDATION_TYPE_7702 = ValidationType.wrap(0x00)`, `EIP7702_PREFIX = bytes3(0xef0100)` 정의

**초기화 흐름**:
1. EOA가 EIP-7702 authorization으로 code를 Kernel에 위임 (오프체인)
2. UserOp의 `initCode = 0x7702 || [calldata]`
3. EntryPoint가 `0x7702` marker 감지 → `SenderCreator.initEip7702Sender(sender, calldata)` 호출
4. sender(EOA with Kernel code)에서 calldata 실행 → Kernel 초기화 (storage 설정, 모듈 설치)
5. 이후 validation에서 `VALIDATION_TYPE_7702`로 EOA 서명 검증

**판정**: 4개 점검 항목 모두 충족
- ✅ 7579 필수 모듈 설치: initCallData를 통해 `installModule`/`onInstall` 호출 가능
- ✅ `entryPoint.senderCreator()`에서만 초기화 호출
- ✅ `EIP7702AccountInitialized` 이벤트 발행 구현
- ⚠️ EIP-7702 미지원 네트워크 fallback: factory 기반 경로가 별도 존재하나, 동일 결과 보장에 대한 통합 테스트 권장

### v0.9 반영 관찰
- **C11 (IgnoredInitCode)**: v0.9에서 이미 배포된 계정에 initCode 제공 시 revert 대신 무시(`IgnoredInitCode` 이벤트 발행). 7579 계정의 초기 모듈 설치가 factory → initCode 경로에 묶여 있는 경우, 재배포 시도 시 모듈이 설치되지 않는 사일런트 실패 가능. R1에 반영 완료.
- **C12 (EIP-7702 초기화)**: EIP-7702 authorization 기반 초기화는 기존 factory 경로와 다른 호출 체인을 사용. 7579 `installModule`/`onInstall`이 이 경로에서 호출되는지 별도 검증 필요. R10으로 신설.
- **`delegateAndRevert` 시뮬레이션**: state override 미지원 네트워크에서 사용하는 v0.9 시뮬레이션 헬퍼. delegatecall 기반이므로 hook 동작이 기존 view/trace call과 다를 수 있음. R4에 SHOULD 규칙으로 반영 완료.
- **EIP-712 헬퍼**: `getPackedUserOpTypeHash()`, `getDomainSeparatorV4()`가 v0.9에서 public 조회 가능. 오프체인 SDK에서 hash 계산 일관성 검증에 활용 권장. R3에 반영 완료.
