## 1. 결론
- ERC-7579 핵심 구조(Account + Module lifecycle + type 분리)는 전체적으로 구현되어 있음.
- ~~**스펙 준수 관점의 핵심 불일치 2건**은 여전히 존재함.~~
  1. ~~`supportsExecutionMode`와 실제 실행 경로 불일치 (STATIC)~~ → **RESOLVED** (2026-02-25)
  2. ~~`isModuleInstalled`에서 Hook(type 4) 조회 누락~~ → **RESOLVED** (2026-02-25)

> **2026-02-25 갱신**: 2건 수정 완료.
> **2026-02-25 2차 갱신**: Fallback sender context 구현 버그 1건 신규 발견. Kernel의 20바이트 ERC-2771 append와 Fallback 모듈의 40바이트 기대 불일치로, 설치된 Fallback 모듈의 프로토콜 콜백이 항상 실패함. ERC-7579 MUST 위반(인터페이스)은 아니나 **모듈 설치-사용 간 일관성 버그**.

### Account 이슈 상세

> **2026-02-25 갱신**: 2건 수정 완료.
> **2026-02-25 2차 갱신**: Fallback sender context 구현 버그 1건 신규 발견.

1. ~~`supportsExecutionMode`가 `CALLTYPE_STATIC`을 true로 반환하지만 `ExecLib.execute`에 static 분기 없음.~~ → **RESOLVED**: `Kernel.sol:625`에서 STATIC 제외됨.
2. ~~`supportsModule(4)`는 true지만 `isModuleInstalled(4, ...)`는 false 경로로 빠짐.~~ → **RESOLVED**: `Kernel.sol:590-601`에서 Hook(type 4) 조회 구현됨.
3. **Fallback sender context 불일치** — `ExecLib.doFallback2771Call`은 `caller()` 20바이트 append. `FlashLoanFallback._extractContext()` / `TokenReceiverFallback._extractContext()`는 40바이트 `[originalCaller:20][smartAccount:20]` 기대. `onInstall` 시 `accountStorage[Kernel]`에 저장하지만, 콜백 시 `accountStorage[caller]`로 조회하여 항상 `ModuleNotEnabled()` revert. 상세: `EIP-4337_EIP-7579_준수검토_보고서.md §4.1`

## 3. Validators 감사 (erc7579-validators)

대상:
- `ECDSAValidator.sol`
- `MultiChainValidator.sol`
- `MultiSigValidator.sol`
- `WebAuthnValidator.sol`
- `WeightedECDSAValidator.sol`

공통 점검 결과:
- `onInstall`/`onUninstall`/`isModuleType` 구현: 모두 확인
- `validateUserOp`/`isValidSignatureWithSender` 구현: 모두 확인
- module type 반환:
  - `MultiSig/WebAuthn/Weighted`: validator(1)만 반환
  - `ECDSA/MultiChain`: validator(1) + hook(4) 겸용 반환

판정: **준수(확장 허용 범위 내)**

비고:
- `ECDSAValidator`, `MultiChainValidator`는 IHook까지 겸해 type 4를 반환하며, 이는 ERC-7579에서 다중 타입 모듈 허용 범위에 해당.

## 5. Fallbacks 감사 (erc7579-fallbacks)

대상:
- `FlashLoanFallback.sol`
- `TokenReceiverFallback.sol`

공통 점검 결과:
- `IFallback` 구현 + lifecycle 함수 구현 확인
- `isModuleType`는 모두 `MODULE_TYPE_FALLBACK(3)` 반환

판정: **인터페이스 준수, 단 sender context 구현 버그 있음**

> **2026-02-25 2차 갱신**: 두 모듈 모두 `_extractContext()`에서 40바이트(originalCaller + smartAccount) 읽기를 기대하지만, Kernel의 `doFallback2771Call`은 20바이트(caller)만 append함. 이로 인해 `onInstall` 시 `accountStorage[Kernel]`에 저장된 config를 콜백 시 조회할 수 없음. 모든 프로토콜 콜백(`executeOperation`, `onERC721Received` 등)이 `ModuleNotEnabled()`로 revert. 상세: `EIP-4337_EIP-7579_준수검토_보고서.md §4.1`
