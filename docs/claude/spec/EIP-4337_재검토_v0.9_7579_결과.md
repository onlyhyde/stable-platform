## 1. 결론
- EntryPoint 코어 플로우는 v0.9 기준으로 전반적으로 정합함.
- Account(Kernel)는 EIP-4337 브리지(`validateUserOp`, `executeUserOp`)와 ERC-7579 모듈 구조를 결합하고 있으며, 과거 지적된 일부 충돌 지점은 이미 수정됨.
- 현재 핵심 미해결 이슈는 **Paymaster validation 단계의 상태변경/외부 상태접근** 계열로, 퍼블릭 bundler 호환성 리스크가 큼.

### 2.2 여전히 미해결
1. Permit2Paymaster validation 중 외부 상태변경 호출
- 심각도: CRITICAL
- 현상: `_validatePaymasterUserOp`에서 `PERMIT2.permit(...)` 수행
- 코드: `/Users/wm-it-22-00661/Work/github/stable-net/poc/poc-contract/src/erc4337-paymaster/Permit2Paymaster.sol:239`

  - 이 구현은 신뢰/화이트리스트 bundler 전제 없이는 운영 리스크가 높음.

2. Verifying/Sponsor Paymaster validation 중 상태변경

- 현상: `senderNonce[userOp.sender]++`를 validation 단계에서 수행
- 코드:

- 평가:

## 5. 스펙 준수 / 스펙 예외 분류 (현재 코드 기준)

### 5.1 스펙 준수로 볼 수 있는 범위
- EntryPoint v0.9 코어 흐름(`handleOps`, `handleAggregatedOps`, `getUserOpHash`, prepayment, postOp)
- Account의 `validateUserOp` / `executeUserOp` bridge 구조
- Paymaster 기본 인터페이스(`validatePaymasterUserOp`, `postOp`, onlyEntryPoint)

### 5.2 스펙 예외(서비스 확장)이며 운영정책 필요
- Envelope 기반 paymasterData 포맷 및 type별 payload ABI
- 캠페인/nonce/오라클/마크업 등 비즈니스 정책
- Permit2 연동 방식

### 5.3 스펙 예외 중에서도 퍼블릭 bundler 호환성 저해 가능 항목
- Permit2 validation 중 `permit()` 실행
- Verifying/Sponsor validation 중 `senderNonce++`
- ERC20 validation 중 외부 token storage 조회

## 6. 수정 우선순위 제안
1. Permit2Paymaster: validation 단계의 `PERMIT2.permit()` 제거 또는 실행 단계로 이동
2. Verifying/Sponsor: `senderNonce` 소모 시점을 validation 밖으로 이동
3. ERC20Paymaster: validation의 balance/allowance 조회 제거(또는 신뢰/staked paymaster 운영정책 고정)
4. 위 1~3 항목에 대해 bundler 호환성 정책을 문서/SDK/백엔드에 동일하게 명시
