# EIP-4337 + EIP-7579 충돌 방지 설계 가이드

작성일: 2026-02-25 (v0.9 반영 갱신)
대상 독자: ERC-4337 기반 모듈형 Account(ERC-7579) 구현 개발자
기준 EntryPoint: **v0.9** (`0x433709009B8330FDa32311DF1C2AFA402eD8D009`) — v0.7/v0.8 ABI 호환

관련 문서:
- `docs/claude/spec/EIP-4337_스펙표준_정리.md`
- `docs/claude/spec/EIP-7579_스펙표준_정리.md`

## 1. 목적
EIP-4337(외부 실행/정산 프로토콜)과 EIP-7579(계정 내부 모듈 표준)를 함께 사용할 때 발생할 수 있는 충돌을 사전에 차단하기 위한 설계 규칙을 정의한다.

핵심 원칙:
- 4337은 외부 계약(EntryPoint/Bundler/Paymaster) 호환성이 우선
- 7579는 계정 내부 확장성(validator/executor/hook/fallback)의 구현 방식
- 내부 확장이 외부 프로토콜 요구사항을 침해하면 안 됨

## 2. 충돌 위험 지점 요약

| ID | 충돌 지점 | 위험 |
|---|---|---|
| C1 | `callData` 실행 경로 vs `execute(mode, executionCalldata)` 경로 | 시뮬레이션 통과 후 실제 실행 실패 |
| C2 | `executeUserOp` 의존성 | 7579 계정 호환성 저하 |
| C3 | validationData 인코딩 해석 불일치 | `AA24/AA22` 등 검증 실패 |
| C4 | Hook의 비결정성/고비용 로직 | bundler 시뮬레이션 불안정 |
| C5 | fallback sender 문맥 처리 오류 | 권한 우회/오탐 |
| C6 | nonce 계층(EntryPoint vs 모듈 내부) 혼선 | replay/nonce mismatch |
| C7 | 7579 확장 모듈 타입(5+) 필수화 | 인프라 상호운용성 저하 |
| C8 | timestamp vs block-number mode 혼용 | 유효성 판정 불일치 |
| C9 | delegatecall mode(0xff) 대상 신뢰 검증 부재 | 계정 storage 오염, 임의 코드 실행 |
| C10 | Hook `postCheck` 실패 → Paymaster `postOp(opReverted)` 상호작용 | hook 실패가 paymaster 비용/정산에 영향 |
| C11 | v0.9 `IgnoredInitCode` 동작과 7579 초기 모듈 설치 | 배포 완료 계정에 initCode 제공 시 무시되어 초기 모듈 미설치 |
| C12 | EIP-7702 초기화 경로와 7579 `installModule` 경로 불일치 | 7702 authorization 기반 초기화가 7579 모듈 onInstall을 트리거하지 않을 수 있음 |

## 3. 충돌 방지 규칙 (MUST/SHOULD)

### R1. 실행 엔트리 정합성 (C1, C11)
- MUST: `userOp.callData`가 실제 계정 실행 함수 ABI와 1:1 대응되도록 고정.
- MUST: 지원하는 실행 모드(`supportsExecutionMode`)와 실제 실행 가능 모드가 일치.
- SHOULD: 실행 경로를 `executeUserOp -> execute(mode, executionCalldata)`로 단일화해 분기 최소화.
- SHOULD: v0.9 `IgnoredInitCode` 동작에 대비하여, 7579 초기 모듈 설치를 `initCode` → factory 경로에만 의존하지 않도록 설계. 배포 완료 계정에 initCode가 제공되면 v0.9에서는 revert 대신 무시(`IgnoredInitCode` 이벤트 발행)되므로, 모듈 초기화 상태를 별도 검증하거나 `installModule` 경로를 fallback으로 제공.

검증 포인트:
- `supportsExecutionMode(mode) == true`인 모든 mode에 대해 실제 실행 테스트 통과
- v0.9: 이미 배포된 계정에 initCode 포함 UserOp 제출 시, 초기 모듈 상태가 기대와 일치하는지 검증

### R2. `executeUserOp` 호환성 처리 (C2)
- MUST: `executeUserOp`를 사용하는 경우 onlyEntryPoint 강제.
- SHOULD: `executeUserOp` 미구현 계정도 지원할 수 있도록 SDK/인코더에서 fallback ABI 전략 제공.
- SHOULD: 문서에 `executeUserOp`가 4337-실무 확장임을 명시.

### R3. validationData 포맷 고정 (C3)
- MUST: validator 모듈 반환값은 ERC-4337 validationData 포맷(`authorizer|validUntil|validAfter`) 준수.
- SHOULD: 서명 실패 시 `SIG_VALIDATION_FAILED(1)` 반환, revert 지양. 가스 추정 정확도를 위해 조기 반환 없이 정상 흐름 완료 SHOULD (EIP-4337 §5 기준).
- SHOULD: 모든 validator 모듈이 동일 helper 라이브러리로 validationData를 pack/unpack.
- SHOULD: v0.9 EIP-712 헬퍼(`getPackedUserOpTypeHash()`, `getDomainSeparatorV4()`)를 활용해 오프체인 hash 계산과 온체인 검증의 일관성을 보장.

### R4. Hook 결정성/비용 제한 (C4, C10)
- MUST: 실행 구간(hook `preCheck`/`postCheck` 포함) 전체에서 비결정적 외부 상태 의존 최소화. (참고: 7579 Hook은 execution 단계에서 동작하며, bundler 시뮬레이션은 validation+execution 전체를 trace하므로 hook 비결정성이 시뮬레이션-온체인 불일치를 유발할 수 있음)
- SHOULD: v0.9 `delegateAndRevert` 시뮬레이션 경로(state override 미지원 네트워크용)와 기존 view/trace call 시뮬레이션 경로에서 hook 동작이 동일한 결과를 산출하는지 검증. 시뮬레이션 방식 차이로 인한 hook 동작 불일치에 주의.
- MUST: hook 실패 시 오류가 명확히 식별 가능해야 함(원인 추적 가능).
- MUST: `postCheck` 실패가 전체 execution을 revert시키고, 이로 인해 Paymaster `postOp(opReverted)` 경로가 트리거됨을 인지하고 설계. hook 실패 비용이 Paymaster에 전가되므로 hook 안정성이 Paymaster 경제성에 직결됨.
- SHOULD: hook은 정책 검증 중심, 고비용 연산/외부 상호작용은 실행 로직 내부로 이동.
- SHOULD: bundler 시뮬레이션 기준 가스 상한을 초과하지 않도록 정책 설계.

### R5. Fallback sender 문맥 일치 (C5)
- MUST: fallback handler에서 직접 `msg.sender` 권한 판정 금지(ERC-2771 문맥 사용).
- MUST: EntryPoint 경유 호출과 direct 호출을 구분 가능한 방식으로 처리.
- SHOULD: fallback 경로별 auth 모델을 문서화(allowed caller matrix).

### R6. nonce 계층 분리 (C6)
- MUST: EntryPoint nonce(4337)와 모듈/세션 nonce(7579 확장)를 논리적으로 분리.
- MUST: replay 방지 키를 도메인 분리(체인/EntryPoint/Account/모듈 컨텍스트).
- SHOULD: nonce 소모 타이밍을 validation/실행 중 어디서 하는지 표준화.

### R7. 확장 모듈 타입 비필수화 (C7)
- MUST: 타입 1~4만으로도 계정이 동작하도록 설계.
- SHOULD: 타입 5+ (policy/signer 등)는 optional capability로 제공.
- SHOULD: bundler/SDK가 확장 타입을 몰라도 기본 흐름이 깨지지 않게 구성.

### R8. delegatecall mode 대상 신뢰 검증 (C9)
- MUST: `callType=0xff`(delegatecall) 실행 시 target이 신뢰할 수 있는 컨트랙트인지 검증 (EIP-7579 §5 Security Considerations 명시).
- MUST: delegatecall 대상을 화이트리스트 또는 레지스트리 기반으로 제한. 임의 주소에 대한 delegatecall 금지.
- SHOULD: delegatecall mode 지원 여부를 `supportsExecutionMode`에서 정확히 반영하고, 미지원 시 revert.

### R9. 시간 유효성 모드 일관성 (C8)
- MUST: 한 UserOp 내 validationData는 timestamp 모드 또는 block-number 모드 중 하나만 사용.
- MUST: Account/Paymaster/Validator의 시간 기준이 일치.
- SHOULD: 운영 환경별(rollup 포함) block-time 변동성 고려해 mode 선택 기준 명시.

### R10. EIP-7702 초기화 경로와 7579 모듈 초기화 정합성 (C12)
- MUST: EIP-7702 경로(`initCode` = `0x7702` prefix)로 계정 초기화 시, 7579 모듈 설치(`installModule`/`onInstall`)가 누락되지 않도록 초기화 함수에서 필수 모듈 설치를 보장.
- MUST: EIP-7702 초기화 함수 호출은 `entryPoint.senderCreator()`에서만 허용 (EIP-4337 §10 명시). 7579 `installModule` 권한 모델과 충돌하지 않도록 초기화 경로를 분리.
- SHOULD: EIP-7702 경로 미지원 네트워크에서도 동일한 모듈 초기화 결과를 달성할 수 있는 factory 기반 fallback 제공.
- SHOULD: SDK에서 EIP-7702 지원 여부를 감지해 초기화 경로를 자동 선택하는 로직 제공.

## 4. 구현 체크리스트

### 4.1 Account
- `validateUserOp` onlyEntryPoint
- `missingAccountFunds` 처리
- `supportsExecutionMode`와 `execute` 정합성
- `isValidSignature` -> validator forwarding 일관성

### 4.2 Module (7579)
- validator: validationData 포맷 준수
- executor: `executeFromExecutor` 권한 검증
- hook: pre/post 호출 순서 및 실패 전략
- fallback: ERC-2771 sender 문맥 사용

### 4.3 Bundler/Paymaster 연계
- 시뮬레이션 결과와 온체인 실행 결과 차이 없는지 확인
- paymaster validation은 가급적 무상태 유지
- `paymasterAndData` 파싱 실패 케이스 명시적 테스트
- v0.9: `delegateAndRevert` 시뮬레이션 경로와 view/trace call 경로에서 동일 결과 확인
- v0.9: `IgnoredInitCode` 이벤트 발생 시 모듈 초기화 상태 모니터링

## 5. 테스트 매트릭스 (최소)

1. 실행 모드 정합성
- `supportsExecutionMode=true`인 mode 전수 실행

2. validationData 일관성
- validator별 성공/실패/만료/미도래 케이스

3. hook 영향
- hook on/off, revert, 고가스 상황에서 시뮬레이션/실행 동일성

4. fallback 권한
- direct call vs EntryPoint 경유 call에서 sender 해석 검증

5. nonce/replay
- EntryPoint nonce 증가 + 모듈 nonce 증가 시나리오 교차 테스트

6. paymaster 결합
- paymaster on/off, context 존재/부재, postOp 실패 케이스

7. hook + paymaster 결합
- `postCheck` 실패 → `postOp(opReverted)` 경로에서 paymaster 정산 정확성 검증
- `preCheck` 고가스 + paymaster deposit 부족 시나리오
- hook revert 시 paymaster 비용 전가 범위 확인

8. delegatecall 보안
- `callType=0xff` 실행 시 target 화이트리스트 검증
- 비신뢰 target에 대한 delegatecall 차단 확인
- delegatecall 후 계정 storage 무결성 검증

9. v0.9 특화 시나리오
- 이미 배포된 계정에 initCode 포함 UserOp 제출 → `IgnoredInitCode` 이벤트 발행 + 모듈 초기화 상태 확인
- `delegateAndRevert` 시뮬레이션과 view/trace call 시뮬레이션에서 hook 동작 동일성 검증
- EIP-7702 초기화 경로(`0x7702` prefix)로 계정 생성 시 7579 필수 모듈 설치 완료 확인
- EIP-7702 초기화 후 `isModuleInstalled` 조회 결과 정합성

## 6. 운영 규칙 (권장)
- EntryPoint 주소/버전 핀닝 필수 (v0.9: `0x433709009B8330FDa32311DF1C2AFA402eD8D009`)
- 계정 구현 버전(accountId)과 SDK encoder 버전 동기화
- 모듈 설치 상태/권한 맵을 인덱싱해 관측 가능성 확보
- 시뮬레이션 실패 사유를 AA 코드 + 모듈 코드로 이중 로깅

## 7. 결론
4337+7579 결합의 핵심은 "외부 프로토콜 준수(4337) 위에 내부 모듈화(7579)를 얹는 구조"를 유지하는 것이다.  
즉, 7579 확장은 자유롭게 하되, 4337의 검증/실행/정산 계약을 침해하지 않도록 위 규칙을 기본 안전장치로 적용해야 한다.
