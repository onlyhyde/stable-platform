# EIP-7702 스펙 표준 정리

작성일: 2026-02-23
기준 문서: https://eips.ethereum.org/EIPS/eip-7702

## 1. 개요
EIP-7702는 EOA(Externally Owned Account)가 트랜잭션을 통해 자신의 계정 코드에 위임 표시를 설정할 수 있게 하는 표준이다.

핵심 아이디어:
- 새 트랜잭션 타입(`0x04`, Set Code Transaction) 도입
- `authorization_list`(서명 튜플 목록)를 통해 위임 대상 코드를 지정
- 권한 부여된 계정의 code에 `0xef0100 || address`(delegation indicator) 기록
- 실행 시 해당 address의 코드를 로드/실행

## 2. 표준 파라미터
| 파라미터 | 값 |
|---|---|
| `SET_CODE_TX_TYPE` | `0x04` |
| `MAGIC` | `0x05` |
| `PER_AUTH_BASE_COST` | `12500` |
| `PER_EMPTY_ACCOUNT_COST` | `25000` |

## 3. 트랜잭션 구조
Set Code Transaction 페이로드는 다음 RLP 구조를 따른다.

```text
rlp([
  chain_id,
  nonce,
  max_priority_fee_per_gas,
  max_fee_per_gas,
  gas_limit,
  destination,
  value,
  data,
  access_list,
  authorization_list,
  signature_y_parity,
  signature_r,
  signature_s
])

authorization_list = [[chain_id, address, nonce, y_parity, r, s], ...]
```

주의:
- 바깥 트랜잭션 필드는 EIP-4844와 동일 의미를 따른다.
- `destination = null`(컨트랙트 생성)은 유효하지 않다.
- 바깥 트랜잭션 서명은 `keccak256(SET_CODE_TX_TYPE || TransactionPayload)` 기준 secp256k1.

## 4. 실행 의미론

### 4.1 authorization tuple 처리
각 tuple(`[chain_id, address, nonce, y_parity, r, s]`)에 대해 권한 부여자(authority)를 복원하고, 해당 authority 계정 code를 delegation indicator로 설정한다.

```text
authority.code = 0xef0100 || address
```

스펙상 처리 순서(중요):
1. 트랜잭션 실행 시작 시 sender nonce를 먼저 1 증가
2. 이후 authorization_list를 순서대로 처리
3. 각 tuple에서 `authority = ecrecover(keccak(MAGIC || rlp([chain_id, address, nonce])), y_parity, r, s)` 복원
4. `authority.nonce == auth.nonce` 검증 후 성공하면 authority nonce를 1 증가
5. 유효한 tuple만 delegation indicator를 적용하고, 유효하지 않은 tuple은 스킵(전체 tx 즉시 무효 아님)

### 4.2 코드 실행 규칙
- delegation indicator가 설정된 계정은 실행 시 실제 코드 대신 가리키는 대상 address의 코드를 사용한다.
- delegation 체인이 이어져도 재귀 추적하지 않고 "첫 번째 코드"만 사용한다.
- precompile 주소를 delegation 대상으로 지정하면 code는 empty로 간주된다.

### 4.3 트랜잭션 발신 규칙(EIP-3607 완화)
- 일반적으로 code가 있는 계정은 tx origin이 될 수 없지만,
- code가 정확히 유효한 delegation indicator(`0xef0100 || address`)인 경우에는 발신 가능하다.
- 다른 임의 code를 가진 계정은 계속 발신 불가.

## 5. 가스 규칙

### 5.1 Intrinsic Gas
기본은 EIP-2930 규칙을 상속한다.

```text
21000
+ 16 * non-zero calldata bytes
+ 4 * zero calldata bytes
+ 1900 * access list storage key 수
+ 2400 * access list address 수
+ PER_EMPTY_ACCOUNT_COST * authorization_list 길이
```

### 5.2 추가 비용
- authorization tuple 처리 비용(`PER_AUTH_BASE_COST`)이 반영된다.
- delegation된 code 해석 시 cold account 접근이면 EIP-2929의 cold read 비용이 추가된다.

## 6. 처리 플로우
1. 사용자가 `type=0x04` 트랜잭션 생성
2. `authorization_list`에 위임 대상 및 권한 증명 서명 포함
3. 노드/클라이언트가 트랜잭션 및 authorization 검증
4. state transition에서 authority code를 delegation indicator로 기록
5. 트랜잭션 실행 중 authority 호출 시 위임 대상 code를 로드해 실행

## 7. EIP-4337과의 관계
- EIP-7702는 EOA의 코드 위임 메커니즘(프로토콜 레벨)
- EIP-4337은 UserOperation 기반 계정 추상화 실행 프레임워크(EntryPoint 중심)
- 실무에서는 7702로 EOA를 위임 가능한 형태로 전환하고, 4337 경로와 결합해 UX를 개선하는 패턴이 자주 사용된다.

## 8. 보안 고려사항
- 위임 대상 코드는 EOA 전체 권한과 사실상 동일 수준이므로 코드 신뢰성 검증 필수
- 초기화 프런트런 위험: 위임 후 초기 설정 트랜잭션 설계 주의
- 스토리지 충돌/관리: 위임 코드 업그레이드 및 상태 레이아웃 관리 필요
- relayer/sponsor 모델 악용 방지: bond, rate-limit, reputation 고려
- tx 전파 정책: 위임된 EOA는 pending tx 상호 무효화 가능성이 커 클라이언트 정책 강화 필요

## 9. 구현 체크리스트
- `type=0x04` 인코딩/서명 로직 구현
- `authorization_list` 생성/검증 유틸 구현
- delegation indicator 포맷 검증(`0xef0100 || address`)
- wallet UI에서 위임 코드 신뢰 고지 및 주소 검증 UX 제공
- 노드/인덱서/시뮬레이터에서 위임 코드 해석 경로 테스트

## 11. 실제 예시

아래 예시는 이해를 위한 샘플이며, 주소/nonce/가스/서명 값은 실행 환경에 맞게 교체해야 한다.

### 11.1 Set Code Transaction (사인 전 payload) 예시

```json
{
  "type": "0x04",
  "chainId": "0x1",
  "nonce": "0x15",
  "maxPriorityFeePerGas": "0x3b9aca00",
  "maxFeePerGas": "0x77359400",
  "gas": "0x0186a0",
  "to": "0x1111111111111111111111111111111111111111",
  "value": "0x0",
  "data": "0x",
  "accessList": [],
  "authorizationList": [
    {
      "chainId": "0x1",
      "address": "0x2222222222222222222222222222222222222222",
      "nonce": "0x16",
      "yParity": "0x0",
      "r": "0x8f4f0f6f4ce9dd3fb90d0eb2ac18a6c4a4db2f28d5a7da3d7f15ef2d9a13c9a1",
      "s": "0x1f3ce7ef567ec4f3183a1d3b7f3aa5bcf2d7a7f65e7e2fbe2f632fd0fdcb9f40"
    }
  ]
}
```

설명:
- `to`: 트랜잭션 실행 대상. `null`은 허용되지 않는다.
- `authorizationList[0].address`: authority 계정이 위임할 코드 주소.
- 실행 시 authority 계정 code는 `0xef0100 || 0x2222...2222` 형태로 세팅된다.

### 11.1-a `authorizationList` 상세 해설 (중요)

- `authorizationList`는 "누가 어떤 delegate 코드를 승인했는지"를 담는 권한 위임 서명 목록이다.
- 각 tuple은 `[chainId, address, nonce, yParity, r, s]` 형태다.
- `address`는 위임할 컨트랙트 주소(delegate target)다.
- `yParity/r/s`는 해당 tuple 내용에 대해 authority EOA가 서명한 값이다.

nonce 해설(바깥 tx nonce vs authorization nonce):
- 바깥 tx의 `nonce`(예시 129라인)는 **트랜잭션 발신자(sender) nonce**다.
- tuple의 `nonce`(예시 141라인)는 **authority 계정 nonce**다.
- 둘은 같은 필드가 아니며, authority가 sender와 다르면 당연히 값이 다를 수 있다.

같은 계정인 경우:
- sender == authority이면 두 nonce의 관계가 중요해진다.
- EIP-7702 규칙상 "sender nonce가 먼저 증가"한 뒤 authorization tuple을 검증한다.
- 따라서 sender == authority라면, 일반적으로 `auth.nonce = tx.nonce + 1` 관계를 만족해야 한다.
- 예: tx nonce가 `0x15`이면 authorization nonce는 `0x16`이 맞다.
- 문서의 `0x15` vs `0x16`은 샘플 값이므로 실제 구성 시에는 체인 상태 nonce 규칙에 맞춰 교체해야 한다.

### 11.2 authorization tuple (개념) 예시

```text
[
  chain_id = 1,
  address  = 0x2222222222222222222222222222222222222222,
  nonce    = 22,
  y_parity = 0,
  r        = 0x8f4f...c9a1,
  s        = 0x1f3c...9f40
]
```

핵심 체크:
- tuple 서명 복구 결과 authority가 기대한 EOA여야 한다.
- tuple nonce는 authority 단위로 관리되어 재사용되면 실패한다.

### 11.3 ethers(v6) 스타일 의사 코드 예시

```ts
import { Wallet, JsonRpcProvider } from "ethers";

const provider = new JsonRpcProvider(process.env.RPC_URL!);
const signer = new Wallet(process.env.PRIVATE_KEY!, provider);

const tx = {
  type: 4,
  chainId: 1,
  nonce: 21,
  maxPriorityFeePerGas: 1_000_000_000n,
  maxFeePerGas: 2_000_000_000n,
  gasLimit: 100000n,
  to: "0x1111111111111111111111111111111111111111",
  value: 0n,
  data: "0x",
  accessList: [],
  authorizationList: [
    {
      chainId: 1n,
      address: "0x2222222222222222222222222222222222222222",
      // sender == authority인 예시: auth nonce = tx nonce + 1
      nonce: 22n,
      yParity: 0,
      r: "0x8f4f0f6f4ce9dd3fb90d0eb2ac18a6c4a4db2f28d5a7da3d7f15ef2d9a13c9a1",
      s: "0x1f3ce7ef567ec4f3183a1d3b7f3aa5bcf2d7a7f65e7e2fbe2f632fd0fdcb9f40"
    }
  ]
};

// 라이브러리/클라이언트가 EIP-7702 type-4를 지원해야 한다.
const sent = await signer.sendTransaction(tx as any);
const receipt = await sent.wait();
console.log(receipt?.hash);
```

### 11.4 실행 후 기대 상태 예시

- authority EOA code:

```text
0xef0100 + 0x2222222222222222222222222222222222222222
```

- 이후 authority를 호출하면, 실제 실행 코드는 `0x2222...2222` 계정 code를 따른다.
- 단, delegation 대상이 precompile이면 code는 empty로 처리된다.
