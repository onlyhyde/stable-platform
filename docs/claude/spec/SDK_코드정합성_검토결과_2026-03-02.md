## 2. 핵심 결론 (요약)

**핵심 암호학적 정합성(해시/서명/인코딩)은 확보**되어 있으며, 검출된 P0/P1/P2 이슈 총 7건 **전수 RESOLVED**.

심각도별 결과:

- P2: 2건 → **✅ 2건 RESOLVED** (Block Number Mode, ModuleStatus 열거형 통일)
- PASS: 아키텍처, 코드 중복, 확장성, 핵심 해시/인코딩

### 3.2 설계 패턴 적용 — PASS

| 패턴 | 적용 위치 | 효과 |
|------|----------|------|
| **Strategy Pattern** | `TransactionRouter` (TS/Go), `GasEstimator` (TS/Go) | 새 트랜잭션 모드 추가 시 기존 코드 수정 없음 (OCP) |
| **Builder Pattern** | `UserOperationBuilder` (wallet-sdk), `PermissionRequestBuilder` (wallet-sdk) | Fluent API로 타입 안전 UserOp 조립 |
| **Dependency Inversion** | `CryptoProvider`, `RpcClient`, `AbiEncoder` (TS/Go) | viem/geth 외 다른 라이브러리로 교체 가능 |
| **Plugin Architecture** | `sdk-ts/plugins/*` | paymaster/validator/executor 추가가 독립적 |
| **Module Registry** | `modules/registry` (TS/Go) | 새 모듈 타입 등록이 선언적 |
| **Singleton** | `getProviderRegistry()` (wallet-sdk) | EIP-6963 provider 중복 탐색 방지 |

### 3.3 코드 중복 평가 — PASS (중복 없음)

| 검토 항목 | 결과 |
|----------|------|
| wallet-sdk ↔ sdk-ts/core | ✓ 위임 패턴. `getUserOperationHash`, `packUserOperation`, 가스 상수 모두 `@stablenet/core` 재수출 |
| sdk-ts/types ↔ 각 패키지 | ✓ 중앙 타입 정의를 재수출 |
| sdk-ts/plugins 간 | ✓ 각 플러그인이 독립적 단일 책임 |
| sdk-ts ↔ sdk-go | ✓ 동일 알고리즘의 언어별 구현 (불가피한 의도적 "중복"). 바이트 레이아웃/해시 일치 |

sdk-ts와 sdk-go 간의 동일 알고리즘 구현은 크로스 언어 SDK에서 불가피하며, 정합성만 보장되면 문제없다.

### 3.4 확장성 평가 — PASS

**잘 설계된 확장 포인트:**

- `TransactionRouter`의 `StrategyRegistry`: 새 전략(예: ERC-3074) 추가 시 register만 호출
- `sdk-ts/plugins`: 독립 패키지로 paymaster/validator/executor 추가 가능
- `ModuleRegistry`: 새 모듈 타입(5: Policy, 6: Signer) 이미 포함
- `NetworkRegistry`: 커스텀 네트워크 동적 추가 + 스토리지 추상화

**개선 가능한 점:**

- Paymaster type별 분기가 `codec.ts`/`codec.go`에서 switch문으로 처리됨 → type별 Strategy로 분리하면 새 paymaster type 추가가 더 깔끔
- `sdk-go`에 plugin 아키텍처가 `sdk-ts`만큼 체계화되어 있지 않음 (interface 기반 확장 포인트 명시 권장)

```typescript
const EIP712_DOMAIN_VERSION_HASH = keccak256(stringToHex('1'))
const PACKED_USEROP_TYPEHASH = keccak256(stringToHex(

```

**Go** (`sdk-go/core/userop/hash.go`):
```go
eip712DomainNameHash    = ethcrypto.Keccak256Hash([]byte("ERC4337"))
eip712DomainVersionHash = ethcrypto.Keccak256Hash([]byte("1"))
```

### 5.2 [P1] paymasterAndData v0.9 Signature Magic 미구현

**스펙 요구사항:**

> v0.9 옵션 suffix:
> - `[optional] paymasterSignature(variable)`
> - `[optional] uint16(paymasterSignature.length)`
> - `[optional] PAYMASTER_SIG_MAGIC (0x22e325a297439656)`
>
> 옵션 suffix는 **병렬 서명**을 위한 설계이다. Paymaster 서명이 `userOpHash`에 포함되지 않으므로, Account 서명과 Paymaster 서명을 독립적으로(병렬로) 생성할 수 있다.
>
> — `EIP-4337_Paymaster_개발자_구현가이드.md:38-43`
> — `EIP-4337_스펙표준_정리.md:999-1001`

**✅ RESOLVED (2026-03-02)**

**수정 내용:**

1. `PAYMASTER_SIG_MAGIC = 0x22e325a297439656` 상수 양쪽 SDK에 추가
2. v0.9 인코딩 함수 추가:
   - sdk-ts: `encodePaymasterDataWithSignatureV09()` — `envelope + signature + uint16(sigLen) + magic`
   - sdk-go: `ConcatWithSignatureV09()` — 동일 포맷
3. v0.9 감지 함수 추가:
   - sdk-ts: `hasSignatureMagic()` — 마지막 8바이트로 magic 검사
   - sdk-go: `HasSignatureMagic()` — 동일
4. `splitEnvelopeAndSignature()` / `SplitEnvelopeAndSignature()` — magic 존재 여부로 v0.9 자동 감지, 없으면 레거시 모드 유지

수정 파일:
- `sdk-ts/core/src/paymaster/paymasterDataCodec.ts`
- `sdk-go/core/paymaster/codec.go`
- `sdk-ts/core/src/paymaster/index.ts` (export 추가)

### 5.3 [P1] EIP-7702 initCode `0x7702` 감지 미구현

**스펙 요구사항:**

> `initCode`가 `0x7702`(20B right-padded)로 시작하면 EIP-7702 경로로 진입
> EntryPoint가 factory를 호출하지 않고, EIP-7702 authorization을 검증
>
> — `EIP-4337_스펙표준_정리.md:640-646`

> `initCode`가 20B 초과 시, 나머지 부분으로 계정 초기화 함수 호출
> 초기화 함수 호출은 `entryPoint.senderCreator()`에서만 MUST 허용
>
> — `EIP-4337_스펙표준_정리.md:644-646`

**✅ RESOLVED (2026-03-02)**

**수정 내용:**

1. `EIP7702_INIT_CODE_ADDRESS = 0x0000...7702` 상수 양쪽 SDK에 추가

### 5.5 [P2] Block Number Mode (bit 47 flag) 미구현

**스펙 요구사항:**

> `validUntil`과 `validAfter`의 최상위 비트(bit 47)를 1로 설정하면 timestamp 대신 **block number** 기준으로 동작한다.
>
> ```
> VALIDITY_BLOCK_RANGE_FLAG = 0x800_000_000_000  // bit 47 플래그
> VALIDITY_BLOCK_RANGE_MASK = 0x7ff_fff_fff_fff  // 하위 47비트 마스크
> ```
>
> — `EIP-4337_스펙표준_정리.md:219-228`

**✅ RESOLVED (2026-03-02)**

**수정 내용:**

1. 상수 추가 (양쪽 SDK):
   - `VALIDITY_BLOCK_RANGE_FLAG = 0x800_000_000_000` (bit 47 플래그)
   - `VALIDITY_BLOCK_RANGE_MASK = 0x7ff_fff_fff_fff` (하위 47비트 마스크)
   - `SIG_VALIDATION_FAILED = 1` (서명 검증 실패)
2. `parseValidationData()` — uint256 validationData를 authorizer/validUntil/validAfter + block mode 플래그로 파싱
3. `packValidationData()` — 컴포넌트를 uint256으로 패킹 (useBlockNumbers 옵션 지원)
4. `isSignatureValidationFailed()` — 서명 실패 여부 검사
5. `usesBlockNumberMode()` — block number 모드 사용 여부 검사

수정 파일:
- `sdk-ts/core/src/utils/validationData.ts` (신규)
- `sdk-ts/core/src/index.ts` (export 추가)
- `sdk-go/core/userop/validation_data.go` (신규)

## 8. 우선순위 실행 계획

### 1주차 (P0 — 즉시)

1. **가스 상수 통일**: EntryPoint 컨트랙트 실행 비용 기준으로 정확한 값 결정 → sdk-ts/sdk-go 동시 반영

3. **크로스 언어 가스 상수 일치 CI 테스트 추가**

### 2주차 (P1 — 단기)

4. **paymasterAndData signature magic 지원**: `PAYMASTER_SIG_MAGIC` 인코딩/디코딩 추가 (하위 호환)
5. **initCode 0x7702 감지 유틸 추가**: UserOp builder에 EIP-7702 경로 분기 지원
6. **ModuleStatus 직렬화 통일**: TS/Go 간 API 호환성 보장
7. **Nonce 상세 파싱 함수 추가**: `parseNonceDetailed`, `encodeNonce`

### 3주차 이후 (P2 — 장기)

8. **Block Number Mode 지원**: `VALIDITY_BLOCK_RANGE_FLAG` 상수 + bit 47 처리
9. **Paymaster codec Strategy 분리**: switch문 → type별 Strategy 패턴
10. **sdk-go plugin 아키텍처 강화**: interface 기반 확장 포인트 명시
