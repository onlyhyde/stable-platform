## 1. 조사 배경

### 1.1 목적

StableNet의 **Authorized Account** (기관 트랜잭션 우선 처리)가 기존 체인에서 구현 가능한지 검증하고, 진짜 차별점을 도출합니다.

### 1.2 핵심 질문

> "기존 Ethereum에서도 Priority Fee나 Private Mempool로 우선 처리가 가능한데,
> StableNet의 Authorized Account가 진짜 차별점인가?"

### 1.3 조사 대상

| 메커니즘 | 설명 |
|----------|------|
| **MEV** | 블록 생산자의 트랜잭션 순서 조작 이익 |
| **Priority Fee** | EIP-1559 기반 우선순위 수수료 |
| **Private Mempool** | MEV 회피를 위한 비공개 트랜잭션 풀 |

## 2. MEV (Maximal Extractable Value)

### 2.1 개요

| 항목 | 정보 |
|------|------|
| **정의** | 블록 생산자가 트랜잭션 순서 조작으로 추출할 수 있는 최대 이익 |
| **규모** | 2024년 기준 일평균 ~$300,000 (Ethereum) |
| **주요 공격** | Sandwich Attack (51.56%), Front-running, Back-running |
| **참조** | https://ethereum.org/developers/docs/mev/ |

### 2.2 MEV 공격 유형

#### Sandwich Attack

```
┌─────────────────────────────────────────────────────────────────┐
│                     Sandwich Attack Flow                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 사용자: DEX 스왑 트랜잭션 제출 (Public Mempool)               │
│     └─ ETH → USDC, 예상 가격: $2,000                            │
│                                                                  │
│  2. MEV Bot 감지 및 공격:                                        │
│     ┌─────────────────────────────────────────────────────┐     │
│     │  ① Front-run: Bot이 먼저 ETH → USDC                 │     │
│     │     → 가격 상승 ($2,000 → $2,050)                   │     │
│     │                                                      │     │
│     │  ② 사용자 TX: 높아진 가격에 스왑 실행                │     │
│     │     → $2,050에 거래 (슬리피지 발생)                  │     │
│     │                                                      │     │
│     │  ③ Back-run: Bot이 USDC → ETH                       │     │
│     │     → 차익 실현                                      │     │
│     └─────────────────────────────────────────────────────┘     │
│                                                                  │
│  결과: 사용자 손실 = Bot 이익                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### Front-running

```
사용자 TX (높은 이익 기회) → Bot 감지 → Bot이 먼저 실행 → 사용자 기회 상실
```

#### Back-running (Arbitrage)

```
사용자 TX (가격 변동 유발) → Bot 감지 → Bot이 즉시 차익거래 실행
```

### 2.3 MEV 통계 (2024년 기준)

| 지표 | 수치 |
|------|------|
| **일평균 MEV** | ~$300,000 |
| **Sandwich Attack 비중** | 51.56% ($289.76M) |
| **총 MEV 거래량** | $561.92M |

### 2.4 MEV의 양면성

| 긍정적 측면 | 부정적 측면 |
|-------------|-------------|
| DEX 차익거래로 가격 효율성 향상 | 일반 사용자 슬리피지 손실 |
| 청산 봇으로 DeFi 프로토콜 안정성 | 네트워크 혼잡 유발 |
| 시장 효율성 증가 | 중앙화 우려 (MEV 전문 업체) |

## 3. EIP-1559 Priority Fee

### 3.1 개요

| 항목 | 정보 |
|------|------|
| **도입일** | 2021년 8월 5일 (London 하드포크) |
| **목적** | 가스비 예측 가능성 향상, 수수료 변동성 감소 |
| **공식 문서** | https://eips.ethereum.org/EIPS/eip-1559 |
| **GitHub** | https://github.com/ethereum/EIPs/blob/master/EIPS/eip-1559.md |

### 3.2 수수료 구조

| 구성 요소 | 설명 | 처리 |
|-----------|------|------|
| **Base Fee** | 네트워크 혼잡도 기반 자동 조정 | 🔥 소각 (Burn) |
| **Priority Fee (Tip)** | 사용자가 지정하는 검증자 인센티브 | 💰 검증자 지급 |
| **Max Fee** | 사용자가 지불 의향 있는 최대 금액 | 상한선 |

### 3.3 Base Fee 조정 메커니즘

```
블록 사용률에 따른 Base Fee 변화:

┌─────────────────────────────────────────────────────────────────┐
│  블록 사용률    │  Base Fee 변화                                │
├─────────────────────────────────────────────────────────────────┤
│  100% (풀)      │  +12.5% 증가                                  │
│  50% (목표)     │  변화 없음                                    │
│  0% (비어있음)  │  -12.5% 감소                                  │
└─────────────────────────────────────────────────────────────────┘

→ 블록 가스 한도: 30M (목표: 15M)
→ 가변 블록 크기로 일시적 혼잡 흡수
```

### 3.4 Priority Fee로 우선 처리

```
┌─────────────────────────────────────────────────────────────────┐
│                     Priority Fee 메커니즘                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Mempool 대기열:                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  TX A: Priority Fee = 1 Gwei   → 낮은 우선순위           │    │
│  │  TX B: Priority Fee = 10 Gwei  → 높은 우선순위 ⭐        │    │
│  │  TX C: Priority Fee = 5 Gwei   → 중간 우선순위           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  블록 포함 순서: TX B → TX C → TX A                              │
│                                                                  │
│  실제 지불액 계산:                                               │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Total Fee = Base Fee + Priority Fee                     │    │
│  │  (단, Max Fee를 초과하지 않음)                            │    │
│  │                                                          │    │
│  │  예: Base Fee = 20 Gwei, Priority Fee = 10 Gwei          │    │
│  │      → Total = 30 Gwei                                   │    │
│  │      → Base Fee (20) 소각, Priority Fee (10) 검증자에게  │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.5 한계점

| 한계 | 설명 |
|------|------|
| **수수료 경쟁** | 높은 Priority Fee = 높은 비용 부담 |
| **불확실성** | 경쟁 상황에서 우선순위 보장 어려움 |
| **MEV 미해결** | Priority Fee로 MEV 문제 해결 불가 |

## 4. Private Mempool

### 4.1 개요

| 항목 | 정보 |
|------|------|
| **현황** | Ethereum DeFi 트랜잭션의 ~80%가 Private RPC 사용 |
| **목적** | MEV 공격 (Front-running, Sandwich) 방지 |
| **작동 원리** | 트랜잭션을 Public Mempool 대신 Private 채널로 전송 |

### 4.2 작동 원리

```
┌─────────────────────────────────────────────────────────────────┐
│                     Private Mempool Flow                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  기존 방식 (Public Mempool):                                     │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  User TX → Public Mempool → MEV Bot 감지 → 공격         │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  Private Mempool 방식:                                           │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  User TX → Private RPC → Searcher 경매 → Builder        │    │
│  │         (숨김)        (Back-run만)    → Block           │    │
│  │                                                          │    │
│  │  • Front-run/Sandwich 불가 (TX가 숨겨짐)                 │    │
│  │  • Back-run으로 발생한 MEV의 90%를 사용자에게 환급       │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 주요 Private RPC 서비스 비교

| 서비스 | 성공률 | 응답시간 | MEV 보호 | 특징 |
|--------|--------|----------|----------|------|
| **Flashbots Protect** | 98.5% | 245ms | Excellent | 90% MEV 리펀드 |
| **MEV Blocker** | 96.2% | 180ms | Good | 60%+ 블록 커버리지 |
| **Merkle** | 94.8% | 220ms | Very Good | - |
| **Blink** | 92.1% | 165ms | Fair | 최고 응답 속도 |

### 4.4 Flashbots Protect

| 항목 | 정보 |
|------|------|
| **보호된 계정** | 2.1M+ |
| **보호된 DEX 거래량** | $43B+ |
| **MEV 리펀드** | 313 ETH+ |
| **특징** | 실패 TX 무료, 90% MEV 환급 |


### 5.2 Proposer-Builder Separation (PBS)

2024년 Ethereum에 도입된 블록 생성 분리 아키텍처:

```
┌─────────────────────────────────────────────────────────────────┐
│                     PBS Architecture                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────┐                                                   │
│  │ Searcher │ ─┐                                                │
│  │(MEV 추출) │  │                                                │
│  └──────────┘  │      ┌──────────┐      ┌──────────┐           │
│  ┌──────────┐  ├────▶ │ Builder  │ ───▶ │  Relay   │           │
│  │ Searcher │  │      │(블록 구성)│      │  (중개)   │           │
│  └──────────┘  │      └──────────┘      └──────────┘           │
│  ┌──────────┐  │                              │                  │
│  │ Searcher │ ─┘                              ▼                  │
│  └──────────┘                          ┌──────────┐             │
│                                        │Proposer  │             │
│                                        │(검증자)   │             │
│                                        │최고가 블록│             │
│                                        │  선택    │             │
│                                        └──────────┘             │
│                                              │                   │
│                                              ▼                   │
│                                        ┌──────────┐             │
│                                        │Ethereum  │             │
│                                        │ Network  │             │
│                                        └──────────┘             │
│                                                                  │
│  역할 분리:                                                      │
│  • Searcher: MEV 기회 탐색 및 번들 생성                          │
│  • Builder: 트랜잭션 순서 최적화 및 블록 구성                    │
│  • Relay: Builder와 Proposer 중개                               │
│  • Proposer: 최고 수익 블록 선택 및 제안                         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 6. 트랜잭션 순서 공정성

### 6.1 접근법 비교

| 접근법 | 설명 | 장점 | 단점 |
|--------|------|------|------|
| **Priority Fee** | 수수료 기반 우선순위 | 단순, 시장 기반 | 자금력 = 우선순위 |
| **FCFS** | 선착순 처리 | 직관적 공정성 | 네트워크 지연 문제 |
| **Batch Ordering** | 일정 시간 배치 처리 | Front-run 완화 | 처리 지연 |
| **TimeBoost** | 시간 + 경매 하이브리드 | 균형잡힌 접근 | 복잡성 |

### 6.2 FCFS (First Come First Served)

```
┌─────────────────────────────────────────────────────────────────┐
│                     FCFS의 한계                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  이상적 FCFS:                                                    │
│  TX1 (t=0ms) → TX2 (t=1ms) → TX3 (t=2ms)                        │
│  처리: TX1 → TX2 → TX3                                          │
│                                                                  │
│  현실 (분산 네트워크):                                           │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Node A: TX1 먼저 수신                                   │    │
│  │  Node B: TX2 먼저 수신 (네트워크 지연)                   │    │
│  │  Node C: TX3 먼저 수신 (지리적 위치)                     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  문제:                                                           │
│  • 분산 노드 간 "누가 먼저인가" 합의 어려움                      │
│  • 지연 시간 불확실성으로 완벽한 순서 보장 불가                  │
│  • 지연 최적화 경쟁 유발 (HFT와 유사)                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 7. StableNet Authorized Account 비교

### 7.1 기존 방식 vs StableNet

| 항목 | 기존 Ethereum | StableNet Authorized Account |
|------|---------------|------------------------------|
| **우선 처리 방법** | Priority Fee 경쟁 | 프로토콜 레벨 보장 |
| **비용** | 높은 수수료 필요 | 수수료 경쟁 없음 |
| **접근성** | 누구나 (자금력 기반) | 인가된 기관만 |
| **예측 가능성** | 불확실 (경쟁 기반) | 보장됨 |
| **공정성 기준** | 자금력 = 우선순위 | 기관 역할 기반 |
| **MEV 노출** | Private RPC 필요 | 프로토콜 레벨 보호 |

### 7.2 구조적 차이 분석

```
┌─────────────────────────────────────────────────────────────────┐
│                     우선 처리 방식 비교                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Ethereum (Priority Fee + Private Mempool):                      │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  기관 TX → Private RPC → 높은 Priority Fee 설정          │    │
│  │         → Builder 선택 → 블록 포함                       │    │
│  │                                                          │    │
│  │  문제:                                                   │    │
│  │  • 수수료 경쟁 비용 발생                                 │    │
│  │  • 100% 우선 처리 보장 어려움                            │    │
│  │  • Private RPC 의존성                                    │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  StableNet (Authorized Account):                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  기관 TX → Authorized Account 식별 → 프로토콜 레벨 우선  │    │
│  │         → 검증자 합의 → 블록 포함 보장                   │    │
│  │                                                          │    │
│  │  장점:                                                   │    │
│  │  • 수수료 경쟁 불필요                                    │    │
│  │  • 확정적 우선 처리                                      │    │
│  │  • 프로토콜 네이티브 보호                                │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.3 차별점 평가

| 평가 항목 | 차별점 수준 | 설명 |
|-----------|-------------|------|
| **기술적 구현** | ⚠️ 제한적 | 유사 결과를 기존 체인에서도 달성 가능 |
| **비용 구조** | 🟡 중간 | 수수료 경쟁 없음은 장점 |
| **확정성** | 🟡 중간 | 프로토콜 레벨 보장은 장점 |
| **접근성** | 🟢 차별점 | 인가 기관 전용 (규제 친화적) |

### 7.4 결론

```
기존 체인에서도 "우선 처리"는 가능하지만:

1. 비용: 지속적인 수수료 경쟁 필요
2. 확정성: 100% 보장 어려움
3. 복잡성: Private RPC + Priority Fee 조합 필요

StableNet Authorized Account의 가치:
→ "프로토콜 레벨에서 인가된 기관에 확정적 우선권 부여"
→ 규제 친화적 설계 (누가 우선권을 가지는지 투명)
→ 기관 입장에서 운영 단순화

그러나:
→ 이것만으로 "따라올 수 없는 차별점"이라고 하기 어려움
→ 다른 체인도 유사한 메커니즘 도입 가능
```
