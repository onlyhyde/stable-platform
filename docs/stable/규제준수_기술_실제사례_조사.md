## 1. TUSD (TrueUSD) - Proof of Reserve 선도 사례

### 1.1 개요

TrueUSD(TUSD)는 **최초로 Chainlink Proof of Reserve를 통한 실시간 준비금 검증 및 Secure Mint를 구현한 USD 스테이블코인**이다.

- **출시**: 2023년 2월 15일 Chainlink PoR 통합
- **결과**: 통합 후 1개월 만에 시가총액 121% 증가 (20억 달러 돌파)
- **배포 체인**: Ethereum, Arbitrum, Avalanche, BNB Chain, Polygon, Optimism, TRON 등 11개 체인

### 1.3 Secure Mint 메커니즘

**핵심 원리**: 준비금이 부족하면 발행(mint) 자체가 불가능

```solidity
// TUSD Secure Mint 핵심 로직
function mint(address to, uint256 amount) external onlyMinter {
    // Chainlink PoR 데이터 조회
    int256 reserves = getLatestReserves();
    int256 currentSupply = getLatestSupply();
    int256 newSupply = currentSupply + int256(amount);

    // 핵심: 새 발행량이 준비금을 초과하면 거부
    require(reserves >= newSupply, "Mint would exceed reserves");

    _mint(to, amount);
}
```

**보호 효과**:
- **무한 발행 공격(Infinite Mint Attack) 방지**: 준비금 없이 토큰 발행 불가
- **실시간 검증**: 24시간마다 또는 5% 변동 시 자동 업데이트
- **투명성**: 누구나 온체인에서 준비금 상태 확인 가능

## 3. Fireblocks MPC 월렛

### 3.1 개요

Fireblocks는 **MPC(Multi-Party Computation) 기반의 기관용 디지털 자산 수탁 플랫폼**으로, 단일 장애점(Single Point of Failure) 없이 키를 관리한다.

### 3.2 MPC-CMP 프로토콜

Fireblocks가 개발한 **MPC-CMP(Multi-Party Computation - Composable Modular Protocol)**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Fireblocks MPC 아키텍처                      │
└─────────────────────────────────────────────────────────────────┘

                    완전한 Private Key는 어디에도 존재하지 않음
                    ↓
┌───────────────────────────────────────────────────────────────┐
│                    Key Share 분산                              │
│                                                               │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐       │
│  │  Share 1    │    │  Share 2    │    │  Share 3    │       │
│  │             │    │             │    │             │       │
│  │ Fireblocks  │    │ Fireblocks  │    │  고객       │       │
│  │ Cosigner 1  │    │ Cosigner 2  │    │ (Mobile/    │       │
│  │             │    │             │    │  Server)    │       │
│  │ [Intel SGX] │    │ [Intel SGX] │    │             │       │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘       │
│         │                  │                  │               │
│         └──────────────────┼──────────────────┘               │
│                            │                                  │
│                            ▼                                  │
│              ┌─────────────────────────┐                      │
│              │  Threshold Signature    │                      │
│              │  (2-of-3 필요)          │                      │
│              │                         │                      │
│              │  Share 1 + Share 2 ✓    │                      │
│              │  Share 1 + Share 3 ✓    │                      │
│              │  Share 2 + Share 3 ✓    │                      │
│              └─────────────────────────┘                      │
│                            │                                  │
│                            ▼                                  │
│              ┌─────────────────────────┐                      │
│              │  Valid Transaction      │                      │
│              │  Signature              │                      │
│              └─────────────────────────┘                      │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 3.3 지갑 유형

| 유형 | Share 3 위치 | 승인 방식 | 용도 |
|------|-------------|----------|------|
| **Hot Wallet** | API Co-signer 서버 | 자동화 가능 | 고빈도 거래 |
| **Warm Wallet** | 인터넷 연결 모바일 | Fireblocks 앱 승인 | 일반 거래 |
| **Cold Wallet** | 에어갭(오프라인) 모바일 | QR 코드 스캔 | 대량 자산 보관 |

### 3.4 핵심 기술

#### Intel SGX (Software Guard Extensions)
```
┌─────────────────────────────────────────────────────────────────┐
│                    Intel SGX 보호                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │                    Enclave (보호 영역)                     │ │
│  │                                                           │ │
│  │   - MPC Key Share 저장                                    │ │
│  │   - 암호화 알고리즘 실행                                  │ │
│  │   - 메모리 공간 암호화                                    │ │
│  │                                                           │ │
│  │   OS나 하이퍼바이저도 접근 불가                           │ │
│  │   내부자(Rogue Admin)도 접근 불가                         │ │
│  │   멀웨어가 서버 장악해도 키 추출 불가                     │ │
│  │                                                           │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Key Refresh (키 갱신)
- 직원 퇴사 시 Share 무효화 가능
- 주소 변경 없이 Share 갱신
- 정기적 자동 갱신으로 보안 강화

### 3.5 성능

- **서명 속도**: 수 밀리초 ~ 수백 밀리초
- **프로토콜**: 단일 라운드 서명 (기존 MPC 대비 8배 빠름)
- **지원 알고리즘**: ECDSA (secp256k1), EdDSA (ed25519)

## 4. HSM (Hardware Security Module)

### 4.1 개요

HSM은 **암호화 키의 생성, 저장, 관리를 물리적으로 격리된 하드웨어에서 수행**하는 장치이다.

### 4.2 FIPS 140-2 보안 레벨

| 레벨 | 요구사항 | 용도 |
|------|---------|------|
| **Level 1** | 기본 보안, 소프트웨어 암호화 | 일반 용도 |
| **Level 2** | 물리적 훼손 증거(Tamper Evidence) | 중간 보안 |
| **Level 3** | 훼손 감지 및 대응, ID 기반 인증 | **암호화폐 수탁 권장** |
| **Level 4** | 환경(전압/온도) 공격 방어 | 최고 보안 |

### 4.3 주요 제품

| 제품 | 제조사 | 특징 |
|------|-------|------|
| **nShield** | Entrust | Security World 아키텍처, 클러스터 지원 |
| **YubiHSM 2 FIPS** | Yubico | 소형 USB-A, M-of-N 키 백업 |
| **CloudHSM** | AWS | 클라우드 기반, FIPS 140-2 Level 3 |
| **Dedicated HSM** | Azure | 단일 테넌트, FIPS 140-3 Level 3 |
| **CryptoServer** | Utimaco | PKCS#11, KMIP, REST API 지원 |

### 4.4 암호화폐 키 관리 사용 사례

```

## 5. Multi-sig (다중 서명) - Gnosis Safe

### 5.1 개요

Multi-sig 월렛은 **여러 서명자의 승인이 필요한 스마트 컨트랙트 월렛**이다. Gnosis Safe(현 Safe)가 업계 표준이다.

### 5.2 2-of-3 구성

```
┌─────────────────────────────────────────────────────────────────┐
│                    2-of-3 Multi-sig 구조                        │
└─────────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────┐
│                                                               │
│      Owner 1           Owner 2           Owner 3              │
│   (Hardware Wallet) (Hardware Wallet) (Hardware Wallet)       │
│         │                  │                │                 │
│         ▼                  ▼                ▼                 │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │                   Safe Smart Contract                    │ │
│  │                                                         │ │
│  │   owners: [0x111..., 0x222..., 0x333...]                │ │
│  │   threshold: 2                                          │ │
│  │                                                         │ │
│  │   트랜잭션 실행 조건:                                   │ │
│  │   - Owner 1 + Owner 2 서명 ✓                            │ │
│  │   - Owner 1 + Owner 3 서명 ✓                            │ │
│  │   - Owner 2 + Owner 3 서명 ✓                            │ │
│  │   - 단일 서명만으로는 실행 불가 ✗                       │ │
│  │                                                         │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                               │
│  장점:                                                        │
│  - 한 명이 키를 분실해도 나머지 두 명이 복구 가능            │
│  - 내부자 단독 범죄 방지                                     │
│  - 트랜잭션 배칭으로 가스비 절약                             │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 5.3 트랜잭션 흐름

```
1. Owner 1이 트랜잭션 제안 (propose)
   └─▶ Safe Contract에 트랜잭션 데이터 저장

2. Owner 2가 서명 (confirm)
   └─▶ threshold(2) 도달

3. 자동 실행 또는 수동 실행
   └─▶ 블록체인에 트랜잭션 제출
```

### 5.4 MPC vs Multi-sig 비교

| 항목 | MPC | Multi-sig |
|------|-----|-----------|
| **키 구조** | 단일 키를 샤드로 분산 | 여러 개의 독립된 키 |
| **온체인 발자국** | 일반 트랜잭션과 동일 | Multi-sig 컨트랙트 호출 |
| **가스비** | 일반 트랜잭션과 동일 | 컨트랙트 실행 비용 추가 |
| **체인 호환성** | 모든 체인 지원 | EVM 체인만 지원 |
| **키 갱신** | 주소 변경 없이 가능 | 새 컨트랙트 배포 필요 |
| **복구** | Share 재생성 가능 | 키 분실 시 복구 어려움 |

## 6. Trust 계좌 / FBO 계좌

### 6.1 FBO (For Benefit Of) 계좌

**정의**: 한 주체(Trustee)가 다른 주체(Beneficiary)를 위해 자금을 관리하는 계좌

```
┌─────────────────────────────────────────────────────────────────┐
│                    FBO 계좌 구조                                │
└─────────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────┐
│                                                               │
│   ┌─────────────────────────────────────────────────────────┐│
│   │              FBO Account at Bank                        ││
│   │              "ABC Corp FBO Customers"                   ││
│   │                                                         ││
│   │   법적 소유자: ABC Corp (계좌 관리자)                   ││
│   │   실질 소유자: 고객들 (수익자)                          ││
│   │                                                         ││
│   │   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐  ││

## 7. Burn After Settlement 패턴

### 7.1 현재 업계 관행 (문제점)

**대부분의 스테이블코인은 "Burn First, Settle Later" 방식**:

```
현재 일반적인 흐름 (위험한 방식):

1. 사용자: 상환 요청
2. 스마트 컨트랙트: 즉시 토큰 burn() 실행 ← 온체인 즉시 반영
3. 발행자: 은행 송금 처리 시작
4. 은행: 2-3 영업일 후 송금 완료 ← 오프체인 지연

문제:
- 토큰은 소각됐는데 돈은 아직 안 감
- 은행 송금 실패 시 토큰 복구 불가
- 스트레스 상황에서 상환 대기열 발생
```

### 7.2 USDC 사례 (2023년 3월 SVB 사태)

```
┌─────────────────────────────────────────────────────────────────┐
│                    USDC Depegging 사태 (2023.03)               │
└─────────────────────────────────────────────────────────────────┘

Circle 발표:
"USDC의 발행과 상환은 미국 은행 시스템의 영업 시간에 제약됩니다"

결과:
- 상환 요청 대기열(Backlog) 발생
- 2차 시장에서 USDC가 $0.87까지 하락
- Primary market과 secondary market 가격 괴리

교훈:
- 온체인 burn은 즉시, 오프체인 settlement는 지연
- 이 시차가 위기 상황에서 증폭됨
```

### 7.3 권장 패턴: Burn After Settlement

```
┌─────────────────────────────────────────────────────────────────┐
│              Burn After Settlement 권장 흐름                    │
└─────────────────────────────────────────────────────────────────┘

1. 사용자: 상환 요청 (requestRedemption)
   │
   ▼
2. 스마트 컨트랙트: 토큰을 에스크로에 잠금 (transfer to escrow)
   │                토큰은 아직 burn 안 됨!
   │                사용자는 더 이상 사용 불가
   ▼
3. 오프체인 시스템: AML 스크리닝
   │
   ▼
4. 은행: 법정화폐 송금 실행
   │
   ▼
5. 발행자: 송금 완료 확인 (은행 API/웹훅)
   │
   ▼
6. 스마트 컨트랙트: completeRedemption() → burn() 실행
   │
   ▼
완료: 토큰 소각됨, 법정화폐 도착 확인됨

실패 케이스:
- 송금 실패 시 → cancelRedemption() → 에스크로에서 사용자에게 토큰 반환
- 토큰 손실 없음!
```

### 7.4 구현 예시

```solidity
contract RedemptionManager {
    enum Status { PENDING, COMPLETED, CANCELLED }

    struct Redemption {
        address user;
        uint256 amount;
        Status status;
        uint256 requestTime;
        bytes32 bankTxRef;  // 은행 송금 참조번호
    }

    mapping(bytes32 => Redemption) public redemptions;
    address public escrow;

    // Step 1: 상환 요청 - 토큰을 에스크로로 이동 (burn 아님!)
    function requestRedemption(uint256 amount) external returns (bytes32) {
        bytes32 id = keccak256(abi.encodePacked(msg.sender, amount, block.timestamp));

        // 토큰을 에스크로로 이동 (아직 burn 안 함)
        token.transferFrom(msg.sender, escrow, amount);

        redemptions[id] = Redemption({
            user: msg.sender,
            amount: amount,
            status: Status.PENDING,
            requestTime: block.timestamp,
            bankTxRef: bytes32(0)
        });

        emit RedemptionRequested(id, msg.sender, amount);
        return id;
    }

    // Step 2: 송금 완료 후에만 burn
    function completeRedemption(
        bytes32 id,
        bytes32 bankTxRef
    ) external onlyOperator {
        Redemption storage r = redemptions[id];
        require(r.status == Status.PENDING, "Not pending");

        // 송금 완료 확인 후 burn
        token.burnFrom(escrow, r.amount);

        r.status = Status.COMPLETED;
        r.bankTxRef = bankTxRef;

        emit RedemptionCompleted(id, bankTxRef);
    }

    // 실패 시: 토큰 반환
    function cancelRedemption(bytes32 id) external onlyOperator {
        Redemption storage r = redemptions[id];
        require(r.status == Status.PENDING, "Not pending");

        // 에스크로에서 사용자에게 토큰 반환
        token.transferFrom(escrow, r.user, r.amount);

        r.status = Status.CANCELLED;

        emit RedemptionCancelled(id);
    }
}
```

│  │                                                         │ │
│  │  보고 항목:                                             │ │
│  │  1. 총 발행 토큰 수량                                   │ │

│  │                                                         │ │

│  │     - 수탁기관 관리                                     │ │

│  │  4. 벤더 관리(Vendor Management) 통제                   │ │

│  │                                                         │ │

## 11. AT-C 205: Assertion-Based Examination Engagements

### 11.1 개요

AT-C 205는 **AICPA의 증명 업무 기준**으로, CPA가 기업 경영진의 "주장(Assertion)"에 대해 높은 수준의 확신을 제공하는 감사 서비스 표준이다.

### 11.2 증명 업무 유형

```
┌─────────────────────────────────────────────────────────────────┐
│                    AICPA 증명 업무 유형                         │
└─────────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────┐
│                                                               │
│  AT-C Section 105: 모든 증명 업무에 공통                      │
│         │                                                     │
│         ├─────────────────────────────────────────────────┐  │
│         │                                                 │  │
│         ▼                                                 ▼  │
│  ┌─────────────────┐  ┌─────────────────┐  ┌────────────────┐│
│  │  AT-C 205       │  │  AT-C 206       │  │  AT-C 215      ││
│  │                 │  │                 │  │                ││
│  │  Assertion-Based│  │  Direct         │  │  Agreed-Upon   ││
│  │  Examination    │  │  Examination    │  │  Procedures    ││
│  │                 │  │                 │  │                ││
│  │  경영진 주장에  │  │  CPA가 직접    │  │  합의된 절차   ││
│  │  대한 검토      │  │  측정/평가     │  │  만 수행       ││
│  │                 │  │                 │  │                ││
│  │  높은 확신      │  │  높은 확신     │  │  확신 없음     ││
│  │  (Reasonable)   │  │  (Reasonable)  │  │  (No Opinion)  ││
│  │                 │  │                 │  │                ││
│  └─────────────────┘  └─────────────────┘  └────────────────┘│
│                                                               │
│  스테이블코인 증명에 적용:                                    │
│  - AT-C 205: 경영진이 "준비금 ≥ 발행량" 주장 → CPA가 검증    │
│  - AT-C 215: 특정 절차만 수행 (예: 은행 잔액 확인만)          │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 11.3 AT-C 205 핵심 요구사항

```
┌─────────────────────────────────────────────────────────────────┐
│                    AT-C 205 검토 프로세스                       │
└─────────────────────────────────────────────────────────────────┘

Step 1: 사전 요건
┌───────────────────────────────────────────────────────────────┐
│  - CPA의 독립성 확인                                          │
│  - 경영진으로부터 서면 주장(Written Assertion) 수령           │
│  - 적합한 기준(Criteria) 사용 여부 확인                       │
│    (예: AICPA 2025 Stablecoin Reporting Criteria)             │
└───────────────────────────────────────────────────────────────┘
                              │
                              ▼
Step 2: 계획 및 위험 평가
┌───────────────────────────────────────────────────────────────┐
│  - 중요한 왜곡표시(Material Misstatement) 위험 식별           │
│  - 절차의 성격, 시기, 범위 결정                               │
│  - 내부 통제 이해                                             │
└───────────────────────────────────────────────────────────────┘
                              │
                              ▼
Step 3: 증거 수집
┌───────────────────────────────────────────────────────────────┐
│  수행 절차:                                                   │
│  - 기록 및 문서 검사 (은행 확인서, 수탁 보고서)               │
│  - 프로세스 관찰 (발행/상환 절차)                             │
│  - 관련자 질의 (경영진, 수탁기관)                             │
│  - 온체인 데이터 확인 (totalSupply, 블록 타임스탬프)          │
│  - 재계산/재수행 (준비금 vs 발행량 계산)                      │
│                                                               │
│  목표: 증명 위험을 수용 가능한 수준으로 낮추기                │
└───────────────────────────────────────────────────────────────┘
                              │
                              ▼
Step 4: 경영진 확인서(Representation Letter) 수령
┌───────────────────────────────────────────────────────────────┐
│  - 보고서 날짜 기준으로 작성                                  │
│  - 경영진의 책임 재확인                                       │
│  - 제공 정보의 완전성 확인                                    │
└───────────────────────────────────────────────────────────────┘
                              │
                              ▼
Step 5: 보고서 발행
┌───────────────────────────────────────────────────────────────┐
│  보고서 필수 요소:                                            │
│  - "Independent" 포함 제목                                    │
│  - 경영진 주장 식별                                           │
│  - 사용된 기준 명시                                           │
│  - CPA의 책임 설명                                            │
│  - 독립성 및 윤리 준수 진술                                   │
│  - 의견(Opinion)                                              │
│    - 무한정 의견(Unqualified): 모든 중요 측면에서 공정       │
│    - 한정 의견(Qualified): 일부 제한 사항 존재               │
│    - 부정적 의견(Adverse): 중요한 왜곡표시 존재              │
└───────────────────────────────────────────────────────────────┘
```

### 11.4 스테이블코인 증명 적용 예시

```
경영진 주장 (Management Assertion):
"2025년 12월 31일 기준, ABC Stablecoin의 총 발행량 1,000,000,000 토큰에
대해 USD 1,000,000,000 이상의 적격 준비금이 분리 보관되어 있습니다."

CPA 검토 절차:
1. 은행 확인서로 준비금 잔액 확인
2. 온체인에서 totalSupply() 확인
3. 수탁기관 보고서로 분리 보관 확인
4. 조정표 검토

CPA 의견 (예시):
"우리 의견으로는, 위의 주장은 AICPA 2025 Criteria for Stablecoin Reporting에
따라 모든 중요한 측면에서 공정하게 표시되어 있습니다."
```
