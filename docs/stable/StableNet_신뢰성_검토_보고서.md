## 2. 기술적 주장 검증

### 2.1 WBFT 합의 알고리즘

#### 문서 주장
- go-ethereum, QBFT 구현체 기반
- 1초 블록 타임, 1 block 블록 확정
- 3000 TPS
- 검증된 비잔틴 장애 내성

#### 검증 결과

| 주장 | 기술적 검증 | 판정 |
|------|------------|------|
| QBFT 기반 | Hyperledger Besu에서 검증된 BFT 변형 | ✅ 실현 가능 |
| 1초 블록타임 | QBFT 설정 가능, Polygon PoS도 2초 달성 | ✅ 가능 |
| 3000 TPS | 허가형 검증자 구조에서 달성 가능 | ⚠️ 조건부 |
| 1 block finality | BFT 계열 합의 특성상 가능 | ✅ 가능 |

#### 미해결 의문점

1. **검증자 수 미공개**: 3000 TPS는 검증자 수가 제한적일 때만 가능
2. **비잔틴 내성 조건**: 검증자 2/3 이상 정직 필요 → 검증자 선정 기준 불명확
3. **증거 부재**: 실제 부하 테스트 결과나 감사 보고서 미제시

### 2.2 EIP-7702 (Native Account Abstraction)

#### 문서 주장
- EIP-7702 구현
- Smart Account로 수수료 대납 가능
- Private key 없이 계정 기반 접근 제어

#### 검증 결과

| 주장 | 기술적 검증 | 판정 |
|------|------------|------|
| EIP-7702 구현 | 2024년 Pectra 업그레이드 예정 표준 | ⚠️ 미검증 |
| 수수료 대납 | EIP-7702로 이론상 가능 | ✅ 가능 |

#### 미해결 의문점

1. EIP-7702는 이더리움 메인넷에 아직 미적용 (2024년 말~2025년 예정)
2. StableNet이 먼저 구현했다면 자체 포크 → 향후 호환성 이슈 가능
3. 구현체나 감사 보고서 미제시

### 2.3 NativeCoinAdapter (ERC20 Wrapper)

#### 문서 주장
- Native coin을 ERC20 인터페이스로 제공
- 기존 ERC20 스테이블코인과 호환

#### 검증 결과

| 주장 | 기술적 검증 | 판정 |
|------|------------|------|
| ERC20 wrapper | WETH 패턴과 유사, 검증된 방식 | ✅ 실현 가능 |
| Circle Arc 동일 방식 | Arc 문서에서 유사 접근 확인 | ✅ 일치 |

**평가**: 기술적 난이도 낮음, 신뢰도 높음

### 2.4 Secret Account (ERC-5564 Stealth Address)

#### 문서 주장
- ERC-5564 활용하여 주소 추적 난해화
- 금액 비공개 (급여, 기관 송금용)
- 규제 당국 요청 시 공개 가능

#### 검증 결과

| 주장 | 기술적 검증 | 판정 |
|------|------------|------|
| ERC-5564 활용 | 2023년 제안, 아직 Draft 상태 | ⚠️ 미성숙 |
| Secret Transfer Server 배치 처리 | 중앙 서버 의존 구조 | ❌ 탈중앙 아님 |
| 규제 요청 시 공개 | 서버가 모든 내역 보유 | ✅ 가능 |

#### 심각한 문제점

```
"Secret"이라는 명칭과 실제 구조의 괴리:

1. Secret Transfer Server가 모든 송수신 정보를 보유
2. 서버 해킹 시 전체 프라이버시 붕괴
3. 서버 운영자가 모든 정보 열람 가능
4. 결론: 사용자 간에만 secret, 운영자에게는 투명

→ "Stealth Account"보다 "Escrow Account"에 가까운 구조
→ 진정한 프라이버시 솔루션이라 보기 어려움
```

## 3. 경제 모델 검증

### 3.1 "1원 수수료" 주장 분석

#### 문서 주장
- 평상시 기본 트랜잭션 수수료 1원
- DoS 공격 방어를 위해 블록 혼잡시 BaseFee 증가
- 마이너들이 단독으로 priority fee를 올릴 수 없음
- 블록 생성 보상 없음 (No Inflation)

#### 핵심 의문

```
Q: 블록 생성 보상 없음 + 1원 수수료 → 검증자는 왜 노드를 운영하는가?

문서에서 이 부분에 대한 설명이 전혀 없음.

가능한 시나리오:
1. Minter들이 검증자 역할 수행 (이해관계 일치로 무보상 감수)
2. 재단/운영사가 검증자에게 보조금 지급
3. 다른 수익 모델 존재 (미공개)

→ 검증자 인센티브 구조가 명확하지 않으면 체인 지속 가능성에 심각한 의문
```

#### 경제성 검토

| 항목 | 분석 |
|------|------|
| 1원 수수료 | 가스 단가 고정 시 가능하나, 유지 비용 대비 수익성 의문 |
| BaseFee 증가 | EIP-1559 방식이면 혼잡 시 급등 가능 |
| Priority fee 제한 | 검증자 인센티브 추가 박탈 → 운영 동기 약화 |

### 3.2 Minter 비즈니스 모델

#### 문서 구조
```
[원화 입금] → [Minter] → [스테이블코인 발행] → [End User 유통]
                ↓
         [원화 출금] ← [Minter만 가능]
```

#### 구조적 문제점

| 구조 | 문제점 |
|------|--------|
| 일반인 직접 출금 불가 | 사용자 불편, Minter 독점적 위치 |
| Minter만 출금 가능 | 유동성 병목, 수수료 남용 가능 |

#### 리스크 시나리오

```
사용자가 StableNet 코인을 원화로 교환하고 싶을 때:

1. Minter(예: 페이업체)에게 요청 필수
2. Minter가 수수료를 임의로 부과 가능 (독점적 위치)
3. Minter가 지급 거부하면? → 사용자 구제책 불명확
4. Minter 부도 시? → 예치금 보호 방안 미설명

"규제 관리를 위해"라는 명목이지만,
실제로는 Minter 이익 보호 및 사용자 lock-in 구조
```

### 3.3 Proof of Reserve 검증

#### 문서 주장
- Certik Proof of Reserve 서비스 사용 가능
- 모든 발행 트랜잭션에 입금 proof 온체인 기록

#### 미확인 사항
- 실시간 PoR인지 정기 감사인지 불명확
- 은행 예치금과 온체인 발행량 불일치 시 대응 방안 미설명
- 감사 주기 및 공개 방식 미정의

## 4. 규제 현실성 검증

### 4.1 한국 스테이블코인 규제 현황

| 규제 현황 | StableNet 대응 | 평가 |
|----------|---------------|------|
| 2024년 가상자산법 시행 | KYC/AML 언급 있음 | ⚠️ 부분적 |
| 스테이블코인 별도 규제 미정 | "향후 규제 대응 커스터마이징" | ⚠️ 불확실 |
| 원화 연동 스테이블코인 | 언급 없음 | ❌ 미검토 |

### 4.2 법적 리스크 분석

#### 핵심 법적 문제

```
한국에서 원화 1:1 스테이블코인 발행 시 검토 필요 사항:

1. 전자금융업 라이선스
   - 전자금융거래법상 선불전자지급수단 해당 가능성
   - 금융위원회 등록/인가 필요 여부

2. 은행법 저촉
   - 은행업 인가 없이 예금 유사 서비스 제공
   - "원화 담보 보관" = 예금 수취로 해석 가능

3. 자본시장법
   - 투자계약증권 해당 여부
   - 증권신고서 제출 의무

4. 외환거래법
   - Bridge 통한 해외 체인 이전 시 외환 신고 필요 가능
   - 해외 거래소 상장 시 자본거래 신고

→ 문서에서 이 법적 검토가 전혀 없음
→ 사업 진행 전 반드시 금융법 전문 법률 검토 필요
```

### 4.3 Blacklist 기능의 법적 정당성

#### 문서 주장
- 해커 계정 즉시 동결
- Chainalysis, Certik 연동으로 이상 감지
- 긴급 시 블랙리스팅

#### 법적 의문점

| 의문 | 내용 |
|------|------|
| 결정 권한 | 누가 동결을 결정하는가? |
| 적법 절차 | 영장 없이 자산 동결 가능한가? |
| 민간 판단 | Chainalysis(민간 기업) 판단으로 동결? |
| 오류 대응 | 오탐으로 정상 계정 동결 시 구제책? |
| 해제 절차 | 동결 해제 요건 및 절차 불명확 |

#### 남용 리스크

```
Blacklist 남용 가능 시나리오:

1. 오탐으로 무고한 사용자 자산 동결
2. 경쟁사 또는 정치적 이유로 특정 계정 동결
3. Governance 권한자의 권한 남용
4. 동결 해제 지연으로 인한 피해

"탈중앙"을 표방하면서 중앙화된 동결 권한 보유
→ 철학적 모순 및 신뢰 이슈
```

### 7.2 SWOT 분석

#### Strengths (강점)
- 명확한 문제 정의와 솔루션 방향
- 검증된 기반 기술 (QBFT, EVM)
- 규제 친화적 설계 (Blacklist, KYC/AML)
- 경쟁 체인 대비 거버넌스 차별화

#### Weaknesses (약점)
- 검증자 경제학 미설명
- 법적 검토 부재
- 구현체/감사 증거 없음
- Secret Account 중앙화 구조

#### Opportunities (기회)
- 한국 스테이블코인 규제 정비 시 선점 가능
- 페이업체/은행의 블록체인 도입 수요
- 글로벌 스테이블코인 규제 강화 트렌드

#### Threats (위협)
- 대형 플레이어 (Circle, Stripe, 빅테크) 진입
- 규제 불확실성
- 테라 사태 이후 스테이블코인 신뢰도 하락
- 기술 경쟁 (CBDC, 토큰증권)
