# 규제 준수 컨트랙트 기술 구현 가이드

## 개요

글로벌 스테이블코인 규제(일본, 싱가포르, EU MiCA, 홍콩, 영국, 미국 GENIUS Act)의 공통 요건을 충족하기 위한 5개 핵심 컨트랙트의 **온체인/오프체인 통합 아키텍처**와 **감사 대응 기술 요건**을 정리한다.

## 1. ProofOfReserves.sol - 준비금 증명

### 1.1 목적
- 100% 준비금 보유 증명 (1:1 backing)
- 실시간 또는 주기적 준비금 검증

### 1.3 프로토콜 옵션

#### Option A: Chainlink Proof of Reserve (권장)

**작동 방식:**
1. 독립 회계법인(예: TNF)이 은행 계좌 잔액을 실시간 집계
2. Chainlink 노드 네트워크가 데이터를 검증하고 온체인에 게시
3. 스마트 컨트랙트가 오라클 데이터를 읽어 mint 함수에서 검증

**실제 사례 - TUSD:**
```solidity
// TUSD 구현 예시 (line 1,404 참조)
interface AggregatorV3Interface {
    function latestRoundData() external view returns (
        uint80 roundId,
        int256 answer,
        uint256 startedAt,
        uint256 updatedAt,
        uint80 answeredInRound
    );
}

contract ProofOfReserves {
    // Ethereum Mainnet TUSD PoR 주소
    AggregatorV3Interface public reservesFeed;  // 0xBE456fd14720C3aCCc30A2013Bffd782c9Cb75D5
    AggregatorV3Interface public supplyFeed;    // 0x807b029DD462D5d9B9DB45dff90D3414013B969e

    function getReserves() public view returns (int256) {
        (, int256 reserves,,,) = reservesFeed.latestRoundData();
        return reserves;
    }

    function getTotalSupply() public view returns (int256) {
        (, int256 supply,,,) = supplyFeed.latestRoundData();
        return supply;
    }

    function isFullyBacked() public view returns (bool) {
        return getReserves() >= getTotalSupply();
    }
}
```

**Chainlink PoR 특징:**
- 17B+ USD 검증 실적, 40+ 활성 피드
- 24시간마다 업데이트 또는 5% 이상 잔액 변동 시 즉시 업데이트
- TUSD, Poundtoken, Cache Gold 등이 Secure Mint 기능 사용 중

#### Option B: Merkle Tree Proof of Reserves

**작동 방식:**
1. 거래소/발행자가 모든 사용자 잔액으로 Merkle Tree 구성
2. Root hash를 온체인에 게시
3. 개별 사용자가 자신의 잔액 포함 여부를 Merkle Proof로 검증

**Kraken 구현 사례:**
- 114.9% 준비금 비율 증명
- 사용자별 개별 검증 가능
- 제3자 감사인(Armanino LLP)이 검증

```solidity
contract MerkleProofOfReserves {
    bytes32 public reservesRoot;
    uint256 public totalReserves;
    uint256 public attestationTimestamp;
    address public auditor;

    event ReservesAttested(bytes32 root, uint256 total, uint256 timestamp);

    function updateReservesRoot(
        bytes32 _root,
        uint256 _totalReserves,
        bytes calldata _auditorSignature
    ) external {
        require(verifyAuditorSignature(_root, _totalReserves, _auditorSignature), "Invalid signature");

        reservesRoot = _root;
        totalReserves = _totalReserves;
        attestationTimestamp = block.timestamp;

        emit ReservesAttested(_root, _totalReserves, block.timestamp);
    }

    function verifyUserBalance(
        address user,
        uint256 balance,
        bytes32[] calldata proof
    ) external view returns (bool) {
        bytes32 leaf = keccak256(abi.encodePacked(user, balance));
        return MerkleProof.verify(proof, reservesRoot, leaf);
    }
}
```

### 2.3 규제 요구사항별 기술 구현

| 규제 요건 | 기술 구현 |
|----------|----------|
| **자산 분리 (GENIUS Act)** | 별도 Trust 계좌 또는 FBO(For Benefit Of) 계좌 사용 |
| **재담보화 금지** | 스마트 컨트랙트에서 출금 목적 제한 (상환만 허용) |

| **직무 분리** | Multi-sig: 트랜잭션 개시자 ≠ 승인자 |

## 3. RedemptionManager.sol - 1:1 상환

### 3.1 목적
- 스테이블코인과 법정화폐 간 1:1 액면가 상환 보장
- 시장 가격 하락 시 가격 바닥(Price Floor) 역할
- 상환 지연/실패 시 보호 메커니즘

### 3.2 온체인/오프체인 아키텍처

```
┌─────────────────────────────────────────────────────────────────┐
│                     User Redemption Flow                        │
└─────────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────────┐
│                        On-Chain Layer                           │
├─────────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────────────┐│

### 3.3 핵심 설계 원칙

#### "Burn After Settlement" 원칙
```
⚠️ 중요: 토큰 소각은 반드시 법정화폐 송금 확인 후에 수행

잘못된 순서:
1. 상환 요청 → 2. 토큰 burn → 3. 법정화폐 송금 (실패 가능!)
   → 토큰은 소각됐는데 돈은 안 간 상황 발생

올바른 순서:
1. 상환 요청 → 2. 토큰 에스크로 잠금 → 3. 법정화폐 송금 확인 → 4. 토큰 burn
   → 송금 실패 시 토큰 반환 가능
```

#### 차익거래(Arbitrage) 메커니즘
```
시장가 $0.99일 때:
  트레이더: 시장에서 $0.99에 구매 → $1.00에 상환 → $0.01 차익
  결과: 매수 압력 → 가격 상승 → $1.00 복귀

시장가 $1.01일 때:
  발행자: $1.00에 새로 발행 → 시장에 $1.01에 매도
  결과: 매도 압력 → 가격 하락 → $1.00 복귀
```

### 3.4 오프체인 구성요소

### 4.3 통합 패턴

#### Chainalysis KYT 연동 흐름
```
1. 사용자가 지갑 주소로 가입 요청
2. 오프체인 서비스가 Chainalysis KYT API 호출
   - 주소 위험도 점수 조회
   - 제재 대상 여부 확인
   - 과거 거래 패턴 분석
3. 위험도가 임계값 이하일 경우:
   - KYC 프로세스 진행
   - 완료 후 온체인 레지스트리 업데이트
4. 위험도가 높을 경우:
   - EDD(Enhanced Due Diligence) 요청
   - 또는 온보딩 거부
```

```
