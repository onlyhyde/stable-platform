## 2. StableNet 고유 차별점 (강점)

### 2.1 차별점 1: 원화 네이티브 L1 (유일)

#### 현황

| 체인 | 기반 통화 | 한국 시장 적합성 |
|------|----------|----------------|
| Arc | USD (USDC) | 환전 필요 |
| Tempo | 다중 (USD 중심) | 환전 필요 |
| Stable | USD (USDT) | 환전 필요 |
| **StableNet** | **KRW (원화)** | **직접 사용 가능** |

#### 차별점 분석

```yaml
경쟁사:
  기반: USD 스테이블코인
  한국 사용: 환전 필요 (USD → KRW)
  수수료: 환전 비용 발생
  UX: 복잡 (달러 단위 사고 필요)

StableNet:
  기반: 원화 스테이블코인
  한국 사용: 직접 사용
  수수료: 환전 비용 없음
  UX: 직관적 (원화 단위)
```

#### 전략적 의미

- **시장 규모**: 한국 5,100만 인구, GDP 세계 13위
- **선점 효과**: 원화 스테이블코인 전용 L1은 StableNet이 유일
- **규제 대응**: 한국 금융 규제에 맞춤 설계 가능
- **파트너십**: 국내 금융기관과 직접 연동 용이

#### 평가

| 항목 | 점수 | 근거 |
|------|------|------|
| 차별화 강도 | ⭐⭐⭐⭐⭐ | 유일한 원화 전용 L1 |
| 지속가능성 | 높음 | Arc/Tempo의 한국 시장 직접 진출 어려움 |
| 방어 가능성 | 높음 | 선점 효과 + 규제 대응 |

### 2.2 차별점 2: Minter 시스템 (규제 친화적 발행 구조)

#### 경쟁사 비교

| 체인 | 발행 구조 | 발행 주체 |
|------|----------|----------|
| Arc | Circle 직접 발행 | Circle (단일) |
| Tempo | Stripe 중심 발행 | Stripe + 파트너 |
| **StableNet** | **Minter 시스템** | **인가된 금융기관 (다수)** |

#### Minter 시스템 구조

```
┌─────────────────────────────────────────────────────────────┐
│                    StableNet Minter 시스템                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────┐                                        │
│  │ GovMasterMinter │  ← 정책/감독 (거버넌스)                  │
│  └────────┬────────┘                                        │
│           │ 인가/감독                                        │
│           ▼                                                 │
│  ┌─────────────────────────────────────────┐               │
│  │              Minter Pool                 │               │
│  │  ┌───────┐ ┌───────┐ ┌───────┐         │               │
│  │  │ 은행A │ │ 은행B │ │ PG사  │  ...    │               │
│  │  │ (NH)  │ │ (KB)  │ │(토스) │         │               │
│  │  └───┬───┘ └───┬───┘ └───┬───┘         │               │
│  └──────┼─────────┼─────────┼─────────────┘               │
│         │         │         │                              │
│         ▼         ▼         ▼                              │
│  ┌─────────────────────────────────────────┐               │
│  │           mint / burn 권한               │               │
│  └─────────────────────────────────────────┘               │
│                     │                                       │
│                     ▼                                       │
│  ┌─────────────────────────────────────────┐               │
│  │              일반 사용자                  │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 강점

| 항목 | 설명 |
|------|------|
| 규제 수용성 | 기존 금융기관이 발행 주체 → 규제 당국 친화적 |
| AML/KYC | 은행/PG사의 기존 AML 인프라 활용 |
| 신뢰도 | 은행 브랜드 활용 → 사용자 신뢰 확보 |
| 온보딩 | 기존 계좌 보유자 쉽게 유입 |
| 분산화 | 다수 Minter → 단일 장애점 방지 |

#### 약점

| 항목 | 설명 |
|------|------|
| 진입장벽 | Minter 자격 제한 → 생태계 성장 제약 |
| 탈중앙화 | 인가된 기관만 참여 → 완전한 탈중앙화 아님 |
| 의존성 | Minter 정책에 사용자 종속 |

#### 평가

| 항목 | 점수 | 근거 |
|------|------|------|
| 차별화 강도 | ⭐⭐⭐⭐ | 규제 친화적 설계 |
| 지속가능성 | 높음 | 한국 금융 규제 구조와 정합 |
| 방어 가능성 | 중간 | 경쟁사도 유사 구조 도입 가능 |

### 2.3 차별점 3: Secret Account (프라이버시 계층)

#### 경쟁사 비교

| 체인 | 프라이버시 기술 | 규제 대응 |
|------|---------------|----------|
| Arc | TEE (Trusted Execution Environment) | 선택적 공개 (view keys) |
| Tempo | 미공개 | 미공개 |
| **StableNet** | **Secret Account (ERC-5564 변형)** | **규제 기관 조회 가능** |

#### Secret Account 구조

```
┌─────────────────────────────────────────────────────────────┐
│                  StableNet Account 유형                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────┐         ┌─────────────────┐           │
│  │  Normal Account │         │  Secret Account │           │
│  │     (기본)      │         │    (프라이버시)  │           │
│  ├─────────────────┤         ├─────────────────┤           │
│  │ • 잔액 공개     │         │ • 잔액 비공개    │           │
│  │ • 거래 공개     │         │ • 거래 비공개    │           │
│  │ • 블록체인 조회 │         │ • 서버 통해 조회 │           │
│  └─────────────────┘         └────────┬────────┘           │
│                                       │                     │
│                              ┌────────▼────────┐           │
│                              │ Secret Transfer │           │
│                              │     Server      │           │
│                              └────────┬────────┘           │
│                                       │                     │
│                              ┌────────▼────────┐           │
│                              │   규제 기관      │           │
│                              │  (필요시 조회)   │           │
│                              └─────────────────┘           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 차별점: "규제 친화적 프라이버시"

```yaml
Arc TEE 방식:
  프라이버시: 높음 (하드웨어 기반)
  규제 대응: view keys로 선택적 공개
  사용자 통제: 사용자가 공개 여부 결정

StableNet Secret Account:
  프라이버시: 중간 (서버 기반)
  규제 대응: 규제 기관 직접 조회 가능
  사용자 통제: 제한적 (서버가 정보 보유)

포지셔닝:
  "일반인에게는 프라이버시, 규제 기관에는 투명성"
```

#### 주의사항

```
⚠️ 설계상 문제점:
- Secret Transfer Server가 모든 정보 보유
- 진정한 Stealth Address가 아님 (중앙화)
- "Secret"이라는 명칭이 오해 유발 가능

개선 필요:
- 명칭 변경: "Privacy Account" 또는 "Escrow Account"
- 또는 진정한 온체인 Stealth 구현 (ZK-proof 활용)
```

#### 평가

| 항목 | 점수 | 근거 |
|------|------|------|
| 차별화 강도 | ⭐⭐⭐ | 규제 친화적 설계이나 기술적 한계 |
| 지속가능성 | 중간 | 설계 개선 필요 |
| 방어 가능성 | 낮음 | Arc TEE가 더 강력한 기술 |

  심리적 장벽: 없음

예시:
  Arc: "This transaction costs $0.001"
