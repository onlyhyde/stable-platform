## 1. 개요

### 1.1 분석 배경

한국에서 스테이블코인 기반 L1 블록체인 및 결제 비즈니스를 추진하려는 사업자들이 직면하는 규제적 문제점을 분석합니다. 글로벌 시장에서는 Circle Arc, Stripe Tempo 등 스테이블코인 네이티브 L1 체인들이 활발히 개발되고 있으나, 한국에서는 법적 프레임워크 부재와 규제 당국의 보수적 접근으로 인해 사업 추진에 어려움이 있습니다.

### 1.2 현황 요약

| 구분 | 글로벌 | 한국 |
|------|--------|------|
| 스테이블코인 전용 규제 | MiCA (EU), 주별 규제 (미국) | 미정 |
| 스테이블코인 L1 체인 | Arc, Tempo, Stable 등 개발 중 | 없음 |
| 원화 스테이블코인 | - | 법적 검토 부재 |
| 금융기관 참여 | Circle-BlackRock 파트너십 등 | 관망 중 |

## 2. 법적 프레임워크 부재 문제

### 2.1 적용 가능한 법률의 불명확성

원화 스테이블코인 발행 시 어떤 법률이 적용되는지 명확하지 않아 사업자들이 법적 리스크를 감수해야 하는 상황입니다.

| 법률 | 적용 가능성 | 문제점 |
|------|------------|--------|
| **전자금융거래법** | 선불전자지급수단 해당 가능 | 금융위 등록/인가 필요 여부 불분명 |
| **은행법** | "원화 담보 보관" = 예금 수취로 해석 가능 | 은행업 인가 없이 운영 시 위법 가능 |
| **자본시장법** | 투자계약증권 해당 가능 | 증권신고서 제출 의무 발생 가능 |
| **외환거래법** | 타체인 브릿지 시 외환 신고 필요 | 글로벌 연동 시 복잡성 증가 |
| **가상자산법** | 가상자산사업자 신고 의무 | KYC/AML만 규정, 발행 규제 부재 |

### 2.2 스테이블코인 전용 규제 부재

```
한국 규제 현황:

2024년 가상자산법 시행
├── 규정 사항: KYC/AML, 가상자산사업자 신고
├── 미규정 사항: 스테이블코인 발행, 담보 요건, 준비금
└── 원화 연동 스테이블코인: 법적 검토 자체가 부재

결과:
→ "향후 규제 대응 커스터마이징"이라는 애매한 상태로 대기
→ 사업자는 법적 리스크를 감수하거나 사업 포기 선택지만 존재
```

### 2.3 법적 검토 필요 사항

한국에서 원화 1:1 스테이블코인 발행 시 반드시 검토가 필요한 법률 사항:

```
1. 전자금융업 라이선스
   - 전자금융거래법상 선불전자지급수단 해당 가능성
   - 금융위원회 등록/인가 필요 여부
   - 기존 선불전자지급수단과의 차이점 정리

2. 은행법 저촉 여부
   - 은행업 인가 없이 예금 유사 서비스 제공 가능 여부
   - "원화 담보 보관" = 예금 수취로 해석될 가능성
   - 예금자보호법 적용 범위

3. 자본시장법 적용
   - 투자계약증권 해당 여부 (Howey Test 기준)
   - 증권신고서 제출 의무
   - 집합투자업 인가 필요 여부

4. 외환거래법 적용
   - Bridge 통한 해외 체인 이전 시 외환 신고 필요
   - 해외 거래소 상장 시 자본거래 신고
   - 비거주자 거래 시 신고 의무
```

## 3. 금융기관 참여의 구조적 장벽

### 3.1 Minter(발행 주체) 유치의 어려움

스테이블코인 생태계에서 핵심이 되는 발행 기관(Minter) 유치가 구조적으로 어려운 상황입니다.

#### 이상적 구조

```
┌─────────────────────────────────────────────────────────────────┐
│                    이상적 Minter 시스템                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [인가된 금융기관] ─────────────────────────────────────────→    │
│       │                                                         │
│       ├── 은행 (NH농협, KB국민, 신한 등)                         │
│       ├── PG사 (토스, NHN, KG이니시스 등)                        │
│       └── 카드사 (BC카드, 신한카드 등)                           │
│               │                                                 │
│               ▼                                                 │
│       ┌─────────────────────────────┐                          │
│       │    스테이블코인 발행 (Minter)  │                          │
│       └─────────────────────────────┘                          │
│               │                                                 │
│               ▼                                                 │
│       [End User 유통]                                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 현실적 장벽

| 장벽 유형 | 내용 |
|----------|------|
| **자체 솔루션 선호** | 대형 금융기관은 외부 블록체인보다 자체 개발 선호 |
| **CBDC 경쟁** | 한국은행 CBDC 추진으로 인한 민간 스테이블코인 위축 |
| **빅테크 진출** | 토스/카카오페이 등이 자체 블록체인 개발 가능성 |
| **법적 불확실성** | 적법성 미검증 사업 참여 시 금융 인허가 리스크 |
| **책임 소재** | 문제 발생 시 책임 범위가 불명확 |

### 3.2 규제 당국과의 사전 협의 부재

사업 추진을 위해 필요하지만 진행이 어려운 사항들:

```
필요 사항:
□ 금융위원회 사전 협의
  - 스테이블코인의 법적 성격 규명
  - 적용 가능한 인허가 유형 확인
  - 규제 샌드박스 적용 가능 여부

□ 한국은행 협의
  - CBDC 방향과의 정합성 논의
  - 통화정책 영향 평가
  - 금융 안정성 검토

□ 금융감독원 협의
  - 감독 방안 논의
  - 소비자 보호 기준 설정
  - 보고 의무 범위 확정

현실:
→ 명확한 협의 채널 부재
→ 사전 협의 요청 시 "법률 정비 후 논의"로 회신
→ 선례 없는 사업에 대한 보수적 접근
```

### 3.3 한국 시장 경쟁 구도

```
원화 스테이블코인 시장 잠재적 경쟁자:

1. 거래소 자체 토큰
   - 업비트/빗썸 등의 자체 원화 페깅 토큰 가능성
   - 이미 대규모 사용자 기반 보유

2. 빅테크
   - 카카오페이/토스 등의 블록체인 진출 가능성
   - 기존 결제 인프라 활용 가능

3. 은행권
   - 토큰증권/CBDC 자체 추진
   - 기존 규제 관계 활용

4. 글로벌 플레이어
   - Circle Arc, Stripe Tempo 한국 진출 시
   - 압도적 자금력과 파트너십 보유

→ 스타트업/신규 사업자의 경쟁 환경 매우 열악
```

## 4. 테라 붕괴 이후 신뢰 문제

### 4.1 스테이블코인에 대한 부정적 인식

2022년 테라USD(UST) 붕괴가 한국 스테이블코인 생태계에 미친 영향:

```
테라USD 붕괴 영향 분석:

규모:
- 약 40조원 규모 시가총액 증발
- 한국 투자자 대규모 피해
- 발행사 대표 법적 처벌

결과:
├── 규제 당국: 보수적 접근 강화
├── 일반 대중: "스테이블코인 = 위험" 인식 형성
├── 금융기관: 스테이블코인 관련 사업 기피
└── 정치권: 강력한 규제 요구
```

### 4.2 신뢰 회복을 위한 필수 요건

| 요건 | 내용 | 현황 |
|------|------|------|
| **담보 증명 (PoR)** | 실시간 예치금 증명 | 기준 미정립 |
| **감사 체계** | 정기적 외부 감사 | 감사 주기/방식 미정의 |
| **보험/보증** | 담보 불일치 시 보상 | 보험 상품 부재 |
| **거버넌스** | 투명한 의사결정 구조 | 표준 가이드라인 부재 |

### 4.3 담보 증명(Proof of Reserve)의 한계

```
현재 PoR 체계의 문제점:

1. 실시간 vs 정기 감사
   - 어느 방식을 적용해야 하는지 기준 불명확
   - 실시간 검증 시 기술적 복잡성

2. 불일치 대응
   - 은행 예치금과 발행량 불일치 시 대응 방안 미정의
   - 자동 발행 중지/소각 메커니즘 부재

3. 감사 신뢰도
   - 감사 기관의 자격 요건 미정립
   - 감사 결과 공개 의무 불명확

4. 법적 효력
   - PoR 감사 결과의 법적 효력 불분명
   - 감사 실패 시 책임 소재 불명확
```

## 5. 탈중앙화 vs 규제 친화성 딜레마

### 5.1 본질적 충돌 구조

블록체인 기반 결제 시스템이 직면하는 근본적인 트릴레마:

```
┌─────────────────────────────────────────────────────────────────┐
│                  블록체인 결제의 트릴레마                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                          탈중앙화                                │
│                             △                                   │
│                            / \                                  │
│                           /   \                                 │
│                          /     \                                │
│                         /   ?   \                               │
│                        /         \                              │
│                       /           \                             │
│                      ▽─────────────▽                            │
│                 규제 친화성      사용 편의성                      │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ 기존 퍼블릭 체인:                                          │  │
│  │   탈중앙화 ✓, 규제 친화성 ✗, 사용 편의성 ✗               │  │
│  │                                                            │  │
│  │ 기존 프라이빗 체인:                                         │  │
│  │   탈중앙화 ✗, 규제 친화성 ✓, 사용 편의성 △               │  │
│  │                                                            │  │
│  │ 이상적 해결:                                                │  │
│  │   세 가지 모두 균형 있게 달성 (현실적으로 어려움)           │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 규제 대응 불가 시나리오

문서에서 명시한 핵심 문제:

```
시나리오:

규제당국: "지금 이 블록체인이 범죄자의 자금세탁처로 활용되고 있습니다.
          규제를 해야겠습니다."

운영자:   "저희는 수억개의 계좌가 있고, 전세계에 수백만개의 노드로 운영되는
          탈중앙 체인입니다. 너네 나라에서 규제할 수 없습니다."

내면적 문제:
- AML/KYC 적용 불가능한 구조
- 자금세탁, 탈세, 범죄 수익 은닉 수단화
- 금융 당국 협조 불가
- 제도권 편입 불가 → 대중화 장벽
```

### 5.3 규제 친화적 설계의 Trade-off

| 규제 친화적 기능 | 탈중앙화 훼손 |
|-----------------|--------------|
| Blacklist/동결 | 중앙 권력 집중 |
| KYC/AML 연동 | 익명성 상실 |
| Minter 시스템 | 발행 권한 제한 |
| 거버넌스 위원회 | 의사결정 중앙화 |
| Secret Account 서버 | 프라이버시 서버 의존 |

## 6. Blacklist/동결 기능의 법적 정당성

### 6.1 법적 의문점

규제 친화적 설계의 핵심인 Blacklist/동결 기능에 대한 법적 검토 필요:

| 의문 | 내용 | 현황 |
|------|------|------|
| **결정 권한** | 누가 동결을 결정하는가? | 거버넌스 구조 미정의 |
| **적법 절차** | 영장 없이 자산 동결 가능한가? | 법적 근거 부재 |
| **민간 판단** | Chainalysis 등 민간 기업 판단으로 동결 가능? | 권한 위임 근거 불명확 |
| **오류 대응** | 오탐으로 정상 계정 동결 시 구제책? | 구제 절차 미정의 |
| **해제 절차** | 동결 해제 요건 및 절차는? | 기준 미정립 |

### 6.2 Blacklist 남용 리스크

```
남용 가능 시나리오:

1. 오탐에 의한 무고한 사용자 피해
   - AI/알고리즘 오류로 정상 거래 의심
   - 자산 동결로 인한 금전적 손실
   - 해제까지 장기간 소요

2. 정치적/경쟁적 남용
   - 경쟁사 또는 정치적 이유로 특정 계정 동결
   - 거버넌스 권한자의 권한 남용
   - 외부 압력에 의한 동결

3. 절차적 문제
   - 동결 사전 통지 부재
   - 이의 제기 절차 불명확
   - 해제 지연으로 인한 추가 피해

철학적 모순:

## 7. 사용자 보호 메커니즘 부재

### 7.1 Minter 독점 구조의 위험

```
현재 설계된 구조:

[원화 입금] → [Minter] → [스테이블코인 발행] → [End User 유통]
                ↓
         [원화 출금] ← [Minter만 가능]

문제점:
- 일반인 직접 원화 출금 불가 (규제 관리 명목)
- Minter가 독점적 위치 점유
- 유동성 병목 발생 가능
```

### 7.2 리스크 시나리오

```
사용자가 스테이블코인을 원화로 교환하고 싶을 때:

시나리오 1: 높은 수수료
├── Minter에게 출금 요청 필수
├── Minter가 수수료를 임의로 부과 가능 (독점적 위치)
└── 사용자는 다른 선택지 없음

시나리오 2: 지급 거부
├── Minter가 특정 사유로 지급 거부
├── 사용자 구제책 불명확
└── 법적 분쟁 시 관할권 문제

시나리오 3: Minter 부도
├── Minter 운영 기업 부도/파산
├── 예치금 보호 방안 미설명
└── 예금자보호법 적용 불가

시나리오 4: 해킹/사고
├── Bridge/컨트랙트 해킹 발생
├── "탈중앙 블록체인"으로 책임 회피 가능
└── 피해자 구제 메커니즘 부재
```

## 8. 이해관계자별 고충 분석

### 8.1 규제 당국 관점

| 고충 | 상세 내용 |
|------|----------|
| **법률 공백** | 스테이블코인 전용 법률 없음, 기존 법률 적용 시 해석 논란 |
| **소비자 보호** | 테라 사태 재발 방지를 위한 보호 장치 설계 어려움 |
| **AML/KYC 실효성** | 탈중앙 구조에서 효과적인 자금세탁 방지 방법론 부재 |
| **감독 관할권** | 해외 노드 운영, 탈중앙 거버넌스 시 감독 권한 한계 |
| **선례 부재** | 유사 사례 없어 의사결정 기준 부재 |
| **책임 회피 우려** | 문제 발생 시 "허용했다"는 비판 우려 |

### 8.2 금융기관 관점

| 고충 | 상세 내용 |
|------|----------|
| **법적 리스크** | 적법성 미검증 사업 참여 시 기존 인허가 취소 우려 |
| **경쟁 딜레마** | 자체 솔루션 vs 외부 블록체인 참여 선택 어려움 |
| **책임 범위** | 문제 발생 시 금융기관 책임 범위 불명확 |
| **기술 검증** | 성능 주장(TPS, Finality 등) 독립 검증 어려움 |
| **평판 리스크** | 스테이블코인 사업 참여로 인한 이미지 훼손 우려 |
| **수익성** | 낮은 수수료(1원) 구조에서 수익 모델 불명확 |

### 8.3 사업자 관점

| 고충 | 상세 내용 |
|------|----------|

| **시장 신뢰** | 테라 사태 이후 스테이블코인 전반에 대한 부정적 인식 |
| **기술 검증** | 테스트넷 공개, 감사 보고서 등 신뢰 구축 비용 |
| **지속 가능성** | 검증자 인센티브 등 장기 운영 모델 설계 어려움 |

## 9. 악순환 구조와 해결 방향

### 9.1 현재의 악순환 구조

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  법률 미비 ───────────────────────→ 금융기관 참여 주저           │
│      ↑                                      │                   │
│      │                                      ↓                   │
│      │                              생태계 형성 실패             │
│      │                                      │                   │
│      │                                      ↓                   │

│      │                                                          │

### 9.2 해결을 위한 잠재적 접근법

#### 접근법 1: 규제 샌드박스 활용

```
금융규제 샌드박스 활용 시나리오:

1단계: 샌드박스 신청
├── 혁신금융서비스 지정 신청
├── 제한된 범위/규모로 시범 운영 허가
└── 2년간 특례 적용

2단계: 파일럿 운영
├── 소규모 Minter (1-2개 금융기관)
├── 제한된 사용자 그룹
└── 규제 당국 모니터링

3단계: 본인가 전환
├── 파일럿 결과 기반 규제 정비
├── 본격적인 사업 인가
└── 확대 운영

문제점:
- 샌드박스 심사 자체의 불확실성
- 2년 제한 기간 내 성과 입증 어려움
- 퇴출 시 사용자 보호 방안 필요
```

#### 접근법 2: 글로벌 규제 정비 대기

```
글로벌 규제 동향 활용:

EU MiCA (Markets in Crypto-Assets):
├── 2024년 발효
├── 스테이블코인 발행 요건 명확화
└── 한국 규제 참고 가능

미국 Stablecoin 법안:
├── 지속적인 법안 논의 중
├── 발행사 요건, 준비금 규정 등
└── 글로벌 표준 형성 가능성

일본 개정 자금결제법:
├── 2023년 시행
├── 스테이블코인 발행 규제 도입
└── 아시아 규제 선례

시사점:
→ 글로벌 규제 정비 시 한국도 유사한 프레임워크 도입 가능
→ 다만 시기 불확실 (1-3년 이상 소요 예상)
```

#### 접근법 3: CBDC 연계

```
한국은행 CBDC 연계 시나리오:

현황:
├── 한국은행 CBDC 파일럿 진행 중
├── 민간 참여 생태계 구상 중
└── 2025년 이후 본격화 예상

연계 가능성:
├── CBDC와 민간 스테이블코인 공존 모델
├── CBDC 위에서 운영되는 프라이버시 레이어
└── CBDC 호환 스마트컨트랙트 플랫폼

장점:
├── 규제 불확실성 해소
├── 중앙은행 신뢰 활용
└── 기존 금융 인프라 연동 용이

문제점:
├── CBDC 정책 방향 불확실
├── 민간 혁신 공간 제한 가능
└── 탈중앙화 철학과 충돌
```

## 10. 결론 및 시사점

### 10.1 핵심 결론

```
한국에서 스테이블코인 기반 L1 및 결제 비즈니스 추진의 핵심 장벽:

1. 법적 프레임워크 부재
   → 스테이블코인 전용 규제 없음
   → 기존 법률 적용 시 해석 논란

2. 금융기관 참여 유인 부족
   → 법적 리스크로 인한 관망
   → 자체 솔루션 선호

3. 테라 사태 이후 신뢰 위기
   → 스테이블코인 전반에 대한 부정적 인식
   → 규제 당국의 보수적 접근

4. 탈중앙화 vs 규제의 딜레마
   → 양립하기 어려운 가치 충돌
   → 절충안의 철학적 모순

5. 사용자 보호 메커니즘 부재
   → 기존 금융 수준의 보호 장치 없음
   → 피해 구제 절차 불명확
```

### 10.2 StableNet 관점의 시사점

```
StableNet의 진짜 경쟁 우위는 기술이 아닌 관계 구축:

기술은 따라할 수 있습니다:
├── Paymaster → ERC-4337로 가능
├── Stealth Address → Umbra로 가능
├── Fast Finality → Solana, Avalanche도 빠름
└── Blacklist → USDC도 가능

파트너십과 규제 관계는 따라하기 어렵습니다:

진짜 해자(Moat):
1. 한국 금융기관과의 Minter 파트너십 (선점)
   → 은행, PG사와의 MOU/계약

2. 한국 규제 당국과의 관계 구축 (선점)
   → 금융위, 한국은행과의 사전 협의

3. 원화 기반 UX 최적화 (문화적 맥락)
   → 언어, 단위, 결제 문화 이해

4. Arc/Tempo 한국 진출 전 시장 점유
   → First Mover Advantage
```
