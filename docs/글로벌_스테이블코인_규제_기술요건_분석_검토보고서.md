## 1. 검토 개요

### 1.1 검토 목적

원본 문서 "글로벌 스테이블코인 규제 및 기술요건 분석"의 내용이 실제 글로벌 규제 프레임워크와 일치하는지 검증하고, 팀 내부에서 활용할 수 있는 실용적인 자료로 정리합니다.

### 1.2 검토 방법

- **외부 출처 검증**: FSB, MiCA, GENIUS Act, 홍콩, 싱가포르 등 공식 규제 문서 및 신뢰 가능한 분석 자료 대조
- **최신 정보 반영**: 2024-2025년 최신 규제 동향 확인
- **기술 요건 실현 가능성**: 실제 구현 가능한 기술 요건인지 평가

### 1.3 검토 범위

| 검토 항목 | 내용 |
|-----------|------|
| **규제 시행일** | 각 국가별 규제 시행 시점 정확성 |
| **기술 요건** | 준비금, 분리 보관, 상환권 등 핵심 요건 |
| **공통 요건 도출** | 6개국 비교표의 정확성 |
| **누락 사항** | 원본 문서에 빠진 중요 항목 |
| **과장/일반화** | 과도하게 일반화된 부분 |

## 4. 보완 필요 사항

### 4.1 원본 문서에 누락된 중요 항목

원본 문서는 기술 요건 중심으로 잘 정리되어 있으며, 한국 규제 예측 및 StableNet 기술 매핑까지 포함되어 있어 실용적입니다. 다만, 다음 항목들이 **추가로 고려**되어야 합니다:

#### 1. 회복 및 해산 계획 (Recovery & Resolution Plan)

**FSB 권고안에서 강조하는 항목**이지만 원본 문서에 명시적으로 포함되지 않음.

| 항목 | 내용 | 중요도 |
|------|------|--------|
| **Recovery Plan** | 비상 상황 시 자산 회수 및 운영 복구 계획 | 🔴 Critical |
| **Resolution Plan** | 발행자 실패 시 청산 및 사용자 보호 절차 | 🔴 Critical |
| **유동성 버퍼** | 스트레스 상황 대비 추가 유동성 확보 | 🟠 High |

**대응 방안**:
- 비상 상황 대응 프로토콜 문서화
- 유동성 스트레스 테스트 시나리오 수립
- 청산 시나리오 및 사용자 자산 보호 절차 정의

#### 2. 크로스보더 협력 및 정보 공유

**FSB 권고안**에서 강조하나 원본 문서에 포함되지 않음.

| 항목 | 내용 | 중요도 |
|------|------|--------|
| **규제당국 간 협력** | 다수 관할권 규제당국과의 정보 공유 | 🟠 High |
| **국경 간 운영** | 해외 발행자 규제 범위 및 준수 요건 | 🟠 High |
| **상호인증 (Passporting)** | EU MiCA 등에서 허용하는 상호인증 체계 | 🟡 Medium |

**대응 방안**:
- 다수 관할권 진출 시 규제당국 협력 체계 구축
- 해외 발행자와의 협업 시 규제 준수 방안 수립

#### 3. 기술적 감사 및 온체인/오프체인 투명성

원본 문서는 "감사"를 언급하나, **기술적 감사 방법론**은 구체화되지 않음.

| 항목 | 내용 | 중요도 |
|------|------|--------|
| **온체인 증명** | 블록체인상 발행량 실시간 추적 | 🟠 High |
| **오프체인 준비금 검증** | 은행 계좌, 수탁사 자산 증명 | 🟠 High |
| **데이터 정합성** | 온체인/오프체인 데이터 일치성 검증 | 🟠 High |

**대응 방안**:
- Proof of Reserves 시스템 구축 (온체인 발행량 ↔ 오프체인 준비금)
- 실시간 모니터링 대시보드 개발
- 외부 감사 기관과의 기술 감사 프로세스 수립
- 온체인 증명 메커니즘 (Merkle Tree 기반)
- 오프체인 준비금 검증 (은행 계좌, 수탁사 API)
- 데이터 정합성 자동 검증 시스템

#### 4. 시스템적 중요성 (Systemic Importance) 평가

원본 문서에 언급되나, **구체적 평가 기준**이 부족함.

| 항목 | 내용 | 중요도 |
|------|------|--------|
| **규모 기준** | 발행량, 거래량, 사용자 수 기준 | 🟡 Medium |
| **차등 규제** | Systemic vs Non-Systemic 요건 차이 | 🟡 Medium |
| **추가 요건** | Systemic 지정 시 더 엄격한 요건 | 🟡 Medium |

**대응 방안**:
- 발행 규모에 따른 단계적 준비
- Systemic 지정 시 추가 요건 대응 계획 수립

### 4.2 과도하게 일반화된 부분

#### 1. "이자 지급 금지"

**원본 문서**: EU만 이자 지급 금지로 표시 ✅ 정확

**주의사항**: 
- 일부 국가에서는 "이자 지급"과 "리워드 프로그램"을 구분
- 실제 규제 해석에 따라 차이가 있을 수 있음

**대응**: 각 관할권별 세부 규정 확인 필요

#### 2. "알고리즘 스테이블코인 금지"

**원본 문서**: EU만 금지로 표시 ✅ 정확

**주의사항**:
- 일부 국가에서는 "알고리즘 기반"과 "담보 없는" 스테이블코인을 구분
- 미래 규제 변화 가능성 존재

**대응**: 알고리즘 메커니즘 사용 시 각 관할권별 확인 필요

#### 3. "상환 기한"

**원본 문서**: 1~5일 범위로 표시 ✅ 정확

**주의사항**:
- 국가별로 "영업일 기준" vs "일 기준" 차이
- 대규모 상환 시 추가 시간 소요 가능

**대응**: 각 관할권별 상환 프로세스 세부 확인 필요

## 5. 팀 대응 방안

### 5.1 즉시 대응 필요 (Critical)

#### 1. Proof of Reserves 시스템 구축

**목적**: 모든 규제에서 요구하는 준비금 100% 증명

**구현 방안**:
```
┌─────────────────────────────────────────┐
│  Proof of Reserves Architecture         │
├─────────────────────────────────────────┤
│                                          │
│  1. 오프체인 준비금 추적                 │
│     - 은행 계좌 잔액 API 연동            │
│     - 수탁사 자산 증명 (Custodian API)  │
│                                          │
│  2. 온체인 발행량 추적                   │
│     - 총 발행량 (Total Supply)           │
│     - 소각량 (Burned Amount)             │
│                                          │
│  3. 실시간 검증                          │
│     - 준비금 ≥ 발행량 검증               │
│     - 일일 자동 검증 리포트               │
│                                          │
│  4. 공개 대시보드                        │
│     - 웹사이트 공개                      │
│     - 월간 검증 보고서 자동 생성         │
│                                          │
└─────────────────────────────────────────┘
```

**우선순위**: 🔴 Critical  
**예상 기간**: 2-3개월

#### 2. Segregated Treasury 컨트랙트 구현

**목적**: 발행자 자산과 준비금 완전 분리

**구현 방안**:
- Multi-sig Treasury Contract (다중 서명 금고)
- 발행자 운영 자산과 완전 분리된 계좌 구조
- 수탁사(Custodian)와의 계약 및 통합
- 파산 원격(Bankruptcy-Remote) 구조 (미국, 영국 요건)
- 법정 신탁(Statutory Trust) 형태 (영국 요건)

**우선순위**: 🔴 Critical  
**예상 기간**: 1-2개월 (기본 구조), 3-4개월 (파산 원격 구조 포함)

#### 3. Redemption Manager 구현

**목적**: 액면가 1:1 상환 보장

**구현 방안**:
- 1:1 상환 컨트랙트 (액면가 보장)
- 상환 요청 큐 시스템 (FIFO 처리)
- 상환 기한 관리 (1~5영업일, 국가별 차이)
- 자동 상환 프로세스 (Fast Finality 활용)
- 최소 상환액 조건 금지 (영국 요건)
- 불합리한 수수료 부과 금지 (홍콩 요건)

**우선순위**: 🔴 Critical  
**예상 기간**: 2-3개월

#### 4. KYC/AML 통합 시스템

**목적**: 모든 규제에서 필수 요건

**구현 방안**:
- 온체인 Allowlist Registry (허가된 주소만 거래)
- 오프체인 KYC 프로바이더 연동 (Sumsub, Onfido 등)
- 실시간 AML 체크 (Chainalysis, Elliptic 등)
- 의심거래 모니터링 및 보고 (SAR)
- BSA 적용 (미국 GENIUS Act)
- AML 담당 임원 지정 (미국 요건)
- Travel Rule 준수 (특금법)

**우선순위**: 🔴 Critical  
**예상 기간**: 3-4개월
